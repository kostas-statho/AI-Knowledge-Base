1. ARCHITECTURE OVERVIEW
Correct Process Architecture — Process Automation Pattern

The alerting process is time-based, not event-based. A regular OIM Process fires once on an INSERT/UPDATE event and does not re-evaluate itself later. For time-gap checks (1 hour, 4 hours), the correct OIM pattern is a Process Automation — a scheduled job that runs a SQL query and then triggers a regular Process for each matching object.

This is the same pattern used in the SCN-3 "Email 15 Days Before Expiry" exercise. The architecture requires four components:

#ComponentTypePurpose
1CCC_AAD_AutomationAlertProcess AutomationRuns hourly. Queries AADUser records with no mailbox, created >1hr ago, flag not yet set
2CCC_AAD_AutomationEscalationProcess AutomationRuns hourly. Same query but for >4hr and escalation flag not set
3CCC_AAD_SendNoMailboxAlertRegular ProcessTriggered by Automation 1. Sends email via Template 1. Sets CCC_IsFirstAlertSent = 1
4CCC_AAD_SendNoMailboxEscalationRegular ProcessTriggered by Automation 2. Sends email via Template 2. Sets CCC_IsEscalationSent = 1
Important — Why Not Retries? Do not use process step retries for time-based logic. Retry intervals are coarse and unreliable for precision timing. The Process Automation approach queries the live database state at execution time, making it accurate and idempotent by design.
OIM Scripting Concepts Clarified

There are three distinct scripting contexts in OIM processes, each with different syntax and capability. These must not be confused:

ContextWhereSyntaxPurpose
Process Generating Condition Process-level or Step-level "Generating Condition" field Single inline VB.NET expression wrapped in { }
e.g. {Value = $IsTemporaryDeactivated:Bool$}		 Boolean gate — should this process or step be queued at all?
Pre-Script on Process Step "Pre-Script" tab on a Process Step Full VB.NET code block. Returns True to execute step, False to skip it Complex conditional logic evaluated at step execution time
Process Automation Query SQL query on the Process Automation object SQL WHERE clause returning UID_AADUser values Collects the objects the automation will act on
Important The Generating Condition is a simple inline expression — it does NOT support multi-line VB.NET, Dim statements, database queries, or async/await. Those belong in a Pre-Script or the Process Automation SQL query instead.

2. PREREQUISITES
Custom Boolean Columns on AADUser

Two custom boolean columns must be added to the AADUser table via Designer Schema Extensions. These are the deduplication flags — without them, each automation run would re-send the email. This follows the same pattern as CCC_IsManagerNotifiedForExpire in the SCN-3 exercise.

How to Add in Designer
Schema Extensions → AADUser table → Add column
Column NameData TypeDefaultPurpose
CCC_IsFirstAlertSent Bool (Bit) 0 Set to 1 after the 1-hour alert email is sent. The Process Automation query filters this out, ensuring the email is only sent once per account.
CCC_IsEscalationSent Bool (Bit) 0 Set to 1 after the 4-hour escalation email is sent. Same mechanism as above.
Note After adding schema extensions in Designer, you must compile and deploy the schema change before these columns appear in the database or are available in process parameters.
Configuration Parameters for Recipient Addresses

Store recipient email addresses as OIM Configuration Parameters rather than hardcoding them in process steps. This allows routing to be changed by an operations administrator without touching Designer.

How to Add in Designer
Base Data → General → Configuration Parameters → New
Config Parameter PathTypeExample Value
Custom\AAD\Notifications\ServiceDeskEmailStringit-servicedesk@contoso.com
Custom\AAD\Notifications\ITManagerEmailStringit-manager@contoso.com
SMTP Server

Verify the OIM Job Server has a working SMTP configuration before building any processes. Navigate to:

Base Data → General → Mail Server

Send a test email using the built-in test function to confirm connectivity before proceeding.

5. PROCESS AUTOMATION — STEP-BY-STEP DESIGNER GUIDE
Part A — Process Automation: CCC_AAD_AutomationAlert (1-Hour)
Process Orchestration → Process Automations → New
FieldValue
NameCCC_AAD_AutomationAlert
Object typeAADUser
ProcessCCC_AAD_SendNoMailboxAlert (created in Part C)
ScheduleEvery 1 hour (create under Base Data → Schedules if needed)
EnabledYes
SQL Query (returns UIDs of matching AADUser objects)
UID_AADUser NOT IN ( SELECT UID_AADUser FROM O365Mailbox WHERE UID_AADUser IS NOT NULL ) AND XDateInserted < DATEADD(HOUR, -1, GETUTCDATE()) AND XDateInserted > DATEADD(HOUR, -48, GETUTCDATE()) AND ISNULL(CCC_IsFirstAlertSent, 0) = 0
Note The 48-hour upper cap prevents the automation from endlessly picking up very old unresolved accounts. Adjust to your operational SLA. Confirm the exact foreign key column name in O365Mailbox by checking the Schema Browser before deploying.

6. TESTING & VALIDATION
Mail Template Preview in Designer
  1. Open the mail template in Notifications → Mail templates.
  2. Click Preview in the toolbar.
  3. Select an existing AADUser test record from the object picker.
  4. Verify all $Token$ values resolve to real data in both HTML and plain-text views.
  5. If a token renders as the literal text $ColumnName$, the column name is wrong — verify via Schema Browser → AADUser table.
End-to-End Test Procedure
  1. Create a test AADUser record in the non-production OIM tenant with no linked O365Mailbox row.
  2. Temporarily shorten the automation query threshold from 1 hour to 0 minutes to avoid waiting:
    -- Temporary test version AND XDateInserted < DATEADD(MINUTE, 0, GETUTCDATE())
  3. Manually trigger the automation from Designer → Process Automations → right-click → Run now.
  4. Watch Job Queue Info — the process should appear and both steps should reach Finished status.
  5. Check the test inbox for the alert email. Verify HTML rendering in Outlook.
  6. Confirm the flag is set: query SELECT CCC_IsFirstAlertSent FROM AADUser WHERE UID_AADUser = '...' — should return 1.
  7. Run the automation again — the object should NOT be picked up a second time (flag = 1 excludes it).
  8. Restore thresholds to 1 hour and 4 hours before go-live.

7. TROUBLESHOOTING
Common Issues & Resolutions
SymptomLikely CauseResolution
Automation never picks up any objects SQL query has wrong column name, or threshold too strict Run the SQL directly in SSMS against the OIM database. Verify O365Mailbox.UID_AADUser column name in Schema Browser. Temporarily remove the time threshold to test.
Process appears in Job Queue but no email arrives SMTP not configured, or wrong recipient address in config param Check Base Data → Mail Server and send a test email. Verify the config parameter value under Base Data → Configuration Parameters.
Token renders as literal $DisplayName$ in email Column name wrong, or base object not set to AADUser Open the template in Designer → check Base object = AADUser. Cross-check column name against AADUser in Schema Browser.
Same account gets the email every automation run CCC_IsFirstAlertSent flag not being set, or Set step failing Check Job Queue for step Set_AlertSentFlag — is it Finished or Error? Check step configuration and WhereClause. Query the DB to confirm flag value.
Pre-Script skips the email step unexpectedly Mailbox appeared between automation query and step execution, OR script logic error This is correct behaviour if a mailbox was provisioned. To debug a logic error, temporarily remove the Pre-Script and re-test. Re-add incrementally.
Process step status = Frozen A Wait step condition is never satisfied (should not apply here unless added) Navigate to Job Queue → Frozen → inspect the frozen step condition. Unfreeze after investigating.
Generating Condition syntax error in Designer Multi-line VB.NET used instead of single inline expression The Generating Condition field only accepts a single inline expression: {Value = SomeBooleanExpression}. Move complex logic to a Pre-Script on the process step instead.
Schema extension columns not visible in process parameters Schema compiled but database not yet updated After adding custom columns in Designer, run Database → Update database to apply the schema change to the physical database.
