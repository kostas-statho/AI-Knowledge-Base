Basic Account Synchronization

Code: BAS  |  Beginner  |  8 exercises  |  ~3 days

Overview

Module Purpose

  • Introduces OIM's role model and inheritance
  • Covers Business Roles, System Roles, and Resources
  • Hands-on IT Shop configuration and usage
  • Approval workflow design and testing

Prerequisites

  • OIM installed and accessible
  • Basic OIM navigation
  • Login credentials provided
  • Web portal and API Server accessible

Key Concepts

  • XOrigin: 1=Direct, 2=Indirect (inherited), 3=Both
  • XIsInEffect: 0=Blocked/Disabled, 1=Active/Effective
  • Block Inheritance: Stops cascade DOWN from Business Role
  • Identities do not inherit: Members of role get NO resources
  • IT Shop: Self-service portal for access requests
  • Assignment Resource: IT Shop wrapper for Business Roles

Mentor Guidance

  • Draw XOrigin/XIsInEffect on whiteboard before BAS-3
  • Show IT Shop as end-user FIRST, then admin config (BAS-5/6)
  • Use two browser sessions for BAS-7/8 approval flows
  • BAS-8 builds on BAS-7: do them in sequence
Exercises
BAS-1

Role Assignment Basics

Overview

Introduces trainees to the fundamental role assignment workflow in OIM Manager. Trainees navigate the role model and make their first assignments involving Business Roles and the identity overview. Steps 1–12 cover initial navigation, first role assignments, and verifying assignments on the identity record.

Key Steps (1–12)
  • Navigate to OIM Manager and locate the Role Model section
  • Identify Business Roles, System Roles, and Resources in the hierarchy
  • Assign a Business Role to a test identity and verify it appears on the identity overview
  • Remove the assignment and confirm it is reflected on the overview
  • Observe how the identity's assignments update in real time
  • Steps 6–12 continue building on direct and indirect role assignment navigation
Mentor Note BAS-1 is intentionally light on configuration — it is about orientation. Make sure trainees can confidently navigate to an identity, see its assignments, and understand what "Business Role" vs "System Role" vs "Resource" means before moving on.
BAS-2

Block Inheritance on Business Roles

Requirements

Configure blocking inheritance on Business Roles. Observe the effect on resources that would normally cascade down through the role hierarchy.

Solution
  • After blocking inheritance on Business Role X: Resource R1 (assigned on System Role A, assigned on Business Role X) is NOT inherited
  • After blocking inheritance on Business Role Y: Resource R2 (assigned on System Role B, assigned on Business Role Y) is NOT inherited
Teaching Point Block Inheritance is a "wall" that stops resources cascading DOWN. Draw the hierarchy: Business Role X → System Role A → Resource R1. Blocking on X means R1 never reaches identities assigned to X.
BAS-3

XOrigin & XIsInEffect — Deep Dive

Requirements

Assign Business Roles X and Y to identity "Akis". Test various block inheritance and "Identities do not inherit" combinations. Directly assign System Roles A and B to Akis. Observe XOrigin and XIsInEffect values throughout.

Solution — Step by Step
  • Step 1: Assign Business Role X to Akis — visible on identity overview. Remove — visible on overview.
  • Step 2: Same procedure with Business Role Y.
  • Step 3: Enable/Disable Block inheritance on X/Y via Main Data:
    • "Block inheritance" = blocks RESOURCES from being inherited TO X/Y (no resources cascade in from above)
    • "Identities do not inherit" = blocks identities assigned to X/Y from inheriting resources FROM X/Y (no resources cascade out to members)
  • Step 4: Assign System Roles A and B directly to Akis.
  • Steps 5–7: Various assignment combinations — test with and without block settings active.
  • Step 8: By blocking inheritance, Akis loses resources inherited from system roles. R1 and R2 are not inherited from A, B to Akis.

BAS-4

Department Assignments — Resources, System Roles, Location, Cost Center

Requirements

Configure a department with multiple assignment types: assigned Resources, System Roles, Location, and Cost Center. Use SQL queries to verify the assignments are reflected correctly in the database.

Solution
  • Step 1: Open the target department in OIM Manager.
  • Step 2: Assign Resources, System Roles, Location, and Cost Center to the department.
  • Step 3: Run the provided SQL queries against the OIM database to verify that the assignment rows appear in the correct join tables (e.g., DepartmentHasQERResource, DepartmentHasEset).
Teaching Point Every assignment in OIM creates a row in a "join" table. If the assignment does not appear in the join table, the relationship does not exist — regardless of what the UI shows. SQL queries are the ground truth.
BAS-5

IT Shop — Enable System Role for Self-Service

Requirements

Enable System Role "A" for the IT Shop so identities can request it via the web portal (ApiServer). Configure a Service Item and add it to the "Access Lifecycle" IT Shop.

Solution
  • Step 1: Open System Role "A" → Main Data → enable the "IT Shop" option. Create Service Item named Service_Item_A.
  • Step 2: Navigate to the IT Shop configuration. Add Service_Item_A to the "Access Lifecycle" IT Shop shelf.
  • Step 3: Log into the ApiServer (web portal) as a test user. Request Service_Item_A from the IT Shop.
  • Step 4: After the request is fulfilled, unsubscribe from the item.

BAS-6

IT Shop — Assignment Resource for Business Role

Requirements

Create an Assignment Resource for Business Role "X" to make it requestable via the IT Shop. Add it to the "Access Lifecycle" shelf. Test the full request/unsubscribe/re-request cycle.

Solution
  • Step 1: Create an Assignment Resource for Business Role "X". This creates an IT Shop-requestable wrapper around the Business Role.
  • Step 2: Add the Assignment Resource "X" (its service item X_Service_Item) to the "Access Lifecycle" IT Shop shelf.
  • Step 3: Request X_Service_Item via the ApiServer web portal:
    • After page refresh, status is automatically set to Assigned
    • After assignment is visible, the Unsubscribe option appears
    • After unsubscribing, the item can be requested again and is auto-approved again
Database Note
SQL Query When adding a service item to a shelf, OIM creates a new row in the ITShopOrgHasQERAssign table. You can verify the shelf assignment with:
SELECT * FROM ITShopOrgHasQERAssign WHERE UID_QERAssign = '<uid of assignment resource>'

BAS-7

Manager Approval Workflow

Requirements

Create a Manager (M) and an Identity (I). Set M as the line manager of I. Configure the CM-Recipient's Manager approval workflow. Test the request and approval cycle from both perspectives.

Solution
  • Step 1: Create Manager identity "M" and Identity "I". Set M as the line manager of I in I's Main Data.
  • Step 3: Assign the line manager approval workflow (CM-Recipient's Manager) to the service item's approval policy.
  • Test 1: Log in as Identity I. Request the IT Shop item. Observe request status — it should show "Waiting for approval" and display who must approve.
  • Test 2: Review the request status from I's perspective — see the pending approval.
  • Test 3: Switch browser session to Manager M. Approve the pending request.
  • Test 4: Confirm the item is now assigned to I after M's approval.
  • Test 5: Log back in as I. Unsubscribe from the item.
Mentor Note Use two browser sessions simultaneously — one as Identity I, one as Manager M. This makes the approval flow tangible. Trainees can see the request appear in M's approval queue in real time when I submits it.
CM-Recipient's Manager This is OIM's built-in "ask the requester's manager" approval workflow. It automatically resolves to the identity set as the line manager in Main Data — no custom configuration needed for the resolver itself.
BAS-8

Complex Approval Workflow with Conditions

Requirements

Build a conditional approval workflow with fallback logic. The workflow checks whether the manager is "valid" (not disabled, not permanently deactivated, has an active AD account). If the manager is invalid, routing falls back to the "FallBack_Approval_Supervisor" Business Role.

Exercise Order Note Test Case 3 → 2 → 1 in execution order. The exercises deliberately build on each other — complete them in sequence.
Solution — Configuration Steps
  • Step 1: Verify the manager identity has a connected AD account. If not, create one: AD Forest → Domain → AD user account (ADS tab).
  • Step 2: Create an approval procedure with the appropriate query for the IsValidManager calculation.
  • Step 3: Create an approval workflow with the following conditions:
    • Identity is NOT temporarily disabled: "Identity/Organizational/Temporarily Inactive"
    • Identity is NOT permanently disabled: "Identity/Tasks/Deactivate identity permanently"
    • Active AD account connected: "Identity/Tasks/Assign AD user acc"
  • Step 4: Create an approval policy.
  • Step 5: Assign the workflow to the policy.
  • Step 6: Assign the policy to service item X_Service_Item via Main Data → Approval Policy.
  • Step 7: Apply conditions on the workflow. Add the required approval steps. Set the condition on the Calculated Approval step: IsValidManager.

Production Pattern — FallBack_Approval_Supervisor The FallBack_Approval_Supervisor Business Role is the safety net when the primary approver chain is broken. In production, this role should always have active, responsible members assigned. If it has no members, requests with invalid managers will become stuck indefinitely. Emphasise this is a real, widely-used production pattern.
Mentor Note Use two browser sessions simultaneously — one as Identity I, one as Manager M. BAS-8 builds directly on BAS-7: trainees must complete BAS-7 first. For AD account creation failures, check that AD Forest and Domain are configured before attempting to create the AD user account in the ADS tab.
