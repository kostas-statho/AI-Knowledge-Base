Approval Workflows (WFL)

Approval workflows control who must approve IT Shop requests and attestation cases. These exercises progress from a simple auto-approve workflow through manager approval, custom SQL-based approval procedures, and complex multi-step workflows with fallback approvers.

Key Concepts

  • Approval Workflow, Approval Level, Approval Step
  • Approval Procedure, Approval Policy
  • Service Item assignment
  • CM - Recipient's Manager (built-in approval level)
  • Custom SQL function for approver lookup
  • Product Owner, FallBack_Approvers role

Exercise Progression

  • WFL-1: Simple auto-approve workflow — baseline
  • WFL-2: Manager approval via CM - Recipient's Manager
  • WFL-3: Custom SQL function as approval procedure
  • WFL-4: Multi-step workflow with Product Owner and Fallback approvers

WFL-1: Simple Workflow — Auto-Approve (No Manager)

1

Simple Workflow — Auto-Approve

Creates the baseline approval workflow: a simple auto-approve (or direct-assignment) workflow. This is the starting point before adding approval logic in WFL-2. The contrast between WFL-1 and WFL-2 makes the approval mechanism clear.

Solution Steps
  1. Step 1 — Create Approval Workflow: Create a new Approval Workflow (simple — auto-approve or direct assignment to the requester). No additional approval level configuration is needed for the baseline.
  2. Step 2 — Create Approval Policy: Create a new Approval Policy and assign the workflow created in Step 1 to it.
  3. Step 3 — Create System Role and Service Item: Create a System Role and a corresponding Service Item. Add the Service Item to the IT Shop. Assign the Approval Policy from Step 2 to the Service Item.
Mentor Note The purpose of WFL-1 is to establish the approval infrastructure (workflow + policy + service item) with zero friction. Trainees learn the configuration chain — Workflow -> Policy -> Service Item — before any approval logic is added. WFL-2 then modifies the approval level to add real approval.
Test Cases
#Test ScenarioIdentitiesExpected ResultResult
1 Request with identity A User logs in as Identity A and requests the product Product is assigned to Identity A automatically (no approval step) PASS
2 Request with identity B User logs in as Identity B and requests the product Product is assigned to Identity B automatically (no approval step) PASS

WFL-2: Manager Approval Workflow

2

Manager Approval Workflow (CM - Recipient's Manager)

Modifies the WFL-1 workflow to require manager approval using the built-in "CM - Recipient's Manager" approval level. Also removes the Unsubscription Workflow from the Approval Policy so that unsubscriptions are auto-approved.

Solution Steps
  1. Step 1 — Modify Approval Level to "CM - Recipient's Manager": In the approval workflow created in WFL-1, change the approval level to the built-in CM - Recipient's Manager type. This causes the request to be sent to the identity's manager (UID_PersonHead) for approval.
  2. Step 2 — Remove Unsubscription Workflow from Approval Policy: In the Approval Policy, remove or clear the Unsubscription Workflow. This ensures that unsubscriptions (product removal requests) are automatically approved without requiring manager approval.
CM - Recipient's Manager This is a built-in OIM approval level that resolves to the UID_PersonHead field of the requesting identity. If the identity has no manager configured (UID_PersonHead is null), the request cannot proceed through this approval level unless a fallback is configured.
Test Cases
#Test ScenarioSetupExpected ResultResult
1 Request for self — manager must approve Identity B requests for himself; Identity A is B's manager (UID_PersonHead = A) Request is routed to Identity A for approval — A must accept before product is assigned to B PASS
2 Request by A for B — requester is approver Identity A requests the product for Identity B; A is also B's manager Request is automatically approved — requester A is the same person as the designated approver (self-approval) PASS
3 Unsubscription by B for himself Identity B unsubscribes (removes) the product from himself Unsubscription is automatically approved — no manager approval needed (Unsubscription Workflow removed from policy) PASS
4 Unsubscription by A for B Identity A unsubscribes the product on behalf of Identity B Unsubscription is automatically approved — no additional approval required PASS

WFL-3: Custom SQL Function as Approval Procedure

3

Custom SQL Function as Approval Procedure

A complex workflow that uses a custom SQL function to find the first internal manager in an identity's management chain. If no internal manager exists, approval falls back to a business role (Operation Engineer AERole). The workflow dynamically recalculates approvers as conditions change mid-workflow.

SQL Function Created Outside OIM The function dbo.CCC_IsInternalManager is created in SQL Server Management Studio (SSMS) — directly in the OIM database — OUTSIDE of OIM configuration. This is a custom database function. Show trainees how OIM approval procedures can call custom SQL that exists in the database.
Solution Steps
  1. Step 1 — Create SQL function "dbo.CCC_IsInternalManager" via SSMS: In SQL Server Management Studio, create a table-valued function in the OIM database that traverses the management chain of a given identity and returns the first manager who is NOT marked as external (isExternal = 0).
  2. Step 2 — Create Approval Procedure calling the SQL function: In OIM, create a new Approval Procedure that invokes the dbo.CCC_IsInternalManager function to resolve the approver. The procedure receives the request UID and returns the appropriate approver identity.
  3. Step 3 — Create Workflow "CCC_Workflow03": Create an approval workflow with a conditional approval structure:
    • CD condition (primary path): {exists (Select * from dbo.CCC_IsInternalManager('@uid_personwantsorg'))} — uses the SQL function if an internal manager exists
    • OR Role (fallback path): {Base roles\Operations support\Operation Engineer} — routes to Operation Engineer AERole members if no internal manager exists
  4. Step 4 — Create Approval Policy "CCC_workflow_policy03": Create and configure an Approval Policy that uses the CCC_Workflow03 workflow. Assign this policy to the relevant Service Item (ESet service item).
The Dynamic Recalculation Behaviour When an approval step is pending and the underlying condition changes (e.g., a manager is switched from external to internal), OIM can recalculate the approver on the next workflow transition. This is the key behaviour validated in Test #3.

2 Manager set to external — fallback to next level Set Identity C to External (isExternal = 1); request ESet with Identity D again Identity B (next internal manager in the chain) receives the approval request PASS

After reconfiguring B to Internal, the next workflow recalculation routes to B. Approve with B to complete. PASS
Test #3 Walk-Through Note Test #3 demonstrates dynamic recalculation and requires careful step-by-step execution. Path: (1) All managers external → request submitted → OE role receives task. (2) Without resolving OE task, set B back to Internal. (3) Advance the workflow to trigger recalculation — approval is now rerouted to B. (4) Approve with B. Walk trainees through this very slowly and draw the approval chain transitions on a whiteboard.

WFL-4: Multi-Step Workflow with Product Owner and Fallback

4

Multi-Step Workflow — Product Owner + Fallback Approvers

A two-step approval workflow: Step 1 requires the product owner (determined by the SQL function) to approve; Step 2 requires a different product owner who did not approve Step 1 to approve again. Both steps use the Operation Engineer AERole as Fallback_Approvers.

Solution Steps
  1. Step 1 — Create Product Owner & Assign to ESet: Create a Product Owner (Application Role or direct person assignment) and assign it to the ESet named CCC_Workflow. The product owner represents who is responsible for approving requests for this resource.
  2. Step 2 — Create Approval Procedure for Remaining Product Owners: Create a new approval procedure that returns product owners who did NOT approve Step 1 (i.e., the remaining product owner for Step 2 approval).
  3. Step 3 — Create Workflow with FallBack_Approvers on Both Steps: Configure the multi-step workflow:
    • Step 1: Product owner approval using the SQL function from WFL-3; Fallback_Approvers = Operation Engineer AERole members
    • Step 2: Remaining product owner (from Step 2 procedure); Fallback_Approvers = Operation Engineer AERole members
Mentor — Product Owner Concept Product Owner approval is an additional approval concept. Instead of the requester's manager approving, the owner of the requested resource approves. This is a common pattern for sensitive or shared resources (AD groups, privileged roles) where the resource owner must consent to who gets access.

Test #4 — Dynamic Recalculation on Both Steps This is the most complex test in the module. It demonstrates that dynamic recalculation works independently on each workflow step. Walk through this test step by step. After Test #4, trainees should fully understand the dynamic approval recalculation mechanism.

Mentor Tips

WFL-1 / WFL-2 — Contrast Is the Teaching Tool Always start with WFL-1 (no approval) before adding manager approval in WFL-2. The contrast between "request is immediately assigned" (WFL-1) and "request waits for manager" (WFL-2) makes the approval mechanism click for trainees. If you start directly with WFL-2, the baseline is unclear.
WFL-2 — No Manager Means No Approval Path The "CM - Recipient's Manager" approval level uses UID_PersonHead. If the requesting identity has no manager configured, the request cannot be approved through this level (unless a fallback is configured). This is a common production issue. Always check manager assignments when testing this workflow.
WFL-3 — SQL Function Is Created Outside OIM The function dbo.CCC_IsInternalManager is created in SSMS directly in the OIM SQL Server database. This is a custom database function, not an OIM script. Show trainees how OIM approval procedures can call custom SQL functions that exist in the database. This is a powerful extensibility pattern.
WFL-3 Test #3 — Walk Very Slowly The dynamic recalculation behaviour in Test #3 is confusing. Draw the approval chain on a whiteboard and show each state transition: all external → OE receives task → configure B as internal → workflow advances → B receives task → B approves. The key insight is that recalculation only happens on workflow transitions, not instantaneously when the manager changes.
WFL-4 — Product Owner Is a Different Approval Axis Manager approval is "who is responsible for the person." Product Owner approval is "who is responsible for the resource." These are two independent axes. WFL-4 combines both via the SQL function (internal manager determines product owner at Step 1) and the remaining product owner at Step 2. Common real-world pattern for sensitive resources.

Troubleshooting Reference

ProblemLikely CauseSolution
Request auto-approves when approval is expected No approval step in the workflow, or "CM-Recipient's Manager" resolves to the requester (self-approval) Check workflow configuration — confirm an approval step is present; verify the manager chain so the approver is not the same person as the requester
Request stuck "waiting for approval" indefinitely Approver does not exist — empty business role, no manager, or SQL function returns no rows Check approver query resolution in Sync Editor or Designer; verify manager assignment on the requesting identity; add a fallback approver to the workflow step
SQL function not found by OIM Function not created in the correct database or schema (dbo) Verify dbo.CCC_IsInternalManager exists in the OIM SQL Server database; test the function directly in SSMS with a known UID_PersonWantsOrg value
Operation Engineer AERole not receiving approval tasks No members assigned to the "Operations support / Operation Engineer" AERole Open OIM Manager; navigate to Business Roles / AERoles; assign at least one identity to the "Operation Engineer" role
Workflow not recalculating after manager change Recalculation only occurs on the next workflow transition — not instantaneously Advance the workflow to the next transition step to trigger recalculation; recalculation requires a workflow event, not just a data change
