Module 012 — Permissions (PMS)

OIM's role model governs who can access what across the organisation. This module covers Business Roles, Application Roles, permission propagation through hierarchies, and a practical CSV employee import exercise using provided sample data.

01 Learning Objectives

  • 1Understand the conceptual difference between Business Roles and Application Roles in OIM
  • 2Create a role hierarchy with a parent and child roles demonstrating inheritance
  • 3Assign permissions to roles and understand how permissions propagate downward through the tree
  • 4Assign roles to Persons both directly and through inheritance from parent roles
  • 5Create employee Person objects from CSV sample data including correct use of the External flag
  • 6Understand XOrigin propagation: 1 = Direct, 2 = Inherited, and additive combinations such as 3 (both direct and inherited simultaneously)

03 Core Concepts

Business Roles vs Application Roles

Business Roles (Organisational)

  • Represents who a person is in the org
  • Arranged in a tree hierarchy
  • Examples: Developer in IT Dept, Manager in Finance
  • People are members of Business Roles
  • Business Roles grant Application Roles
  • Drives provisioning and entitlement decisions

Application Roles (Functional)

  • Represents what a person can do
  • Flat list structure — no hierarchy
  • Examples: Read-Only Portal, Admin Access
  • Granted via Business Roles (not directly to persons)
  • Controls OIM UI and API permissions
  • Compiled and deployed via Designer to take effect
Analogy for Trainees
Business Role = "who you are in the org" (e.g. Developer in IT Department). Application Role = "what you can do" (e.g. Read-only portal access). An org role GRANTS application roles; people are MEMBERS of org roles. Avoid mixing these up — it is one of the most common architecture mistakes in OIM implementations. Make the trainee state this distinction back in their own words before proceeding.

XOrigin Bitmask

XOrigin is a bitmask field on PersonInRole (and similar assignment tables) that records how an assignment came to exist. Multiple bits can be set simultaneously — the value is additive, not exclusive.

Label Value (Integer) Bit Pattern Meaning
Direct 1 bit 0 set Person was explicitly assigned to this role by a user or process
Inherited 2 bit 1 set Assignment flowed down from a parent role in the hierarchy
Both 3 bits 0+1 set Directly assigned AND inherited simultaneously — both bits are set; value = 1 + 2
Common Misunderstanding — Bitmask Is Not Exclusive
Trainees often expect XOrigin to be a single exclusive value. Show them that 3 means both bits are set at once (1 + 2 = 3). This matters operationally: removing a direct assignment when XOrigin = 3 does not delete the PersonInRole record — it drops from 3 to 2. The assignment persists via the inherited path even after the direct assignment is removed.

05 Exercises

PMS-1 Business Roles vs Application Roles
Goal: Understand the conceptual difference between Business Roles and Application Roles; explore both interfaces in OIM Manager and compare their structure.

Key Steps

  • Open OIM Manager and navigate to Business Roles — observe the tree view structure and note that roles can be nested
  • Navigate to Application Roles — observe the flat list and confirm there is no hierarchy
  • Compare the two interfaces side by side: note the structural differences
  • Open an existing Business Role — examine the Application Roles tab to see what functional permissions this org role grants
  • Open an existing Application Role — examine which Business Roles reference it
  • Ask the trainee: "If a new employee joins as a Developer, which role type do you assign to them, and why?"
  • Confirm understanding: trainee should answer Business Role, because it expresses their organisational position
Mentor Hint
Use the analogy directly: Business Role = "who you are in the org" (e.g. Developer in IT Dept); Application Role = "what you can do" (e.g. Read-only portal access). An org role GRANTS application roles; people are MEMBERS of org roles. Make the trainee explain it back in their own words before proceeding to PMS-2.
Common Mistake
Assigning Application Roles directly to Persons instead of routing through Business Roles. This bypasses the governance structure — individual assignments are invisible to role management reports, cannot be bulk-revoked when someone changes roles, and do not follow the recertification process. Always route access through Business Roles.
PMS-2 Create Role Hierarchy
Goal: Create an "IT Department" parent Business Role with three child roles — Tester, Developer, Publisher — matching the sample CSV employees.

Key Steps

  • Manager → Business Roles → New → set Name = "IT Department"
  • Enable the Inherit assignments flag on the IT Department role before saving
  • Save the parent role and confirm it appears in the tree as a root-level entry
  • Create a new Business Role named "Tester"; in the Parent field, select "IT Department"
  • Create a new Business Role named "Developer"; set Parent = "IT Department"
  • Create a new Business Role named "Publisher"; set Parent = "IT Department"
  • Verify the tree view shows IT Department with three child nodes nested beneath it
  • Refresh the tree if the children do not appear immediately
Mentor Hint
Connect the role names to the CSV employees now: Kalfas = Tester, Antwniou = Developer, Lyra = Publisher. This makes PMS-3 and PMS-4 feel purposeful rather than arbitrary. Trainees retain information significantly better when example data carries real meaning — avoid purely abstract examples like "Role A" and "Role B".
Common Mistake — Missing Parent Field
Not setting the Parent field on child roles. Without it, all three roles appear as separate root-level entries with no hierarchy, and inheritance cannot work at all. Always verify the tree view after creating each child role — the tree is the definitive visual confirmation.
Inherit Assignments Flag
The Inherit assignments flag on the IT Department (parent) Business Role controls whether entitlements assigned to the parent propagate down to all child role members. When this flag is disabled, child role members receive only entitlements directly assigned to their own role — they do not inherit from the parent — and XOrigin on any parent-sourced PersonInRole row will never reach 2 (Inherited). Enable this flag before starting PMS-4 so that the inheritance bitmask scenario is observable.
PMS-3 Import Employees from CSV
Goal: Create the three sample employees in OIM as Person objects, matching the CSV data exactly — including the correct IsExternal flag value for Anestis Antwniou.

Key Steps

  • Manager → Persons → New
  • Create Person 1: First Name = Dimitris, Last Name = Kalfas, Description = Tester, IsExternal = No → Save
  • Create Person 2: First Name = Anestis, Last Name = Antwniou, Description = Developer, IsExternal = Yes (tick the checkbox) → Save
  • Create Person 3: First Name = Anna, Last Name = Lyra, Description = Publisher, IsExternal = No → Save
  • Verify all three records exist: search for each by surname and confirm the IsExternal field value matches the CSV
  • Note the OIM-assigned UID on each Person record for reference in PMS-4
Mentor Hint
For Anestis, explicitly tick the IsExternal checkbox and have the trainee confirm its value before saving. External contractors in OIM commonly receive different provisioning rules — for example, excluded from Active Directory group provisioning, given time-limited accounts, or subject to additional approval workflows. This flag feeds into conditional process logic throughout OIM. Getting it right at the Person level prevents incorrect provisioning downstream.
Common Mistake
Creating all three employees with IsExternal = 0. The CSV clearly marks Anestis as External = 1. If this is missed, Anestis receives the same provisioning path as internal staff — which is incorrect and may violate security policy. Cross-reference source data carefully and verify every field, not just names.
Optional Extension
If the environment supports CSV synchronisation, briefly show how an OIM Sync project could import the CSV automatically rather than requiring manual entry. This previews Module SYN and demonstrates that manual Person creation is the exception, not the operational norm.
PMS-4 Assign Roles and Verify XOrigin
Goal: Assign each employee to their matching Business Role; run the DBQueue; inspect XOrigin values on PersonInRole records to understand the difference between direct and inherited assignments.

Key Steps

  • Open Dimitris Kalfas → Business Roles tab → click Assign → select "Tester" → Save
  • Open Anestis Antwniou → Business Roles tab → click Assign → select "Developer" → Save
  • Open Anna Lyra → Business Roles tab → click Assign → select "Publisher" → Save
  • Run the DBQueue Processor (or trigger the job service cycle)
  • After processing, open a database query tool and query PersonInRole filtered by each Person's UID
  • Examine the XOrigin column on each row — note which rows show 1, 2, or 3
  • Locate the row where the linked Business Role is "IT Department" (the parent) — this row shows XOrigin = 2 (inherited via child membership)
  • Discussion point: if you also directly assigned Kalfas to "IT Department" in addition to "Tester", what value would XOrigin show on the IT Department row, and why?
Mentor Hint — Reading the Bitmask
Direct assignment gives XOrigin = 1 on the Tester/Developer/Publisher row. The IT Department parent row shows XOrigin = 2 (inherited via child membership). If Kalfas is also directly assigned to IT Department, that parent row becomes 3 (1 + 2 — both bits set). Walk trainees through the arithmetic: Direct bit = 1, Inherited bit = 2, both active = 3. The database stores this as an integer; the bit positions encode how the assignment was established, and multiple sources can be active simultaneously.
Common Mistake — Expecting Exclusive Values
Trainees consistently assume XOrigin is exclusive: either 1 (Direct) or 2 (Inherited), never both. Show the actual database rows to prove that 3 is a valid and common production value. Operationally: removing a direct assignment when XOrigin = 3 does not delete the PersonInRole row — it drops to 2. The person retains the role via the inherited path. This distinction is critical when troubleshooting why a role removal appears to have no effect.
XIsInEffect
If XOrigin looks correct but nothing is taking effect in the portal or provisioning, check the XIsInEffect column on PersonInRole. A value of 0 means the assignment is inactive — typically due to date restrictions or an unmet condition. XIsInEffect is separate from XOrigin and both must be 1 for the assignment to be active.
PMS-5 Application Roles and Portal Access
Goal: Create a "Portal Read-Only" Application Role; link it to the Publisher Business Role; verify that Anna Lyra can log into the portal read-only while Dimitris Kalfas cannot access the portal.

Key Steps

  • Manager → Application Roles → New → Name = "Portal Read-Only"
  • Assign appropriate read permissions to the Application Role (portal navigation nodes and view rights)
  • Open the Publisher Business Role → Application Roles tab → click Add → select "Portal Read-Only"
  • Save all changes and run DBQueue
  • In Designer, compile and deploy the Application Role configuration to the portal
  • Log into the web portal as Anna Lyra → verify she can browse content but the edit/save controls are absent or disabled
  • Log into the web portal as Dimitris Kalfas → verify he has no portal access (login fails or only default empty access)
  • Confirm the difference is entirely explained by role membership, not individual configuration
Mentor Hint
This exercise is the payoff of the entire module — role membership directly controls system access without any individual exceptions. Test with both Anna and Kalfas and have the trainee narrate why each person sees what they see, tracing back through: Person → Business Role → Application Role → Permission. This confirms the full chain is understood.
Common Mistake — Skipping Compile and Deploy
Application Role changes made in Manager do not take effect in the portal until the configuration is compiled and deployed in Designer. This step is frequently forgotten. If Anna's access looks wrong after the exercise, the first question is always: "Was Designer compiled and deployed after the change?"
Expected Outcome: The governance-driven role model is working. Anna Lyra has portal read-only access via the chain Publisher Business Role → Portal Read-Only Application Role. Kalfas has no portal access because the Tester Business Role has no Application Role grants. Access is controlled entirely by role membership, not individual assignment.

06 Troubleshooting Reference

# Symptom Likely Cause Resolution
1 Role assignment not visible on account DBQueue not processed after the assignment was saved Run DBQueue Processor manually, or wait for the job service cycle; then refresh the Person view
2 XOrigin shows 0 despite assignment existing Assignment not active — XIsInEffect = 0 Check effective start/end dates and conditions on the PersonInRole record; ensure XIsInEffect = 1
3 CSV employee creation silently fails Mandatory Person fields missing or data type mismatch Check OIM server error log; validate that all required fields (e.g. LastName) are populated
4 Child role not inheriting from parent Inherit assignments flag not enabled on parent role Open parent Business Role → enable "Inherit assignments" → save → re-run DBQueue
5 External employee receiving internal provisioning IsExternal flag not set on the Person object Open Person → set IsExternal = Yes → save → re-trigger provisioning via DBQueue
6 Application Role not granting portal access after assignment Designer compile and deploy not run after the change Open Designer → compile Application Role configuration → deploy to portal → retest login
7 All employees appear to have the same access level Role hierarchy not configured; all roles sitting at root level with no parent Verify parent-child relationships on each role; confirm Application Roles are linked to the correct Business Roles only
