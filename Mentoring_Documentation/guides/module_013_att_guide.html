Attestations (ATT)

Build complete attestation systems using workflows, approval policies, SQL-based approver lookup, and process-driven email notification on membership loss.

Key Concepts

  • Attestation Policy
  • Attestation Approval Workflow
  • Approval Procedure
  • dbo.CCC_IsInternalManager SQL function
  • UID_QERTermsOfUse / UID_QERJustificationDeny
  • Event: CCC_SystemRole_Membership_Lost
  • Mail templates + process automation

Module Objectives

  • Create end-to-end attestation from scratch
  • Use custom SQL for dynamic approver resolution
  • Configure auto-denial when no approver found
  • Trigger email notification on workflow denial
  • Enforce mandatory denial reason via web designer

Prerequisites

  • Module 011 (WFL) complete — especially WFL-3 for the SQL function
  • Understanding of OIM approval workflows
  • OIM Designer access
  • Web Designer access (for mandatory reason config)

Exercise Progression

  • ATT-1: Simple manager attestation, auto-denial on no manager
  • ATT-2: SQL function resolves first internal manager
  • ATT-3: Extended workflow + email on denial event

Exercises

ATT-1

Basic Attestation with Manager Approval

Build a complete attestation flow from scratch: approval workflow, approval policy, and attestation policy. The attestation engine automatically denies when no approver is found.

Requirements

Build a complete attestation from scratch, demonstrating both the happy path (manager approves) and the auto-denial path (no manager assigned to identity).

Solution Steps
  1. Step 1 — Create Approval Workflow for Attestations: In OIM Designer, create a new approval workflow. Add an approval step pointing to the identity's manager (UID_PersonHead). Configure the step to auto-deny if no approver is found.
  2. Step 2 — Create Approval Policy: Create a new Approval Policy and assign the workflow created in Step 1. This policy will be referenced by the attestation policy.
  3. Step 3 — Create Attestation Policy: Create a new Attestation Policy. Set the attestation object (e.g., system role membership), assign the Approval Policy from Step 2, and configure the attestation schedule or run manually.
Mentor Note When the approval workflow step finds no approver (no manager on the identity), OIM's attestation engine will automatically deny the attestation and remove the membership. This is built-in behaviour — trainees do not need to write custom logic for auto-denial.
Test Cases
#Test ScenarioStepsExpected ResultResult
1 Run attestation for assignment with manager Execute the attestation policy for an identity that has a manager assigned Manager receives an attestation task and can approve/deny PASS
2 Run attestation for assignment without manager Execute the attestation policy for an identity that has NO manager assigned Attestation is automatically denied and the system role membership is removed PASS
Verification Tip After running the attestation for a manager-less identity, check the attestation run logs. You should see the auto-denial event recorded. Also verify the system role membership has been physically removed from the identity.
ATT-2

Attestation with SQL Function for Internal Manager Lookup

Replace simple manager lookup with a custom SQL function dbo.CCC_IsInternalManager that climbs the management chain to find the first internal manager. When no internal manager is found, the task falls back to the Operation Engineers group.

Requirements

Use the custom SQL function (same as used in WFL-3) to find the first internal manager in the hierarchy. When no internal manager exists anywhere in the chain, the "Operation Engineers" group handles the attestation.

Reuse from WFL-3 If trainees completed WFL-3, the dbo.CCC_IsInternalManager SQL function already exists on the SQL Server instance. They can create the Approval Procedure pointing to it directly — no need to recreate the function.
Solution Steps
  1. Step 1 — Create SQL Function on SSMS: Create (or verify) the dbo.CCC_IsInternalManager scalar function on the OIM database. The function takes a UID_Person parameter and walks the management chain to return the UID of the first internal manager found.
  2. Step 2 — Create Approval Procedure: In OIM Designer, create a new Approval Procedure that calls dbo.CCC_IsInternalManager. Configure it to fall through to "Operation Engineers" when the function returns NULL (no internal manager found). Enable "deny if no approver found" for the fallback step.
  3. Step 3 — Create new Workflow: Create a new Attestation Approval Workflow (same structure as ATT-1 but referencing the new approval procedure from Step 2).
  4. Step 4 — Create new Approval Policy: Create a new Approval Policy using the workflow from Step 3. This separates ATT-2 from ATT-1's configuration so both can be tested independently.
Test Cases
#Test ScenarioStepsExpected ResultResult
1 Run attestation with internal manager in chain Execute attestation policy for an identity whose management chain contains at least one internal manager The first internal manager found in the chain receives the attestation task PASS
2 Run attestation with no internal manager in chain Execute attestation policy for an identity whose entire management chain has no internal managers Operation Engineers group receives the attestation task PASS
3 Deny attestation Log in as the approver (manager or Operation Engineers member) and deny the attestation The system role membership is removed from the identity PASS
Important The SQL function walks UP the management chain — it checks the identity's manager, then their manager, and so on. Make sure test identities have a correctly configured management chain in OIM before running this exercise.
ATT-3

Full Attestation with Mail Notification on Denial

The most complete attestation scenario: an extended workflow with extra approval and deny steps, a custom mail template, and a process that fires on the CCC_SystemRole_Membership_Lost custom event to send an email notification when a membership is removed via denial.

Requirements

Complex workflow with extra approval/deny steps, plus email notification triggered by a custom event (CCC_SystemRole_Membership_Lost) on the attestation workflow denial.

Solution Steps
  1. Step 1 — Create new Workflow with extra steps: Build an attestation approval workflow with an extra step on the APPROVE path and an extra step on the DENY path. The deny path step should raise the custom event CCC_SystemRole_Membership_Lost when denial is completed.
  2. Step 2 — Create new Approval Policy: Create a new Approval Policy using the ATT-3 workflow. Keep this separate from ATT-1 and ATT-2 policies.
  3. Step 3 — Create new Attestation Policy: Create a new Attestation Policy referencing the ATT-3 Approval Policy. Configure the same system role membership object as previous exercises.
  4. Step 4 — Create mail template: In OIM Designer, create a mail template for the "membership removed" notification. The template should include the identity name, the system role name, and the denial reason. Use the UID_QERTermsOfUse parameter if applicable.
  5. Step 5 — Create process on event: Create a new process in OIM Designer triggered by the event CCC_SystemRole_Membership_Lost. Add a process step that sends the mail template created in Step 4 to the relevant recipient (e.g., the identity whose membership was removed).
Mentor Note — Custom Events on Workflow Steps The event CCC_SystemRole_Membership_Lost is a CUSTOM event — it does not exist by default in OIM. Show trainees where to register custom events on workflow steps (in the step's "Event" or "Raise Event" configuration). The process listens for this event and fires when it is raised by the workflow denial step.
Test Cases
#Test ScenarioStepsExpected ResultResult
1 Run attestation with internal manager in chain Execute the ATT-3 attestation policy for an identity with an internal manager Manager receives attestation task; extra approval step visible in workflow PASS
2 Run attestation with no internal manager Execute for identity with no internal manager in chain Operation Engineers receive the attestation task as fallback PASS
3 Deny attestation — verify membership removed AND email sent Deny the attestation as the assigned approver Membership is removed from the identity AND an email notification is received (check mail client) PASS
Verification Steps for Test Case 3 After denial: (1) Check the identity's system role memberships — the role should be gone. (2) Check the OIM job queue or process log — the process for CCC_SystemRole_Membership_Lost should show as completed. (3) Check the email client for the notification.

Configuration: Mandatory Standard Reason When Denying

CFG

Enforce Mandatory Denial Reason in Web Portal

This is a Web Designer configuration change (not a Designer change) that makes the denial reason field mandatory when an approver denies an attestation in the web portal.

Web Designer vs Designer This change is made in the Web Designer (web portal configuration editor), NOT in the OIM Designer (database object editor). These are two separate tools. Make sure trainees understand the distinction before proceeding.
Configuration Steps
  1. Open the Web Designer and navigate to the VI_Attestation module.
  2. Locate the UID_QERJustificationDeny node within the VI_Attestation module.
  3. Add an extension in the CCC module (custom module) — do not modify the base VI_Attestation module directly.
  4. Copy all elements from the base UID_QERJustificationDeny node into the CCC extension.
  5. In the copied element, set the "minimum character count" property to 1.
  6. Save and rebuild / deploy the web portal for the change to take effect.
Effect Setting minimum character count to 1 makes the reason field required — the approver cannot submit a denial without entering at least one character in the reason field. This applies to all attestation denials in the web portal.
Mentor Note After making the Web Designer change, you must rebuild the web portal for the change to be visible. Show trainees the rebuild/deploy process. Also demonstrate what happens before the change (reason optional) vs. after (reason required) by attempting to deny without a reason.

Mentor Tips

ATT-1: Auto-Denial Behaviour The "no manager = automatic denial" behaviour is built into OIM's attestation engine. When the approver query returns no one, OIM auto-denies. Show trainees this in the attestation run logs — they can see the decision reason. No custom code is needed for this. Emphasise this is a feature, not a bug.
ATT-2: Reusing the SQL Function from WFL-3 The SQL function dbo.CCC_IsInternalManager was introduced in WFL-3. If trainees completed WFL-3 first, they can reuse the existing function — no need to recreate it. This is a good opportunity to show how custom SQL assets are reused across multiple OIM features (workflows AND attestation). If they skipped WFL-3, they need to create the function now.
ATT-3: Registering Custom Events on Workflow Steps The event CCC_SystemRole_Membership_Lost is a CUSTOM event that must be explicitly registered. Show trainees the workflow step editor and where the "raise event" or "event name" field is. Once the event is raised by the denial step, the OIM process engine picks it up and fires the associated process. Walk through the full chain: denial step fires event → process engine receives event → process sends email.
Mandatory Reason: Web Designer vs Designer The web designer change for mandatory denial reason is a web configuration change, not a Designer change. Show trainees both tools and explain the difference: Designer = database objects (workflows, policies, processes); Web Designer = web portal UI configuration (forms, fields, layouts). Both are part of OIM but serve different purposes.
General: Attestation Run Logs Always show trainees how to read attestation run logs. After each test case, pull up the run log and point out: who was selected as approver, when the task was created, when it was resolved, what decision was made. This builds confidence in debugging attestation issues.

Troubleshooting

ProblemLikely CauseSolution
Attestation not generating tasks Attestation Policy has not been executed — policies must be manually run or scheduled; they do not fire automatically Check if the policy has been executed. Navigate to the attestation policy and run it manually from the policy page. Check the OIM job queue for any errors during execution.
Auto-denial not happening for identity without manager The approval procedure is not configured to auto-deny when no approver is found Open the Approval Procedure in Designer. Find the "deny if no approver found" or equivalent option and enable it. Re-run the attestation to verify auto-denial now occurs.
Email not sent on denial (ATT-3) The process for the CCC_SystemRole_Membership_Lost event is not correctly configured or the event is not being raised by the workflow step Verify the workflow deny step is configured to raise the CCC_SystemRole_Membership_Lost event. Check the process in Designer — ensure it is set to trigger on this event. Check the OIM job queue for process execution errors. Verify mail step configuration.
Mandatory reason not enforced in portal The Web Designer change has not been deployed — a web portal rebuild is required after any Web Designer change Trigger a web portal rebuild after the Web Designer change. Clear browser cache and reload. Test again by attempting to deny without entering a reason.
SQL function not found (ATT-2) dbo.CCC_IsInternalManager was not created or was created in the wrong database Open SSMS, connect to the OIM database, and run SELECT OBJECT_ID('dbo.CCC_IsInternalManager'). If NULL, the function does not exist — create it using the WFL-3 script.
Operation Engineers fallback not triggering (ATT-2) The approval procedure fallback step is not configured correctly, or the Operation Engineers group UID is wrong Open the Approval Procedure in Designer. Check the fallback step — verify the Operation Engineers group UID or dynamic role is correctly assigned as the fallback approver group.
