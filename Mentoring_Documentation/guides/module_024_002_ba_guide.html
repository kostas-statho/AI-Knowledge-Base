Module 024_002 — Bulk Actions

Advanced track: Angular front-end + C# CompositionAPI back-end + Git workflow

Architecture Overview — 4-Endpoint HTTP Contract

The Angular front-end calls these endpoints in sequence for every bulk CSV action. Understanding this contract is essential before writing any endpoint code.

  1. /csvtemplate — Returns CSV column headers as a downloadable template file
  2. /startvalidate — Pre-flight check; returns { message, permission, collectImportData? }
  3. /validate — Per-row validation; returns an array of { column } or { column, errorMsg }
  4. /startaction, /action, /endaction — Per-row processing and post-flight summary
BA_01 Full Environment Setup
Requirements
Set up the complete development environment: Node.js, Angular CLI 18.2.14, the IMX Angular source (v93), the bulk-actions plugin configuration, Git, and the C# plugin DLL in both required locations. The environment must support building and serving the Angular app and deploying C# plugins.
Solution Steps
  1. Install Angular CLI globally: npm install -g @angular/cli@18.2.14
  2. Verify installed versions of Node.js and Angular CLI are compatible.
  3. Download IdentityManager.Imx-93 from GitHub using tag v93.
  4. Copy the bulk-actions project folder into IdentityManager.Imx-93\imxweb\projects\bulk-actions.
  5. Modify qer-app-portal\project.json to register the bulk-actions module.
  6. Modify imxweb\tsconfig.json to include the bulk-actions path mapping.
  7. In angular.json: copy the rmb block and rename ALL occurrences of "rmb" to "bulk-actions" — use VS Code Find & Replace All to avoid missing any instance.
  8. In the OIM imxweb folder: copy the Html_aob directory to bulk-actions; copy imx-plugin-config.json to imxweb\bulk-actions\imx-plugin-config.
  9. Download and install Git for Windows.
  10. Open Git Bash in the imxweb folder: run git init, create a branch, git add ., commit with a descriptive message, then git push.
  11. Create a new directory BulkActions_SC, then run git clone https://github.com/kostas-statho/BulkActions_V93.git. Verify files and the correct branch are present.
  12. Open BulkActions_SC\BulkActions_V93\imxweb in VS Code and run npm install --skip-dialog.
  13. Copy BulkActions93.CompositionApi.Server.Plugin.dll to both C:\Program Files\One Identity\One Identity Manager AND C:\inetpub\wwwroot\ApiServer\bin.
  14. Run npm run nx:build-all. If any package is missing, install it manually: npm install <package>.
  15. Create a RoleBased permission group CCC_BulkActions.
  16. Create an AERole and assign the permission group to it.
  17. Login to ITShop with an Identity that has the CCC_BulkActions group and verify the Bulk Actions feature is accessible.
Test Cases

No formal test table for BA_01. Verify environment by confirming each setup step completes without errors, the Angular build succeeds, and the ITShop login with the test identity shows the Bulk Actions feature.

Mentor Tips / Troubleshooting
Mentor Notes
  • npm run nx:build-all takes 10–20 minutes on the first run — advise trainees to be patient.
  • If packages are missing after the build, install them one by one with npm install <package>; there is no batch shortcut.
  • The DLL must be copied to both the OIM install directory and ApiServer\bin. Missing either location causes intermittent failures that are hard to diagnose.
  • In angular.json Step 7, ALL occurrences of "rmb" must be renamed. Use VS Code Find & Replace All (Ctrl+Shift+H) to guarantee nothing is missed.
  • Avoid spaces in any folder path along the development chain — Angular CLI and some npm scripts fail silently with spaces in paths.
BA_02 First Bulk Action Endpoint
Requirements
Create the first working C# endpoint for a bulk action. The endpoint must follow the IApiProvider + IApiProviderFor<PortalApiProject> pattern, register correctly with the plugin, build without errors, and respond in ITShop.
Solution Steps
  1. Create the proper endpoint class implementing IApiProvider and IApiProviderFor<PortalApiProject>.
  2. Build the solution and copy the updated .dll to the Api Server bin directory.
  3. Restart the API Server application pool and verify functionality in ITShop.
Test Cases

No formal test table. Verify the endpoint responds correctly in ITShop by triggering the action from the Bulk Actions UI and confirming the expected response is returned.

Mentor Tips / Troubleshooting
Mentor Notes
  • All endpoint classes must use namespace QBM.CompositionApi.
  • Endpoint URL prefix must be webportalplus/ — verify this matches what the Angular component calls.
  • After deploying a new DLL, always recycle the API Server application pool in IIS.
BA_03 Validation Endpoint
Requirements
Implement the per-row validation endpoint. For each CSV row, the endpoint must return an array of objects: { column } for a valid field, or { column, errorMsg } when validation fails.
Solution Steps
  1. Create the /validate endpoint returning the correct per-row validation array structure.
  2. Build the solution and copy the updated DLL to the Api Server.
  3. Verify functionality by uploading a test CSV in ITShop and confirming per-row errors are displayed.
Test Cases

No formal test table. Validate by uploading CSVs with known bad rows and confirming errors surface in the correct columns in the ITShop UI.

Mentor Tips / Troubleshooting
Mentor Notes
  • The errorMsg field must be present on invalid rows only — sending an empty string is not the same as omitting the field.
  • All DB calls inside the endpoint must use .ConfigureAwait(false).
BA_04 Action Endpoint
Requirements
Implement the per-row /action endpoint that performs the actual database write or delete using a UnitOfWork. The endpoint must commit changes transactionally and handle errors gracefully.
Solution Steps
  1. Create the /action endpoint using qr.Session.StartUnitOfWork(), u.PutAsync() or u.DeleteAsync(), and u.CommitAsync().
  2. Build the solution and copy the updated DLL to the Api Server.
  3. Verify functionality by processing a valid CSV row and confirming the DB change.
Test Cases

No formal test table. Verify by running a full action through the ITShop UI and confirming the expected records are created or modified in the OIM database.

Mentor Tips / Troubleshooting
Mentor Notes
  • Always using var u = qr.Session.StartUnitOfWork(); — the using block ensures disposal even on exception.
  • Use $"..." string interpolation (not string.Format) for strings with \n line breaks — Format with verbatim strings produces \\n literally.
BA_05 Full Identity Import with Validation
Requirements
Build a complete identity import bulk action with validation. Enforce OIM rules: Personnel number must be unique, LastName is mandatory. Successfully import single and multiple complete rows. Permissions must be correctly assigned so the importing user has write access to the Person table.
Solution Steps
  1. Assign the required permissions to the importing identity so it can write to the Person table.
  2. Create the two required endpoints: the validation endpoint and the action endpoint, with rules for Personnel number uniqueness and LastName being mandatory.
Test Cases
# Test Scenario Test Steps Expected Results Mentor Notes
1 Import duplicate user Upload a CSV containing a Personnel number that already exists in OIM Validation error: "Value in field Personnel number must be unique" Personnel number uniqueness is enforced at OIM DB level; check the Person table directly if in doubt.
2 Import user without last name Upload a CSV row with the LastName field empty or missing Validation error: "Last Name is a mandatory field." LastName (CentralAccount) is marked mandatory in the OIM Person table schema.
3 Import user with all fields Upload a single complete CSV row with all required fields populated Successful import — identity appears in OIM Verify the new Person record in Designer or ITShop after import.
4 Import multiple users with all fields Upload a CSV with multiple complete rows (at least 3) All rows import successfully Check that row count in the summary message matches the number of CSV rows processed.
Mentor Tips / Troubleshooting
Mentor Notes
  • Personnel number uniqueness is enforced at the OIM level — the validation endpoint should query the Person table before accepting the row.
  • If the importing user lacks write permission on the Person table, the action endpoint will throw an authorization exception rather than a user-friendly error.
  • Ensure the /startvalidate endpoint correctly signals that data collection (import) is enabled via collectImportData.
BA_07 Add Entitlement Bulk Action
Requirements
Implement a complete six-endpoint bulk action for adding entitlements. All required Configuration Parameters must be created. Each endpoint URL must exactly match what the Angular component calls.
Solution Steps
  1. Create all required Configuration Parameters in OIM Designer for the Add Entitlement feature.
  2. Create and register all six required endpoints:
    • webportalplus/addentitlement/csvtemplate
    • webportalplus/addentitlement/startvalidate
    • webportalplus/addentitlement/validate
    • webportalplus/addentitlement/startaction
    • webportalplus/addentitlement/action
    • webportalplus/addentitlement/endaction
Test Cases
# Test Scenario Test Steps Expected Results Mentor Notes
1 Import existing membership Upload a CSV where the identity already has the listed entitlement assigned Validation error: "Assignment already exists." Check ESetHasEntitlement or relevant assignment table before inserting.
2 Import for non-existent Person Upload a CSV with a Personnel number that does not exist in any department Error: "Identity does not exist in the Department." The lookup should scope to the user's department context as configured in Config Parameters.
3 Import for non-existent Eset Upload a CSV with an entitlement name/ID that does not exist or is not permitted Error: "Entitlement does not exist or the assignment is not permitted." Verify the Eset exists AND that assignment is allowed per OIM configuration.
4 Import proper memberships Upload a valid CSV with correct Personnel numbers and valid entitlements Successful import — assignments created in OIM Confirm assignments in Designer or the Entitlement view in ITShop.
Mentor Tips / Troubleshooting
Mentor Notes
  • Endpoint URL prefix webportalplus/addentitlement/ must match exactly what the Angular component calls — typos cause silent 404 errors that are difficult to trace.
  • Config Parameters drive the department scope and Eset lookup — review them carefully if validation errors are unexpected.

  2. Create and register all six required endpoints:

    • webportalplus/removeentitlements/startaction
    • webportalplus/removeentitlements/action
    • webportalplus/removeentitlements/endaction
Test Cases
# Test Scenario Test Steps Expected Results Mentor Notes
1 Remove non-existent membership Upload a CSV with an assignment that does not currently exist for the identity Error: "Entitlement is not assigned to this identity." Validate the assignment exists in the relevant table before attempting delete.
2 Non-existent Person Upload a CSV with an invalid/unknown Personnel number Error: "Identity does not exist in the Department." Same person-lookup logic as BA_07 — share/reuse if possible.
3 Non-existent Eset Upload a CSV with an invalid entitlement identifier Error: "Entitlement does not exist" Check Eset lookup returns null before proceeding to assignment check.
4 Remove proper memberships Upload a valid CSV where listed assignments genuinely exist for the identities Successful import — assignments removed from OIM Confirm assignments are deleted in Designer or ITShop after processing.
Mentor Tips / Troubleshooting
Mentor Notes
  • Endpoint URL prefix webportalplus/removeentitlements/ — note the plural "removeentitlements" — must match Angular exactly.
  • The action endpoint uses u.DeleteAsync(entity, ct); ensure the entity is loaded first with TryGetAsync before deletion.
