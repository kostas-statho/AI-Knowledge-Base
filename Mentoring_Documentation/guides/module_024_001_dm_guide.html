Module 024_001 — Deployment Manager

Two related sub-topics: PowerShell OIM Console Mode (PSM) for direct DB interaction via scripts, and Deployment Manager (DM) for automated, repeatable deployment of OIM customizations from structured task folders.

01 Module Overview

PSM — PowerShell + OIM Console Mode

Direct OIM database interaction using the OIM PowerShell module in Console Mode. Queries, updates, and automation scripts run against a live OIM session. Changes are immediate and permanent.

  • PSM-01: Console Mode basic queries
  • PSM-02: Create .ps1 automation file
  • PSM-03: Config-controlled identity deactivation

DM — Deployment Manager

Automated deployment of OIM customizations from version-controlled task folders. Each task is a structured combination of XML definitions, CSV data, and templates. Supports dry runs, selective execution, and compile orchestration.

  • DM-01 through DM-03: Install and configure
  • DM-04 through DM-07: Core deployment patterns
  • DM-08 through DM-11: Advanced tasks
PSM Console Mode Warning
PSM Console Mode establishes a live OIM database session. Changes made via PutAsync / CommitAsync are applied immediately with no built-in undo. Always run PSM exercises against a dev or lab OIM instance, never production.

PSM PSM — PowerShell Console Mode

PSM-01 Console Mode Basic Queries
Goal: Connect to OIM in Console Mode and perform basic identity queries using the OIM PowerShell session object. Learn to find multiple records, find a single record, query via the Query object, and update a property.
Requirements
  • OIM PowerShell module available and importable
  • Console Mode session established against the lab OIM instance
  • Queries use WHERE clause (mandatory; full table scan is prohibited)
Solution Steps
  1. 1Run OIM session in Console Mode
  2. 2Find all Persons using a WHERE clause filter
  3. 3Find a single Person record by specific criteria
  4. 4Retrieve a record via the Query object pattern
  5. 5Update a property on a retrieved entity and commit
Verification
Each step returns the expected records in the console output. The property update is confirmed by re-querying the entity and checking the updated value.
Mentor Tips / Troubleshooting
  • WHERE clause is not optional — querying without a filter can return the entire Person table and cause performance issues
  • If the session drops during a long query, it means the OIM Connection Timeout was reached; use a more specific filter
  • Show the difference between GetAsync (returns a collection) and loading a specific entity by UID — trainees often mix these up
PSM-02 Create a .ps1 OIM Automation File
Goal: Move Console Mode commands into a reusable .ps1 script file. Establishes the pattern for all subsequent PSM automation: import module → connect → query/update → commit → disconnect.
Requirements
  • Script file structured with module import, session setup, logic, and cleanup
  • Script executes without errors from PowerShell 7
  • Output is visible in the console when the script runs
Solution Steps
  1. 1Create a .ps1 file with the OIM automation script structure, encapsulating the PSM-01 logic in a runnable file
Verification
Script runs to completion from the PowerShell 7 terminal without errors. Output matches what was produced interactively in PSM-01.
Mentor Tips / Troubleshooting
  • If the module fails to import, check the $env:PSModulePath includes the OIM module directory
  • Execution policy may block the script on first run; Set-ExecutionPolicy -Scope Process -ExecutionPolicy Bypass in PS7 for the lab session
PSM-03 Config Parameter Controlled Identity Deactivation
Goal: Create a config parameter in OIM that controls which Primary Department triggers identity deactivation. Build a PowerShell script that reads this config parameter and deactivates matching identities. Verify that changing the config re-enables those identities.
Requirements
  • Config parameter created in OIM Designer under a custom path
  • Script reads the config parameter value at runtime (not hardcoded)
  • Script deactivates all identities whose Primary Department matches the config value
  • Changing the config parameter value causes the script to re-enable previously deactivated identities
Solution Steps
  1. 1Create a config parameter in OIM Designer (e.g., Custom\PSM\DeactivateDept) with a test department value
  2. 2Write the .ps1 script that reads the config parameter, queries Person by Primary Department, and sets IsInActive accordingly
  3. 3Verify one or more identities with the matching Primary Department exist in the lab
  4. 4Execute the script and confirm target identities are deactivated
  5. 5Change the config parameter value to a non-existent department; re-execute; verify previously deactivated identities are re-enabled
Verification
After Step 4: target identity is deactivated (IsInActive = 1) in OIM Manager. After Step 5: the same identity is re-enabled (IsInActive = 0).
Mentor Tips / Troubleshooting
  • Config parameter path must be created in OIM Designer before the script can read it; if the path doesn't exist, GetConfigParmAsync returns null or throws
  • Ensure the test identity actually has the target Primary Department set — check in OIM Manager before running the script
  • Deactivation sets IsInActive = 1; re-enabling sets it back to 0; confirm the exact column name used in the target OIM version

DM DM — Deployment Manager

DM-01 Install Deployment Manager + Prerequisites
Goal: Install PowerShell 7.5.4, extract the Deployment Manager installation folder, and unblock all DLLs so the module can be imported without security errors.
Requirements
  • PowerShell 7.5.4 installed
  • DM installation folder extracted
  • All DLLs in the installation folder unblocked using Unblock-File
  • DM module imports without errors after unblocking
Solution Steps
  1. 1Extract the DM installation folder to a known location
  2. 2Install PowerShell 7.5.4 (download from Microsoft)
  3. 3Run in PS7 to unblock all DLLs:
Get-ChildItem "installation folder" -Recurse | Unblock-File
Verification
DM module loads successfully in PS7 after the Unblock-File step. Run Import-Module on the DM module path — no security or signature errors.
Mentor Tips / Troubleshooting
  • If DLLs are not unblocked, Windows marks them as downloaded from the internet and PS blocks loading — Unblock-File is the fix
  • The unblock command must be run as Administrator or with sufficient rights to modify the file Zone.Identifier alternate data stream
  • PS7 must be used, not Windows PowerShell 5 — the DM module targets PS7 APIs

DM-03 Initial Configuration

DM-04 Dry Execution
Goal: Run a controlled first deployment using PS7 and the DM module. Use the dry run (preview) capability to review planned actions before confirming execution. Check the deployment log to verify results.
Requirements
  • DM module imported in PS7
  • Dry run (without -NoExecute) shows planned actions and asks for confirmation
  • After confirmation, changes are applied to OIM
  • Log file reviewed at the specified path
Solution Steps
  1. 1Start PowerShell 7
  2. 2Import-Module the DM module; verify via Get-Command that DM commands are available
  3. 3Run the DM deployment command (dry execution) — DM will show planned actions and ask for confirmation before applying
  4. 4Confirm at the prompt; DM executes the deployment
  5. 5Check the deployment log at:
C:\Users\OneIM\Desktop\Deployment\Logs\Example\Deployment.log
Critical — Never Use -NoExecute
Do NOT use -NoExecute as a "safe" mode. The -NoExecute flag marks all changes as already done without actually applying them to OIM — this corrupts the deployment state. The DM already asks for confirmation before executing; that confirmation prompt IS the dry run preview. Always use it without -NoExecute.
Verification
Deployment.log shows the actions taken. Confirm the expected changes are visible in OIM (OIM Manager or OIM Designer) after execution.
Mentor Tips / Troubleshooting
  • If a trainee accidentally uses -NoExecute, they must reset the deployment state markers before re-running; mentor should guide through this recovery
  • The confirmation prompt before execution is the intended safety mechanism — read the planned action list carefully before confirming
  • Log files are essential for diagnosing failed deployments — teach trainees to check the log before asking for help

Solution Steps
  1. 1Create the XML template file at:
    C:\Users\OneIM\Desktop\Deployment\Common\Template
  2. 2Create the CSV data file at:
    C:\Users\OneIM\Desktop\Deployment\Tasks\10-CustomObjects
  3. 3Deploy and verify the objects in OIM
Verification
Each row in the CSV produces a corresponding object in OIM. All field values match the CSV data applied via the template.
Mentor Tips / Troubleshooting
  • Template placeholders must use the same variable names as the CSV column headers — a mismatch produces empty values, not an error
  • Always check the output count: if the CSV has 3 rows, expect 3 created objects
DM-08 Push Template with Overwrite
Goal: Deploy a VB template to OIM with the overwrite flag (-o suffix in filename). Without this flag, existing templates are silently skipped. With it, the existing template is replaced. Compile and test after deployment.
Requirements
  • Template file named with -o suffix before the extension: e.g., 030-ColumnTemplate_Person-CustomProperty10-o.vb
  • Existing template with the same target must already exist in OIM for this test to demonstrate the overwrite behaviour
Solution Steps
  1. 1Create the template file with the -o suffix in the filename: 030-ColumnTemplate_Person-CustomProperty10-o.vb
  2. 2Run the DM deployment to push the template
  3. 3Compile the OIM database and test the template behaviour
Verification
The existing template content in OIM is replaced by the new content after deployment. Without the -o suffix, the template is silently skipped — demonstrate this contrast.
Mentor Tips / Troubleshooting
  • The -o suffix is embedded in the filename, not a runtime flag — it is read by DM during file processing
  • Without -o, DM detects that a template with that target already exists and silently skips it — no warning is logged; it simply doesn't update
  • This is a very common source of confusion: "I deployed but nothing changed" — always check for the -o suffix
DM-09 Deploy Config Parameter Enablement
Goal: Enable specific OIM config parameters using a template + CSV pattern with a taskConfig.xml that orchestrates post-folder compilation via CompileDatabaseOnSuccess.ps1.
Requirements
  • Template file, reference CSV, empty trigger file, and taskConfig.xml all created correctly
  • taskConfig.xml sets aftercontainer to run CompileDatabaseOnSuccess.ps1
  • taskConfig.xml sets noCompile = True to prevent per-file compile
  • OIM DB compile runs after the entire folder completes (not per file)
Solution Steps
  1. 1Create Template_ConfigParmsEnablement.xml at:
    ...\Deployment\Common\Template
  2. 2Create ConfigParmsEnablement.csv at:
    ...\Deployment\Common\Reference
  3. 3Create an empty file 300-ConfigParmsEnablement.csv.ref at:
    ...\Deployment\Tasks\30-EnableActions
  4. 4Create taskConfig.xml at ...\Tasks\30-EnableActions with these settings:
Setting KeyValuePurpose
intragen.deployment.aftercontainer CompileDatabaseOnSuccess.ps1 Triggers DB compile after all files in the folder complete successfully
intragen.deployment.transport.noCompile True Prevents individual per-file compiles (which would be extremely slow)
  1. 5Deploy and verify Config Parameters are enabled in OIM Designer
OIM Compile Downtime
The DB compile triggered by CompileDatabaseOnSuccess.ps1 (aftercontainer) takes time. OIM will NOT be accessible via the web portal or OIM Manager during compilation. Plan for this downtime window when scheduling this deployment.
Verification
Config Parameters listed in the CSV are now Enabled in OIM Designer. DB compile completed successfully (no errors in the compile log).
Mentor Tips / Troubleshooting
  • aftercontainer fires once after ALL files in the folder succeed — if any file fails, the compile does not run
  • noCompile = True is critical performance protection — without it, OIM compiles after every single transport file, which can take many minutes per file
  • The empty .csv.ref file in the Tasks folder is the trigger that tells DM to process this CSV against the template — without it, nothing happens

03 Mentor Tips & Key Concepts

DM-04 Critical — Never Use -NoExecute
-NoExecute marks all tasks as DONE without applying any changes to OIM. This corrupts the deployment state — subsequent runs will see those tasks as already completed and skip them. Recovery requires manually resetting the state file. The DM already shows a confirmation prompt; that IS the safe preview mechanism.
DM-08 — Overwrite Suffix Is Silent Without -o
Without the -o suffix in the template filename, DM silently skips any template that already exists in OIM. No error, no warning, no log entry. The -o suffix in the filename is the only way to force a replace. This catches many trainees who wonder why their changes didn't apply.
DM-09 — aftercontainer vs Per-File Compile
aftercontainer fires CompileDatabaseOnSuccess.ps1 once after the entire folder completes. noCompile = True prevents a compile after each individual transport file. Always use both together for folders that contain multiple transport files — per-file compile would make a 20-file folder take 20x as long.
PSM — Live Database, No Undo
PSM Console Mode is a live OIM database session. Every CommitAsync writes permanently to the OIM database. There is no undo button. Always use a dev/lab OIM instance, never production. Establish this expectation before PSM-01 begins.
Config File Hierarchy
masterProjectConfig.xml holds global settings for all projects. projectConfig.xml is a per-project override — values here take precedence over master. When something behaves unexpectedly, check projectConfig for overrides before modifying master.
