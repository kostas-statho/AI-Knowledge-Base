Connectors (CNR)

Two separate connector tracks: (1) Generic/CSV connectors using OIM-to-OIM and CSV sync projects; (2) Active Directory connector for real AD sync. Covers read-only sync, bidirectional mapping, custom scripts, and outstanding object handling.

Key Concepts

  • Sync Project, CSV Connector, OIM-to-OIM Connector
  • Object Type, Mapping, Workflow (sync direction)
  • Provisioning, Outstanding Objects
  • Single Object Operation (push on demand)
  • Startup Configurations
  • Deletion sync patterns (markForDeletion)

Two Tracks

  • CSV/OIM Track (CNR-1 to CNR-4): Staging tables, ESet creation from CSV, IT Shop products via CSV, custom script connectors
  • AD Track (CNR_AD-1 to CNR_AD-3): Read-only initial sync, bidirectional mapping, single object operation for AD group provisioning
CSV / OIM Connector Track

CNR-1: CSV Connector — Read Data into CCCStaging Table

1

CSV Connector — Read Data into CCCStaging

The foundational connector exercise. Reads data from a CSV file into a custom staging table in the OIM database. All subsequent CNR exercises build on this staging table.

Before You Start — Table First Create the CCCStaging table FIRST — either via SQL or schema extension — before opening Sync Editor. The table must exist in the OIM database before the connector can discover and map to it. A common trainee mistake is opening Sync Editor first and not finding the table in the schema list.
Solution Steps
  1. Step 1 — Create CCCStaging Table: Create the staging table via schema extension (preferred in OIM) or directly via SQL. Define all necessary columns to match the CSV file structure — column names, data types, and key fields.
  2. Step 2 — Create CSV Connector with CSV System: In Sync Editor, create a new Sync Project using the "CSV Connector" type. Create and configure a new "CSV System" pointing to your source CSV file. Configure the file path, delimiter, and encoding.
  3. Step 3 — Setup OneIM Manager Connection: Configure the OIM side of the connection — connection string, credentials, and the target OIM database.
  4. Step 4 — Setup Target System: Configure the target system schema to reflect the CSV file structure and the CCCStaging table columns. Update the schema so OIM discovers the CCCStaging table.
  5. Step 5 — Create a Base Object: Define the Base Object to scope which records from the CSV file the sync will process. Configure any filter conditions if needed.
  6. Step 6 — Create Mapping: Map CSV columns to CCCStaging table columns. Pay close attention to key properties and matching rules — these determine how records are matched between source and target.
  7. Step 7 — Create Workflow with Step: Create a sync workflow with the appropriate direction step (CSV source → OIM CCCStaging target) to populate the staging table on each run.
Test Cases
#Test ScenarioStepsExpected ResultResult
1 Run Sync Execute the CSV sync project Data from the CSV file is populated into the CCCStaging table — verify row count matches CSV PASS

CNR-2: OIM-to-OIM Connector — Create ESet from CSV

2

OIM-to-OIM Connector — Create ESet from CSV (Multi-Sync-Project)

The most complex exercise in this module. A multi-sync-project architecture: the CSV sync project populates the staging table (CNR-1), then a second OIM-to-OIM sync project creates ESet objects from that staging data. A third deletion sync project handles cleanup of removed records.

Complexity Warning — Three Sync Projects This exercise involves three separate Sync Projects working together in sequence. Draw the complete data flow diagram with trainees BEFORE starting any configuration. Running the projects out of order will produce incorrect results.
Data Flow
CSV File --> CSV Sync Project (CNR-1) --> CCCStaging Table --> OIM Connector Sync Project --> ESet Objects in OIM

Deletion Sync Project (3rd) --> markForDeletion + cleanup of PersonHasEset
Solution Steps
  1. Step 1 — Create Sync Project "CCC_CreateSR": Create a new Sync Project using the One Identity Manager connector type. Name it CCC_CreateSR.
  2. Step 2 — Setup Connections & Update Schema: Configure both the OneIM Manager connection (source — the CCCStaging table) and the Target System connection (destination — the ESet object type). Run schema update to discover all available tables and columns.
  3. Step 3 — Create Base Object: Define the Base Object scoping which records from the CCCStaging table will drive ESet creation.
  4. Step 4 — Create Mapping: Map CCCStaging fields to ESet properties. Configure the key/match property to determine how staging records are matched to existing ESet objects.
  5. Step 5 — Create Workflow: Create the provisioning workflow with the correct direction step (CCCStaging → ESet, OIM-to-OIM).
  6. Step 6 — Create Startup Configurations: Configure Startup Configurations for scheduled or on-demand execution of the sync.
  7. Step 7 — Create Deletion Sync Project: Create a separate third sync project for deletion handling. Configure two filtered schemas within it:
    • PersonHasEset_Active: All identities assigned to active ESets that were created from the CSV staging data
    • PersonHasESet_Filtered_Outstanding: All identities assigned to ESets from CSV that are currently markedForDeletion
  8. Step 8 — Configure Deletion Workflows: Two workflows in the deletion sync project:
    • "Create ESet" workflow: Configured to set markForDeletion = true on ESets that no longer have a corresponding record in the CCCStaging table
    • "Delete_Removed_Eset" workflow: Configured to delete PersonHasEset rows belonging to ESets that are markedForDeletion, then delete the ESet itself
Test Cases
#Test ScenarioStepsExpected ResultResult
1 Run Sync Execute the OIM connector sync project (after CNR-1 CSV sync has run) ESet objects are created from staging data and appear in the ESet table PASS
2 Update CSV, run both syncs in sequence Modify CSV data (add/remove entries); run CSV Sync first; then run OIM Table Sync Both the CCCStaging table and ESet table are updated; ESets no longer in CSV are marked markedForDeletion PASS
3 Run Sync with markedForDeletion ESets assigned to identities Run the Table Sync when ESets exist in markedForDeletion state with active identity assignments PersonHasEset assignment rows are deleted; ESets are then fully deleted from the system PASS

CNR-3: Products in IT Shop from CSV

3

Products in IT Shop from CSV

Sync IT Shop products (Service Items / AccProducts) directly from a CSV file, including a deletion sync to clean up products that are removed from the CSV source.

Solution Steps
  1. Step 1 — Create Mappings: In Sync Editor, create mappings from the CSV columns to the IT Shop product fields (Service Item / AccProduct properties — name, description, shelf assignment, etc.).
  2. Step 2 — Create Workflows for Population: Create the sync workflow so that after running the sync, products are populated in the IT Shop. Verify after running that products appear correctly on the configured IT Shop shelf and are requestable.
  3. Step 3 — Create Deletion Mapping and Workflow: Create the deletion-direction mapping and corresponding workflow named Delete_Sync (or equivalent). After running this deletion sync, products that were removed from the CSV file are no longer available in the IT Shop — they are deactivated or removed from the shelf.
Mentor Note — Run Order Matters The deletion sync must be run AFTER the main CSV sync. The main sync first updates the staging/target state, then the deletion sync identifies products that no longer exist in CSV and removes them. Running deletion sync first will delete everything.
Test Cases
#Test ScenarioStepsExpected ResultResult
1 Run Sync Execute the CSV sync project Products are populated and visible in the IT Shop on the configured shelf PASS
2 Remove products from CSV, run delete sync Delete entries from the CSV file; run CSV Sync to update staging; run AccProdSync (deletion workflow) Products corresponding to the deleted CSV rows are no longer available in the IT Shop PASS

CNR-4: Custom Script Connector — Disable Missing ESets

4

Custom Script Connector — Disable Missing ESets

Instead of deleting ESets that are missing from the staging table, use a custom script connector to disable them. Introduces writing custom connector logic that runs within the sync cycle.

Mentor — When to Use Custom Script vs. Standard Mapping Standard sync mappings handle INSERT, UPDATE, DELETE in straightforward ways. Sometimes you need custom logic: disabling instead of deleting, complex multi-table transformations, or conditional operations. The Custom Script connector allows VB.NET/C# code to execute during the sync cycle. Discuss with trainees when each approach is appropriate.
Solution Steps
  1. Create Custom Script via Designer: Write and register the custom script in OIM Designer. The script must contain logic to: query for ESets that do not have a corresponding record in the CCCStaging table, and set the disabled/inactive flag on those ESets.
  2. Restart Services & Update Schema: After registering the custom script in Designer, restart the relevant OIM services (Job Service / Application Server). Then run a schema update in Sync Editor to make the new custom script available as a connector option.
  3. Configure Workflow Step: In the sync workflow, add and configure a step that invokes the custom script to perform the disable operation on the identified ESets.
Service Restart Reminder Registering a new script in OIM Designer requires a service restart before the Sync Editor schema update will discover it. Trainees frequently forget the restart step and wonder why the script is not visible in Sync Editor.
Test Cases
#Test ScenarioStepsExpected ResultResult
1 Run Sync Execute the sync project with the custom script step configured ESets that are missing from the CCCStaging table are disabled (not deleted) in OIM PASS
Active Directory Connector Track

CNR_AD-1: Active Directory — Read-Only Initial Sync

AD1

Active Directory — Read-Only Initial Sync

The first step in AD integration is always a read-only sync to verify the schema and data mapping before enabling any writes. This exercise establishes the "verify read-only first" production pattern.

Never Start With Write Access Always configure the first AD sync project as Read_Only. Verify the schema and data mapping thoroughly before enabling write access. One misconfigured write mapping can corrupt AD user objects in a real environment.
Requirements and Solution
  1. Verify the AD Forest used for this exercise exists and is accessible.
  2. Create new AD Users in the Active Directory environment. Set the Description field on these users to Managed_by_OneIM — this allows easy identification of test users during sync verification.
  3. In OIM Sync Editor, create a new Sync Project using the AD Connector type. Set the connection mode to Read_Only.
  4. Configure the AD schema — map the relevant AD attributes (sAMAccountName, displayName, mail, description, etc.) to the corresponding OIM ADS Account fields.
  5. Run the initial sync to import AD objects into the OIM database. Verify the AD users appear as ADS Account objects in OIM Manager.
Verification Step After the sync completes, open OIM Manager and navigate to Active Directory accounts. Confirm the test users created with Managed_by_OneIM description are visible. Also confirm that NO changes were made to AD (read-only means zero writes).
Test Cases
#Test ScenarioStepsExpected ResultResult
1 Run Initial Sync Execute the read-only AD sync project AD data is synced into the OIM database — AD user objects appear as ADS Accounts in OIM. No changes written to AD. PASS

CNR_AD-2: Active Directory — Bidirectional Mapping

AD2

Active Directory — Bidirectional Mapping

Extends the read-only sync project to support write operations. Enables OIM-initiated changes to attributes to be provisioned out to Active Directory in real time via the sync cycle.

Solution Steps
  1. Configure Write Access: Update the AD Sync Project connection from Read_Only mode to a connection that allows write operations. Configure the service account credentials with the necessary AD permissions to write user attributes.
  2. Create Mapping: Create the attribute mapping between OIM Person/ADS Account fields and AD user attributes. Define directionality — which fields are authoritative (OIM writes to AD) and which are read from AD. Common fields: displayName, telephoneNumber, department, title.
  3. Create Provision Workflow: Create a provisioning workflow with the direction step set to OIM → AD. This workflow handles writing changes initiated in OIM out to the AD target system when the sync runs.
Mentor — Directionality Is Critical Each mapped attribute has a direction: READ (AD is authoritative), WRITE (OIM is authoritative), or BOTH. Getting this wrong causes sync conflicts. For example, if "department" is mapped as BOTH, OIM and AD may overwrite each other. Discuss the concept of system of record with trainees — which system owns each field?
Test Cases
#Test ScenarioStepsExpected ResultResult
1 Update AD User via OneIM Modify a user attribute (e.g. telephone number or title) in OIM Manager; run the sync The attribute change is provisioned to the corresponding Active Directory user object — verify in ADUC or ADSI Edit PASS

CNR_AD-3: Active Directory — Single Object Operation for Groups

AD3

Active Directory — Single Object Operation for Groups

Implements real-time "push on demand" provisioning for AD group membership. A change to group membership in OIM immediately triggers a targeted sync of only that group object to AD — no waiting for the scheduled full sync cycle.

Why Single Object Operations? Full sync cycles run on a schedule — minutes to hours. For AD group membership changes (especially security group changes), waiting for the next scheduled sync is often unacceptable. A Single Object Operation provisions the specific changed object to AD immediately when the change occurs in OIM.
Solution Steps
  1. Step 1 — Create Single Object Operation via Designer: In OIM Designer, define a Single Object Operation for the ADS Group object type. This configuration enables on-demand, per-object AD synchronization independent of the scheduled full sync cycle.
  2. Step 2 — Copy and Modify "ADS_ADSGroup_Update" Process:
    • Locate the out-of-the-box process ADS_ADSGroup_Update in OIM Designer
    • Copy it to a new custom process (do NOT modify the original)
    • Deactivate the original out-of-the-box process, leaving only the copy active
    • In the copied process, modify ONLY the PreScript to add any required custom logic. Leave all other process steps unchanged.
Copy First — Never Edit OOB Directly Never modify the original out-of-the-box process directly. Always copy it, then modify the copy, then deactivate the original. This preserves the ability to restore original behavior if needed and prevents breakage during OIM version upgrades.
Test Cases
#Test ScenarioStepsExpected ResultResult
1 Update AD group assignments Assign or remove an AD group membership for an AD account in OIM Manager; save the change After saving, the custom process is triggered immediately and the group membership change is provisioned to Active Directory in real time — no manual sync needed PASS

Mentor Tips

CNR-1 — Table Must Exist Before Sync Editor Always create the CCCStaging table BEFORE opening Sync Editor. The table must exist in the OIM database for the connector schema discovery to find it. Trainees commonly try to create the table after creating the Sync Project and then can't find the table in the schema.
CNR-2 — Draw the Data Flow First This is the most complex exercise in the module. Three separate Sync Projects must work together. Spend time drawing the complete data flow on a whiteboard before any configuration begins: CSV → CSV Sync → CCCStaging → OIM Sync → ESet objects. The deletion project is a third parallel path. Trainees who skip the diagram consistently configure the projects in the wrong order.
CNR_AD-1 — Read-Only Is Non-Negotiable Read_Only mode as a first step is the production standard for AD connectors. Never start with a read-write connector in any environment. Verify the schema and data mapping with read-only before enabling writes. Stress this to trainees as a professional practice, not just an exercise requirement.
CNR_AD-3 — Single Object Operations Are Production-Critical Single Object Operations enable "push on demand" provisioning. A change in OIM immediately pushes to AD without waiting for the scheduled sync. This is the standard production pattern for Group membership changes because security group membership often has immediate business impact — waiting hours for the nightly sync is not acceptable.

Troubleshooting Reference

ProblemLikely CauseSolution
CSV Sync runs but CCCStaging table stays empty Base Object filter too restrictive, or wrong CSV delimiter configured Check CSV file format and delimiter setting in the connector; verify the Base Object filter does not exclude all records; test with no filter first
ESet not created after Sync Mapping not configured correctly — key properties or provisioning workflow direction wrong Review mapping in Sync Editor; verify the provisioning workflow direction is correct (OIM→ESet); check the key property match configuration
Outstanding objects not being deleted Deletion Sync Project not configured or not run after the main sync Run the deletion sync project separately after the main sync; verify the markForDeletion workflow step is configured and both projects are activated
AD Sync fails with authentication error Service account credentials wrong, account locked, or insufficient AD permissions Check AD connector connection parameters; verify service account credentials and ensure the account is not locked in AD; confirm the account has required AD permissions
Single Object Operation not triggering after group change Process not activated, or PreScript condition evaluates to false Verify the custom process is activated in Designer; review the PreScript logic; temporarily simplify or remove the PreScript condition to isolate the issue
