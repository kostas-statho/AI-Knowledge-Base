Code: PRO  |  Intermediate  |  6 exercises  |  ~2 days

Overview

Module Purpose

  • OIM processes are event-driven automation chains
  • Progress from a single-step process through conditional triggers
  • Delete operations, wait steps, IT Shop automation
  • Complex multi-table cleanup processes

Prerequisites

  • BAS module complete
  • VB.NET basics (expressions, string methods)
  • OIM Designer access
  • Understanding of Job Queue / DBQueue

Key Concepts

  • Trigger: Table + Event (INSERT, UPDATE, DELETE)
  • Generating Condition: VB.NET expression — should process run?
  • Process Steps: handlers (SQL, Script, Mail, Connector, etc.)
  • Step Generating Condition: per-step conditional logic
  • HandleObjectComponent-Delete: deletes objects matching WhereClause
  • Wait Step: pauses execution until a condition becomes true
  • EventName: string — "INSERT" or "UPDATE" at runtime

Mentor Guidance

  • Distinguish trigger (when) from step handler (what) — two separate concepts
  • PRO-3 Generating Condition is the standard "changed to true" pattern
  • PRO-4 Frozen state: show Job Queue → Frozen → Unfreeze flow
  • PRO-5 EventName.Equals() avoids two separate processes with identical steps
  • PRO-6 $ColumnName$ injects current object's column value at runtime
Exercises
PRO-1

CustomProcess1 — Fires on Person INSERT

Requirements

Create a process named CustomProcess1 in Designer / Process Orchestration. The process fires when a new identity (Person) is created and writes a desired value to CustomProperty10.

Solution — Configuration Steps
  • Open Designer → Process Orchestration → create new process CustomProcess1
  • Set Table = Person
  • Set Events = INSERT
  • Add new process step named handle_update:
    • Set ObjectType to get the Person table
    • Configure the WhereClause to target the correct row
    • Rename parameter val_Property10 to val_CustomProperty10
    • Set the Parameter value to the desired output value
    • Set Server Function = SQL processing server
    • Enable Process History on Process Step properties
Test Cases
#Test ScenarioStepsExpectedStatus
1Create new identityCreate a new identity in OIM ManagerProcess appears in JobQueue; after processing, CustomProperty10 gets the desired valuePASS
Mentor Note Emphasise the two distinct layers: (1) the process trigger — Table=Person, Event=INSERT — decides WHEN the process fires; (2) the step handler — SQL action — decides WHAT it does. These are independently configured and both must be correct for the process to work as intended.

PRO-3

Conditional Delete on Temporary Deactivation

Requirements

Process that fires ONLY when an identity is temporarily deactivated. Uses a Generating Condition to check the IsTemporaryDeactivated flag. On trigger, deletes all Business Roles, System Roles, and Resources from the identity.

Solution — Configuration Steps
  • Table = Person, Event = UPDATE
  • Set Process Generating Condition:
{Value = $IsTemporaryDeactivated:Bool$}
  • Create one HandleObjectComponent-Delete step for EACH join table:
    • Step for PersonHasQERResource — removes Resource assignments
    • Step for PersonInOrg — removes Business Role memberships
    • Step for PersonHasEset — removes System Role assignments
  • Per step: only the "ObjectType" value changes — the WhereClause pattern is the same across all three steps
Generating Condition — Standard Pattern {Value = $IsTemporaryDeactivated:Bool$} is the standard pattern for "only run when this boolean field changed to true". This exact syntax appears throughout production OIM environments. Memorise it — it is used in templates, processes, and approval workflows alike.
Test Cases
#Test ScenarioStepsExpectedStatus
1Update unrelated fieldsChange Lastname; change Firstname on the identityProcess does NOT appear in JobQueue — Generating Condition evaluates to FalsePASS
2Temporarily disable identitySet IsTemporaryDeactivated = True on the identityProcess fires; Business Roles, System Roles, and Resources are deleted from the identityPASS
Mentor Note The Generating Condition is evaluated BEFORE the process is queued. If it returns False, the process never enters the JobQueue at all — it is not "skipped", it is never created. This is more efficient than creating the process and having a step condition abort it midway.
PRO-4

Multi-Delete with Wait Step — Frozen Process Demo

Requirements

Build Process_4 with the same three delete steps as PRO-3, PLUS one Wait step. Demonstrate what happens when the wait condition is never satisfied — the "Frozen" process state.

Solution
  • Copy or rebuild the PRO-3 structure (delete steps: PersonHasQERResource, PersonInOrg, PersonHasEset)
  • Add a Wait Step — the wait condition checks that a specific dependency no longer exists (e.g., PersonHasEset_NotExists evaluates to True)
  • The wait step is intentionally designed so that indirect assignments prevent its condition from ever becoming True in test scenario #3
Test Cases
#Test ScenarioStepsExpectedStatus
1Update unrelated fieldUpdate Lastname or Firstname on the identityProcess does NOT runPASS
2Temporarily disable — directly assigned roles onlySet IsTemporaryDeactivated = True; identity has only directly assigned Business RolesProcess runs; Business Roles, System Roles, and Resources are deletedPASS
3Temporarily disable WITH indirectly assigned Business RolesSet IsTemporaryDeactivated = True; identity has at least one Business Role via indirect (inherited) assignmentProcess runs but FREEZES — PersonHasEset_NotExists evaluates to False; process enters Frozen state in Job QueuePASS
Frozen Process — Key Teaching Moment Test #3 Frozen state is intentional. A Frozen process means: "I was waiting for a condition but it never became True — I stopped." In production, Frozen = needs human intervention. Navigation: Job Queue → filter "Frozen" → select process → click Unfreeze. Monitoring for Frozen processes is a real OIM admin task.
Mentor Note After demonstrating the Frozen state, show trainees the Unfreeze button and explain when it is appropriate to use it. Unreviewed Frozen processes can mask real automation failures in production environments.
PRO-5

Auto-Create IT Shop Service Item from Business Role

Requirements

A single process handles both Business Role INSERT (auto-creates the Service Item and adds it to the IT Shop) and UPDATE (renames the Service Item when the BR is renamed). EventName.Equals() separates INSERT and UPDATE logic within the same process.

Solution — Configuration Steps
  • Table = Org (Business Role table), Events = Insert AND Update
  • Process step Insert_AccProduct — Generating Condition:
{Value = EventName.Equals("INSERT", StringComparison.OrdinalIgnoreCase)}
  • Process step Insert_QERAssign — same INSERT Generating Condition as above
  • Process step Insert_ITShopOrgHasQERAssign — same INSERT Generating Condition
  • Process step UPDATE_Ident_Org — Generating Condition:
{Value = EventName.Equals("UPDATE", StringComparison.OrdinalIgnoreCase)}
EventName Pattern EventName.Equals("INSERT", StringComparison.OrdinalIgnoreCase) is the standard way to distinguish INSERT from UPDATE within a single process. Using OrdinalIgnoreCase makes the comparison robust. Do not use the = operator for string comparison in VB.NET process conditions.
Test Cases
#Test ScenarioStepsExpectedStatus

2Create Business Role WITH proper descriptionCreate BR with a valid description populatedProcess runs; Service Item automatically created; item available in IT Shop; request routed to Manager for approvalPASS
3Rename Business Role after Service Item createdUpdate the BR's display name after the Service Item has been createdService Item display name is updated to match the renamed BRPASS
Mentor Note The EventName.Equals() approach avoids maintaining two separate processes with nearly identical steps. Both INSERT steps and the UPDATE step live in the SAME process — the step-level Generating Conditions branch the logic. Show trainees the process in Designer and point out both types of Generating Conditions side by side.
PRO-6

Complex Cleanup — Two Processes (Person + Department)

Requirements

Create two cleanup processes: Process6 (Person table, triggered by IsInActive) and Process6b (Department table, triggered by IsCutNode). Each removes AD Groups and Identities from a System Role and disables the System Role.

Solution — Process6 (Person table)
  • Table = Person, Event = Update
  • Process Generating Conditions: {Value = $IsInActive:Bool$} and {Event = Update}
  • Process step Remove_ADGroups — WhereClause:
{value = String.Format(
  "UID_ESetHasEntitlement In(Select UID_ESetHasEntitlement from ESetHasEntitlement
   Join Eset on ... WHERE Eset.DisplayName = '{0}')",
  $LastName$)}
  • Process step Remove_Identities: removes all identities from the System Role
  • Process step Disable_SystemRole: disables the System Role object
Solution — Process6b (Department table)
  • Table = Department, Event = Update
  • Process Generating Condition: {Value = $IsCutNode:Bool$}
  • Same three process steps as Process6 (Remove_ADGroups, Remove_Identities, Disable_SystemRole)
  • Key difference: use $DepartmentName$ instead of $LastName$ in the WhereClause String.Format call
Runtime Parameter Injection The $ColumnName$ syntax (e.g., $LastName$, $DepartmentName$) injects the current object's column value at process execution time. This makes the WhereClause dynamic — it automatically targets the System Role named after the specific Person or Department that triggered the process.
Test Cases
#Test ScenarioStepsExpectedStatus
1Update unrelated field on Department or PersonUpdate any field that is not IsInActive or IsCutNodeNeither process runsPASS
2Update the trigger field on Department OR PersonSet IsInActive = True on Person, OR set IsCutNode = True on DepartmentThe corresponding process runs; all AD groups removed from System Role; all identities removed from System Role; System Role is disabledPASS
Mentor Note Process6 and Process6b are structurally identical — only the trigger table, the Generating Condition field, and the column name in the WhereClause differ. Show trainees how to clone Process6 in Designer and adapt it for Process6b — this is faster and reduces transcription errors.
