Module 019 — Data Importer

OIM's Data Importer (DIM) tool allows bulk loading of identity and object data from CSV files or SQL-connected databases. This module covers CSV file structure, key column selection, handling options, database connections, and reusable import definition files.

01 Key Concepts

CSV File Settings Encoding: UTF-8. Delimiter: semicolon (preferred). Use a text qualifier for values that contain commas. Key column selection can be composite (multiple columns form the unique key).
SQL Import Path Object Browser → write SELECT → copy to Data Importer → Import from Database → SQLClient Data Provider. Connection string and query are saved in the definition file.
Definition File Save the entire import definition as a .xml file immediately after the first successful import. Rebuilding the full column mapping is very time-consuming.
Handling Options Controls what the importer does when it finds a match or no match: Insert (create new), Update (modify existing), or Delete (remove matching records). Options can be combined.

Handling Options Reference

Option Behaviour Safe to Use?
Insert Creates new records for rows in the source that have no match in OIM Yes — safe default for initial loads
Update Updates existing OIM records when a key match is found Yes — safe for subsequent re-imports
Delete Deletes OIM records that match rows in the source file DANGEROUS — only use when you explicitly intend to delete records
Critical Warning — Delete Option
NEVER select the Delete option in Handling Options unless you explicitly and intentionally want to delete records from OIM. There is no undo. Always verify which option is selected before clicking Import.

Prerequisites

  • OIM Designer access
  • OIM Manager access
  • Access to OIM Data Importer tool
  • Basic CSV knowledge
  • Basic SQL SELECT knowledge (DIM-3 onwards)

02 Exercises

DIM-1 Import Identities from CSV
Goal: Create a CSV file containing identity data and use the Data Importer to load new Person records into OIM. Configure all required import wizard sections and save the definition for reuse.
Requirements
  • CSV file must be UTF-8 encoded with semicolon delimiter
  • All mandatory Person fields must be present in the CSV (FirstName, LastName, at minimum)
  • Correct key column selected in the Match Target step
  • Handling option set to Insert only (no Update, no Delete)
  • Import definition saved as .xml after successful import
Solution Steps
  1. 1Create a properly structured CSV file with all mandatory Person fields and at least 2-3 test rows
  2. 2Open Data Importer and load the CSV file
  3. 3File Structure: set encoding (UTF-8), delimiter (semicolon), and text qualifier as needed
  4. 4Line Structure: confirm column mapping matches CSV headers
  5. 5Line Condition: leave as-is (no filter needed for this exercise)
  6. 6Match Target: select target table (Person), map columns, and set the Primary Key column
  7. 7Hierarchy: leave as-is
  8. 8Options: select Insert only — do NOT select Update or Delete at this stage
  9. 9Connection Variables: leave as-is
  10. 10Save the import definition as a .xml file for use in DIM-2
Test Cases
# Test Scenario Test Steps Expected Results Mentor Notes
1 CSV missing mandatory fields Attempt to import a CSV that is missing one or more mandatory Person columns Error on saving / validation failure before import runs Helps trainee understand which fields are mandatory; check OIM Designer for mandatory column flags on Person
2 CSV with all mandatory fields Import a complete, correctly structured CSV Import runs successfully; new Person records visible in OIM Manager Verify each row in the CSV has a corresponding Person record; also confirm the definition XML was saved
Mentor Tips / Troubleshooting
  • Wrong key column causes duplicate Person records instead of updates — verify key selection before import; a unique business identifier (e.g., employee number) is a better key than FirstName
  • Always test with 2-3 rows first before importing a large dataset — much easier to clean up if something goes wrong
  • Save the definition XML immediately after the first successful import — rebuilding the mapping is time-consuming
  • If delimiter errors appear, open the CSV in a text editor (not Excel) to verify the actual delimiter character
DIM-2 Update Identities via CSV
Goal: Re-use the DIM-1 definition file to update existing Person records. The ONLY change from DIM-1 is switching the Handling Option from Insert to Update.
Requirements
  • Updated CSV with modified values on existing records (same key column values)
  • DIM-1 definition file loaded
  • Handling option changed from Insert to Update — this is the only required change
Solution Steps
  1. 1Update the CSV: modify field values on existing rows (keep key column values the same)
  2. 2Open Data Importer, load the DIM-1 definition file, change Handling Option to Update, then run the import
Key Teaching Point
The only change between DIM-1 and DIM-2 is the Handling Option (Insert → Update). Every other setting — file structure, column mapping, key column, connection variables — stays identical. This is why saving the definition file in DIM-1 matters.
Test Cases
# Test Scenario Test Steps Expected Results Mentor Notes
1 CSV missing mandatory fields Attempt to import with Update option but CSV is missing mandatory fields Error on saving / validation failure Same validation applies to Update as to Insert; mandatory fields are always required
2 CSV with all mandatory fields Import complete updated CSV with Update handling option Import runs successfully; updated values visible on Person records in OIM Manager Spot-check 2-3 records in OIM Manager to confirm values were changed — particularly fields that were intentionally modified in the CSV
Mentor Tips / Troubleshooting
  • If records are created as new (Insert behaviour) instead of updated, the key column values in the CSV do not match existing OIM records — check for whitespace or encoding differences in the key field
  • If records are not updated (silently skipped), verify the Handling Option was actually changed to Update before running
  • Remind trainees: never select Delete unless explicitly intentional — re-emphasise the warning from the Key Concepts section
DIM-3 DB Connection Configuration
Goal: Configure a database connection in Data Importer using the SQLClient Data Provider. Use the Object Browser to write and validate a SELECT query, then copy it into Data Importer for a database-sourced import.
Requirements
  • Object Browser accessible in OIM Designer for query development
  • Valid SELECT query targeting an OIM table or view
  • SQLClient Data Provider configured with correct connection string
  • Connection succeeds and query returns expected rows
Solution Steps
  1. 1Configure the database connection in Data Importer: select Import from Database, choose SQLClient Data Provider, enter connection string
  2. 2Write the SELECT query in Object Browser, verify results, then copy the query into Data Importer
  3. 3Complete remaining wizard steps (column mapping, key, options) and verify the connection succeeds
Verification
Connection is confirmed successful when Data Importer returns the expected rows from the database without connection errors. Save the definition file after confirming.
Mentor Tips / Troubleshooting
  • Connection string format: Server=HOSTNAME;Database=OIM_DB;Integrated Security=True; or with username/password depending on environment
  • If "Login failed" errors occur, verify SQL Server authentication mode and that the connecting account has db_datareader rights
  • Always test the SELECT query in Object Browser first — confirmed row results there means the query is valid before pasting into Data Importer
DIM-4 Advanced Connection Configuration
Goal: Extend the DIM-3 database connection with additional configuration options. Explores more advanced Data Importer connection features for handling complex import scenarios.
Requirements
  • Builds on the DIM-3 connection setup
  • Additional configuration steps as directed by mentor
  • Connection and import must succeed without errors
Solution Steps
  1. 1Start from the DIM-3 connection; apply advanced configuration options as directed
  2. 2Additional configuration steps (mentor-guided per environment)
  3. 3Verify connection and import succeed
Verification
Configuration is valid and no errors occur on the import run. Confirm data loaded matches the source query results.
Mentor Tips / Troubleshooting
  • Advanced options vary by environment — mentor should tailor this exercise to the specific connection requirements of the training lab
  • If the connection worked in DIM-3 but fails with advanced options, isolate which specific option was added last and revert to identify the cause
DIM-5 Connection Variable Configuration
Goal: Use Connection Variables in Data Importer to parameterise the import definition. Connection Variables allow the same definition file to be run against different targets without editing the XML directly.
Requirements
  • Import definition uses at least one Connection Variable
  • Variable is referenced in the import configuration (e.g., in a WHERE clause or connection string)
  • Import runs successfully using the variable value
Solution Steps
  1. 1Configure Connection Variables in the Data Importer Connection Variables wizard step
  2. 2Reference the variable in the appropriate location (query, connection string, or filter)
  3. 3Run the import and verify the variable value is applied correctly
Verification
Import runs using the variable value. Change the variable value and re-run; confirm the different value produces different results (e.g., different filtered row set).
Mentor Tips / Troubleshooting
  • Connection Variables make definition files reusable across environments (dev/test/prod) — emphasise this as a best practice
  • Variable name syntax must match exactly between the declaration and the reference; a mismatch is a common error and produces no obvious error message
  • After adding variables, always re-save the definition XML so the variables are persisted in the file

03 Mentor Tips & Key Concepts

Delete Option — Critical Warning (Repeat)
The Delete handling option WILL DELETE records from OIM that match the import source. This is permanent and cannot be undone via Data Importer. Always verify the Handling Options panel before clicking Import. Repeat this warning at the start of every DIM session.
Key Column Selection
Selecting the wrong key column is the most common mistake. A wrong key causes Data Importer to treat every row as a new record (all inserts instead of updates). Always pick a column that uniquely identifies each identity in the source system — employee number, email, or similar.
Test Small First
Always test with 2-3 rows before importing a full dataset. If something is misconfigured (wrong key, wrong handling option, missing fields), it is far easier to clean up 3 records than 3,000.
Save Definition XML Immediately
Save the .xml definition file after every successful import configuration. Rebuilding the full wizard from scratch — file structure, column mapping, key, options — takes significant time. The XML file is the single most valuable artefact from each exercise.
DIM-1 to DIM-2 is a Single Change
The only difference between a CSV insert import and a CSV update import is the Handling Option toggle. Everything else (column mapping, key, file structure, definition file) stays the same. This is intentional — it demonstrates how reusable the definition file is.
