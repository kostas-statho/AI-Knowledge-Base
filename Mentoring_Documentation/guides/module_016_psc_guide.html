Module 016 — PowerShell Connector (PSC)
Mentor Guide  |  Advanced  |  Server: StathopoulosK  |  DB: CCC_PSC

Background & Architecture

This module builds a complete custom PowerShell Target System Connector for One Identity Manager. Trainees start by writing raw PowerShell CRUD scripts against a custom SQL database, then progressively wire those scripts into an OIM XML Connector definition, configure synchronization mappings, and finally create Single Object Operations for real-time provisioning.

The custom target database CCC_PSC on server StathopoulosK contains three tables representing a simple identity store:

dbo.[User]
dbo.[Role]
dbo.[UserInRole]

Exercises flow: PSC-Prep (DB creation) → PSC-1 to PSC-4 (interactive PS scripts) → PSC-5 (strip interactivity for XML connector) → PSC-6 (UNS Root) → PSC-7 (mappings & sync workflow) → PSC-8 (Single Object Operations).

Prep

Database Setup — CCC_PSC

Pre-requisite
Requirements
  • Create a new SQL Server database named CCC_PSC on StathopoulosK.
  • Create the three required tables with correct columns, data types, and constraints.
  • Ensure SQL Server is configured for Mixed Mode authentication so both Windows and SQL logins work.
Solution Steps
  1. In SSMS, connect to StathopoulosK and create a new database named CCC_PSC.
  2. Create table dbo.[User]: columns UserID int PK, FirstName nvarchar, LastName nvarchar, EmailAddress nvarchar, JobDescription nvarchar.
  3. Create table dbo.[Role]: columns RoleID int PK, RoleName nvarchar.
  4. Create table dbo.[UserInRole]: columns UserID int FK → dbo.[User](UserID), RoleID int FK → dbo.[Role](RoleID). Composite PK on both columns.
Mentor Tips / Troubleshooting
Mentor Tips
  • CCC_PSC uses SQL authentication in later exercises — verify SQL Server allows mixed mode auth (Server Properties → Security → SQL Server and Windows Authentication mode).
  • The FK constraints in dbo.[UserInRole] are intentional — they will cause expected errors in PSC-5 tests 13 & 15 when trying to delete parents with children.
  • StathopoulosK is the server name used throughout every exercise — confirm trainees do not hardcode a different server.
PSC-1

Read Functions — Get-Users, Get-Roles, Get-UserInRole

PowerShell
Requirements
  • Create a single .ps1 file containing all three read functions plus shared helper functions.
  • Get-Users — reads from dbo.[User].
  • Get-Roles — reads from dbo.[Role].
  • Get-UserInRole — reads from dbo.[UserInRole] joined with user & role info.
  • Auth modes: -Integrated (Windows auth), -Credential (switch — triggers internal GUI popup with retry on wrong creds and graceful exit on cancel), -CredentialObject <PSCredential>.
  • Filtering via ASK switches only: -AskUserID (prompts for numeric UserID), -AskRoleID (prompts for numeric RoleID). No -UserID/-RoleID direct parameters (avoids binding conflicts).
  • Per-run log files written to C:\Users\OneIM\Desktop\PowerShell_Scripts\logs\, named <FunctionName>_yyyyMMdd_HHmmss.log.
  • Log levels: INFO (Write-Host + file), DEBUG (Write-Host + file), TRACE (Write-Verbose + file, visible only with -Verbose), ERROR (Write-Warning + file).
Key Design Notes
-Credential as SWITCH (not [PSCredential])

PowerShell normally requires a value for [pscredential] parameters. By declaring -Credential as a [switch] and calling Get-Credential internally, the UX becomes: type -Credential and a popup appears. This pattern includes retry logic on authentication failure and clean exit on dialog cancel. This is a deliberate advanced UX decision — trainees should understand why it is implemented this way.

Test Commands
# Auth --- SQL (also test wrong credentials + cancel dialog) # Get-Users Get-Users -SqlServer "StathopoulosK" -SqlDatabase "CCC_PSC" -Integrated | Format-Table -Auto $cred = Get-Credential Get-Users -SqlServer "StathopoulosK" -SqlDatabase "CCC_PSC" -Credential $cred | Format-Table -Auto Get-Users -SqlServer "StathopoulosK" -SqlDatabase "CCC_PSC" -Integrated -AskUserID | Format-Table -Auto # Get-Roles Get-Roles -SqlServer "StathopoulosK" -SqlDatabase "CCC_PSC" -Integrated | Format-Table -Auto Get-Roles -SqlServer "StathopoulosK" -SqlDatabase "CCC_PSC" -Credential $cred | Format-Table -Auto Get-Roles -SqlServer "StathopoulosK" -SqlDatabase "CCC_PSC" -Integrated -AskRoleId | Format-Table -Auto # Get-UserInRole Get-UserInRole -SqlServer "StathopoulosK" -SqlDatabase "CCC_PSC" -Integrated | Format-Table -Auto Get-UserInRole -SqlServer "StathopoulosK" -SqlDatabase "CCC_PSC" -Credential $cred | Format-Table -Auto Get-UserInRole -SqlServer "StathopoulosK" -SqlDatabase "CCC_PSC" -Integrated -askuserid | Format-Table -Auto Get-UserInRole -SqlServer "StathopoulosK" -SqlDatabase "CCC_PSC" -Integrated -askroleid | Format-Table -Auto Get-UserInRole -SqlServer "StathopoulosK" -SqlDatabase "CCC_PSC" -Integrated -askuserid -askroleid | Format-Table -Auto

No formal test table — the commands above cover all verification scenarios.

Mentor Tips / Troubleshooting
Mentor Tips
  • The -Credential switch pattern is intentional — do not let trainees "simplify" it to a [PSCredential] parameter, as this removes the retry-on-wrong-credentials behavior.
  • The logging helper functions defined in PSC-1 must be reused in PSC-2, PSC-3, and PSC-4 — establish this shared pattern early.
  • TRACE level messages are hidden unless the user adds -Verbose — this is the correct behavior; confirm trainees understand the difference between Verbose and Debug output.
  • Log files must include timestamp in filename — multiple test runs in the same session should produce separate log files.
Common Issues
  • If Get-UserInRole returns no rows, check that JOIN columns (UserID, RoleID) are correctly aliased and that data exists in all three tables.
  • If the -Credential popup does not appear, ensure the parameter is declared as [switch] not [pscredential].
  • If log directory does not exist, the function should create it automatically — remind trainee to add a New-Item -Force guard.
PSC-2

Write Functions — Create-User, Create-Role, Create-UserInRole

PowerShell
Requirements
  • Create a single .ps1 file with three create functions.
  • Create-User — inserts into dbo.[User]. If -SetUserID switch → prompt for unique numeric ID; otherwise → auto-pick smallest free integer.
  • Create-Role — inserts into dbo.[Role]. If -SetRoleID switch → prompt for unique numeric ID; otherwise → auto-pick smallest free integer.
  • Create-UserInRole — UserID and RoleID mandatory parameters. Function checks both IDs exist; if not → warns and re-prompts until valid, then inserts into dbo.[UserInRole].
  • Logs to C:\Users\OneIM\Desktop\PowerShell_Scripts\logs\<Function>_yyyyMMdd_HHmmss.log. Reuse logging helpers from PSC-1.
Test Cases
# Test Scenario Test Steps Expected Results Mentor Notes
1 Create-User with auto-generated UserID Run Create-User without -SetUserID User inserted; UserID is smallest available integer Verify the auto-ID logic handles gaps in the sequence correctly
2 Create-User with manually set UserID Use -SetUserID; enter wrong format, then already-used ID, then valid unique ID Warns on invalid/duplicate; user inserted on valid ID All three sub-cases must be tested in one run to verify retry loop
3 Create-Role with auto-generated RoleID Run Create-Role without -SetRoleID Role inserted; RoleID is smallest available integer Same auto-ID logic as Create-User
4 Create-Role with manually set RoleID Use -SetRoleID; enter wrong format, already-used ID, then valid unique ID Warns on invalid/duplicate; role inserted on valid ID Confirm duplicate check queries DB, not just in-memory
5 Create-UserInRole with existing membership Attempt to create a membership that already exists Message displayed: "MEMBERSHIP ALREADY EXISTS..." Should check before insert, not rely on PK violation
6 Create-UserInRole with non-existing / wrong format UserID Enter non-existing or non-numeric UserID; then enter a valid UserID Warns and re-prompts; membership inserted on valid input Verify existence check runs against DB for each attempt
7 Create-UserInRole with non-existing / wrong format RoleID Enter non-existing or non-numeric RoleID; then enter a valid RoleID Warns and re-prompts; membership inserted on valid input Both UserID and RoleID validation must be independent
Mentor Tips / Troubleshooting
Mentor Tips
  • The "smallest free integer" logic: query existing IDs into an array and find the first gap starting from 1. Avoid MAX(ID)+1 — it does not handle gaps.
  • Create-UserInRole must validate existence of both IDs before attempting insert — not rely on FK constraint errors as user feedback.
  • Ensure logging helpers from PSC-1 are dot-sourced or imported, not duplicated.
Common Issues
  • If auto-ID picks wrong value, the trainee is likely using MAX+1 — redirect to gap-finding approach.
  • If "MEMBERSHIP ALREADY EXISTS" is not shown and a PK error is thrown instead, the trainee is missing the pre-insert existence check.
PSC-3

Modify-User

PowerShell
Requirements
  • Create a .ps1 file with the Modify-User function.
  • If UserID does not exist — warn and re-prompt until a valid UserID is entered.
  • If EmailAddress is provided but invalid — warn and re-prompt until a valid email is entered, or user presses Enter to skip the email update entirely.
  • Only update fields that are provided, non-empty, and different from current DB values. Never overwrite with empty strings or null.
  • Reuse PSC-1 helper functions (logging, connection).
  • Logs to C:\Users\OneIM\Desktop\PowerShell_Scripts\logs\Modify-User_yyyyMMdd_HHmmss.log.
Test Cases
# Test Scenario Test Steps Expected Results Mentor Notes
1 Modify-User with all parameters Provide all updatable parameters with valid values All provided fields updated in Target System Verify each column is updated, not just the last one
2 Modify-User with wrong ID format or non-existing ID Enter non-numeric or non-existing UserID; then enter a valid UserID Warns and re-prompts; user updated on valid UserID Retry loop must re-query the DB each iteration
3 Modify-User with invalid email format Provide a badly formatted EmailAddress (e.g. badEmail) User is NOT updated; warning shown about email format Test both: re-prompt with valid email vs. pressing Enter to skip
Test Commands
$Server = "StathopoulosK"; $Db = "CCC_PSC" # All parameters — should update Modify-User -SqlServer $Server -SqlDatabase $Db -Integrated -UserID 5 ` -EmailAddress "valid@example.com" -JobDescription "Senior Analyst" # Non-existent UserID — should re-prompt Modify-User -SqlServer $Server -SqlDatabase $Db -Integrated -UserID 999999 ` -JobDescription "Test JD" # Invalid email — should warn and re-prompt or skip Modify-User -SqlServer $Server -SqlDatabase $Db -Integrated -UserID 5 ` -EmailAddress "badEmail"
Mentor Tips / Troubleshooting
Mentor Tips
  • The "only update changed fields" requirement means the function must first SELECT the current row, compare each incoming value, and build a dynamic SET clause.
  • Pressing Enter on the email re-prompt should result in the email column being left unchanged — not set to an empty string.
  • Email validation can use a simple regex: [^@]+@[^@]+\.[^@]+ is sufficient for this exercise.
Common Issues
  • If all fields are always updated regardless of change, the trainee is missing the current-value comparison step.
  • If pressing Enter on email prompt clears the email in DB, the trainee is not distinguishing between "empty input" and "no update".
PSC-4

Delete-User & Delete-UserInRole

PowerShell
Requirements
  • Create a .ps1 file with Delete-User and Delete-UserInRole.
  • Delete-User: removes a user by UserID; lists existing memberships; prompts Y/N confirmation. -Force switch skips the prompt entirely.
  • Delete-UserInRole: removes a single membership by (UserID, RoleID) pair. If UserID does not exist — warn and re-prompt.
  • Both functions: if UserID does not exist → warn and re-prompt.
Test Cases
# Test Scenario Test Steps Expected Results Mentor Notes
1 Delete existing membership Run Delete-UserInRole with valid UserID + RoleID that exist Membership deleted from Target System Verify with Get-UserInRole after deletion
2 Delete non-existing membership Run Delete-UserInRole with IDs that have no matching row Message: "Status: NotFound" Must not throw an exception — graceful not-found handling
3 Delete membership with non-existing UserID Provide a UserID that does not exist in dbo.[User] Message: "Status: NotFound" Distinction from test 2: user does not exist at all
4 Delete membership with non-existing RoleID Provide a RoleID that does not exist in dbo.[Role] Message: "Status: NotFound" Same graceful handling as tests 2 & 3
5 Delete User with no memberships Run Delete-User on a UserID with no rows in dbo.[UserInRole] User deleted directly (no membership confirmation needed) Function should still prompt Y/N unless -Force is used
6 Delete User with memberships Run Delete-User on a UserID that has memberships; confirm Y All memberships deleted first; then user deleted List of memberships must be shown before prompt
7 Delete User with memberships + -Force Run Delete-User -Force on a UserID with memberships User and all memberships deleted without any prompt Confirm -Force skips Y/N entirely, not just auto-answers Y
8 Reject membership deletion Enter wrong UserID; then answer N to reject deletion Message: "Status: AbortedByUser" Ensure nothing is deleted when user answers N
Mentor Tips / Troubleshooting
Mentor Tips
  • The -Force flag skips the Y/N prompt entirely — it does not just auto-answer Y. The confirmation logic should be inside an if (-not $Force) block.
  • Delete-User with memberships must delete child rows from dbo.[UserInRole] before deleting from dbo.[User] due to FK constraints.
  • Distinguish clearly between "Status: NotFound" (resource does not exist) and "Status: AbortedByUser" (user chose not to proceed).
Common Issues
  • FK violation on Delete-User: trainee is deleting from dbo.[User] before removing dbo.[UserInRole] rows — fix deletion order.
  • If -Force still shows the prompt, the trainee has the condition inverted — check the if guard logic.
PSC-5

OIM XML Connector Definition

Connector
Requirements
  • Update PSC-1 through PSC-4 scripts to use direct parameter-based IDs instead of interactive ASK/SET switches.
  • Replace: -AskUserID → -UserID <value>, -SetUserID → -UserID <value>, -AskRoleID → -RoleID <value>, -SetRoleID → -RoleID <value>.
  • Remove -Integrated and SQL prompt authentication modes from all scripts.
  • Remove all interactive data-entry checks and email format validation — data validation is OIM's responsibility, not the PowerShell script's.
  • Create the XML connector file, set required parameters, and verify all PSC-1 to PSC-4 functions through the OIM PSC browser.
Solution Steps
  1. Strip all interactive prompts from the PS scripts — functions must accept all values as parameters only.
  2. Create the XML connector definition file, referencing the updated PS scripts.
  3. Set all required connection parameters in the XML (server, database, credentials).
  4. In the OIM PSC Connector browser, test all functions from PSC-1, PSC-2, PSC-3, and PSC-4.

Mentor Tips / Troubleshooting
Mentor Tips
  • The core lesson of PSC-5: OIM XML connectors call scripts with explicit parameters — there is no console for interactive prompts. All interactivity must be removed.
  • Data validation is OIM's responsibility, not PowerShell's — stripping email checks and existence re-prompts is correct and expected behavior.
  • FK errors on Delete (tests 13 & 15) are the correct outcome — the connector reflects real DB behavior. In a full sync workflow, OIM would handle deletion ordering.
Common Issues
  • If functions still use Read-Host internally, the connector will hang — all prompts must be removed.
  • If -Integrated is still present, the connector may fail to authenticate — ensure SQL credential parameters are used instead.
  • PSC-9 duplicate errors (tests 7, 8, 9) should surface as DB errors, not custom "already exists" messages — confirm the pre-insert check was removed.

PSC-7

OIM Mappings & Sync Workflow

OIM Configuration
Requirements
  • Build three OIM connector mappings — one per DB table.
  • Users mapping: include virtual attributes vrt_InternalName and vrt_UID_UNSRootB.
  • Roles mapping: include virtual attributes vrt_UID_UNSRootB and vrt_RoleName (used as reference in UsersInRoles mapping).
  • UsersInRoles mapping: include virtual attributes vrt_RoleName and vrt_UserName.
  • Create a Sync workflow (Target → OIM, read direction) and a Provision workflow (OIM → Target, write direction).
Solution Steps
  1. Create the Users mapping with vrt_InternalName and vrt_UID_UNSRootB virtual attributes.
  2. Create the Roles mapping with vrt_UID_UNSRootB and vrt_RoleName virtual attributes.
  3. Create the UsersInRoles mapping with vrt_RoleName and vrt_UserName virtual attributes.
  4. Create the Provision and Sync workflows and associate them with the connector.
Test Cases

3 Run Provision — create Add a User Account, Role, and membership in OIM; run Provision workflow Data from OIM passed to Target System; objects visible in DB Confirms write direction (OIM → Target)

Sample SQL for Test 1
INSERT INTO dbo.[User] (UserID, FirstName, LastName, EmailAddress, JobDescription) VALUES (5, 'Alice NEW', 'Smith', 'alice.smith@example.com', 'Developer'), (6, 'Bob NEW', 'Jones', 'bob.jones@example.com', 'Tester'), (7, 'CharlieNEW', 'Brown', 'charlie.brown@example.com', 'Analyst'); -- Then run the Sync workflow in OIM
Mentor Tips / Troubleshooting
Mentor Tips
  • The vrt_ prefix means virtual attribute — these are computed/mapped values not stored directly in the target schema column.
  • Three separate mappings are required (one per table) — trainees sometimes try to combine them into one.
  • Sync workflow direction: Target → OIM (read data into OIM). Provision workflow direction: OIM → Target (push changes out).
  • vrt_RoleName defined in the Roles mapping is referenced again in the UsersInRoles mapping — the order of mapping creation matters.
Common Issues
  • If Sync runs but no data appears in OIM, check that vrt_UID_UNSRootB resolves to the correct UNSRootB UID created in PSC-6.
  • If Provision creates objects but they are not in the Target DB, verify the PS script paths referenced in the XML connector are correct and scripts are not still reading from interactive mode.
  • Missing vrt_RoleName in the Roles mapping will break the UsersInRoles mapping — always configure mappings in order.
PSC-8

Single Object Operations

OIM Configuration
Requirements
  • Create Single Object Operations in OIM for real-time sync without waiting for a scheduled full sync.
  • Create process CCC_UNS_AccountB_Adhoc: configure Prescript, Generation Condition, Events, and Process Step Configuration for user account changes.
  • Create process CCC_UNSAccountHasUNSGroup_Adhoc: configure Prescript, Generation Condition, Events, and Process Step Configuration for group membership changes.
Solution Steps
  1. In OIM Synchronization Editor, create the Single Object Operations configuration linked to the PSC connector.
  2. Create process CCC_UNS_AccountB_Adhoc: set Prescript (if required), Generation Condition to control when the process fires, Events (Insert/Update/Delete), and Process Step Config pointing to the connector's user account operation.
  3. Create process CCC_UNSAccountHasUNSGroup_Adhoc: same structure as above but targeting group membership operations.
Test Cases
# Test Scenario Test Steps Expected Results Mentor Notes
1 Add & Remove User Account Add a User Account via OIM; verify process fires; remove the account; verify process fires again Process CCC_UNS_AccountB_Adhoc runs; data from Target System passed to OIM in real time Check Job Queue in OIM to confirm the adhoc process was triggered — do not wait for scheduled sync
2 Add & Remove Account, Role, Membership via Manager Add and remove all object types via OIM Manager; verify each triggers the appropriate adhoc process Both adhoc processes run; data from OIM passed to Target System in real time Test Add and Remove separately. Verify Target DB reflects each change immediately after the process runs.
Mentor Tips / Troubleshooting
Mentor Tips
  • Single Object Operations allow real-time sync — the key advantage over scheduled full sync is immediacy. Emphasize this to trainees.
  • CCC_UNS_AccountB_Adhoc handles user account changes; CCC_UNSAccountHasUNSGroup_Adhoc handles group membership changes — these are separate processes with separate triggers.
  • The Generation Condition controls exactly when the process fires — an incorrect condition means the process either never fires or fires too broadly.
  • Trainees should check the Job Queue / process history in OIM after each test to confirm the adhoc process was triggered and completed successfully.
Common Issues
  • If the process does not fire, inspect the Generation Condition — it may reference a wrong object type or use an incorrect condition expression.
  • If the process fires but the Target DB is not updated, verify the PS script paths in the Process Step Config and confirm the scripts run without interactive prompts.
  • If both processes fire for every change (membership and account changes triggering both), the Generation Conditions are not scoped narrowly enough.
  • Common pitfall: DLL not copied to both required locations — if the OIM server cannot load the connector DLL, adhoc processes will fail silently.
