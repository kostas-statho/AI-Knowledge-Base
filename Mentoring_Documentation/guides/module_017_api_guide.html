Module 017 — APIs

Build custom REST API endpoints for the OIM Web Portal using C# CompositionApi Server Plugins. Covers GET/POST endpoints, authentication, CRUD operations, IT Shop integration, and OOTB AppServer APIs.

01 Dev Environment Setup

IDE Visual Studio 2022 with .NET 8.0 SDK installed (not .NET Framework — all OIM 9.3 plugins target net8.0)
DLL Naming Project output DLL must match *.CompositionApi.Server.Plugin.dll — the API Server plugin loader only picks up DLLs matching this exact suffix. A single deviation means the plugin is silently ignored at startup.
Debug Output Point the Debug build output path to the OIM install directory (same folder as ImxClient.exe). Release builds go to bin\Release\net8.0\.
Postman XSRF POST requests to the OIM API require the X-XSRF-TOKEN header. Add the pre-request script below to the Postman collection. GET endpoints do not require it.

Postman XSRF Pre-Request Script

Add this to the Pre-request Script tab at the Postman collection level so it applies automatically to all requests:

// OIM CompositionAPI — XSRF pre-request script
// Reads the XSRF-TOKEN cookie (set by OIM on login) and injects
// the X-XSRF-TOKEN header required on all non-GET requests.
const xsrfToken = pm.cookies.get('XSRF-TOKEN');
if (xsrfToken) {
    pm.request.headers.add({
        key: 'X-XSRF-TOKEN',
        value: xsrfToken
    });
}
XSRF Token Source
The XSRF-TOKEN cookie is set by OIM on successful login. If you call a POST endpoint without first logging in through the same Postman session, the cookie will be absent and the script will not inject the header — resulting in a 403 or "session expired" response even though the endpoint URL is correct. Always log in before testing authenticated endpoints.

Starting the Local API Server

imxclient.exe run-apiserver -B /plugin CCC.CompositionApi.Server.Plugin.dll /conndialog fallback
Administration Portal Activation
Navigate to http://localhost:8182/ on first run and activate globally. This is a one-time step per API Server instance. Without it, all custom endpoints return 404 regardless of DLL deployment status.

02 Exercises

API-1 Hello World GET Endpoint
Goal: Create the simplest possible custom API endpoint — a GET endpoint that returns a static string. Establishes the plugin compile → deploy → activate → test loop.
Requirements
  • C# class implementing IApiProvider and IApiProviderFor<PortalApiProject>
  • Endpoint registered via builder.AddMethod(Method.Define(...)) with GET handler
  • Returns static string "Hello World!"
  • Plugin DLL compiled and placed in API Server plugin directory
Solution Steps
  • Step 1: Write C# code for the endpoint class and compile the project
  • Step 2: Navigate to Administration Portal http://localhost:8182/ and activate the plugin globally
  • Step 3: Send a GET request via Postman to the registered endpoint path
Test Cases

Mentor Tips / Troubleshooting

  • Make sure the API Server was restarted after copying the new DLL

03 Mentor Tips & Key Concepts

DLL Naming Rule
The API Server only loads DLLs whose filename matches *.CompositionApi.Server.Plugin.dll. Any deviation (wrong suffix, wrong casing on Windows) silently prevents loading — no error is shown in the portal.
Authentication Toggle
.AllowUnauthenticated() is the single line that controls whether a session is required. Removing it enforces login. Its presence or absence determines the behaviour of tests 1 and 4 in API-3 through API-9.
Permission Testing Protocol
Always test with both a privileged account (Test,A with CCC_Api01) and an unprivileged account (Test,B without CCC_Api01). Permission enforcement is only proven by showing access is both granted AND denied correctly.
OOTB API Visibility Difference
Super_user accounts see records with XMarkedForDeletion=1 (soft-deleted entities). Standard identity accounts do not. When comparing results between account types, this is expected behaviour.
Custom AppID Scope
A custom AppID (e.g., apisamples) creates an isolated URL namespace. Endpoints under this AppID are not reachable via the default portal URL prefix. This is useful for separating admin-facing APIs from user-facing portal features.
