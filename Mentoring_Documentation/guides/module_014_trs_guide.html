Module 014 — Transport (TRS)

OIM transports carry customisation changes between environments. This module covers creating transport packages, exporting and importing ZIPs, reading import logs, and resolving conflicts — the foundation for safely moving changes from Dev through Test to Production.

01 Learning Objectives

  • 1Understand what is transportable (configuration) versus what stays per-environment (data)
  • 2Create a transport package in OIM Designer using both Manual and Automatic modes
  • 3Add specific objects to a Manual transport package with precision
  • 4Export the package to a ZIP file and verify its integrity
  • 5Import the ZIP package in a target environment and read the import log
  • 6Interpret all import log statuses: Inserted, Updated, Skipped, Error
  • 7Simulate and resolve a transport conflict by choosing Overwrite or Skip

02 Prerequisites

  • OIM Designer access
  • Basic OIM customisation knowledge
  • At least one custom template or process created
  • Access to two environments (Dev + Test) recommended
Mentor Note
If a full Dev/Test environment split is not available, TRS-3 can be performed by re-importing the same package into the same environment. The import log will show "Skipped" for all objects (since they are already current), but the mechanics are identical. Explain this to the trainee at the start so expectations are set correctly.

Analogy for Trainees
OIM Transport is like a git commit of configuration changes. The transport package is your commit, the ZIP file is your deployable artifact, and importing it is your deployment. Person data (UIDs, account records, employee records) are environment-specific and stay in each environment — they are never transported. Templates, processes, scripts, and sync definitions are transportable because they are configuration, not data.

Transportable vs Non-Transportable

Transportable (Config)

  • Process definitions and templates
  • Mail templates
  • Scripts and functions
  • Synchronisation project definitions
  • Report definitions
  • Job chain definitions
  • Web portal configuration
  • Role model structure (Business Roles)

Not Transportable (Data)

  • Person records
  • Account objects (AD, SAP, etc.)
  • OIM UIDs (environment-specific)
  • Role memberships (PersonInRole)
  • Audit logs and history
  • Job queue entries
  • Synchronisation run history
  • Environment-specific configuration values

Manual vs Automatic Packages

OIM Designer supports two package collection modes:

Mode How it collects objects Best used for Risk
Automatic Detects all changes made during the current Designer session Full feature releases; large-scale deployments May include unintended changes if session was not clean
Manual You explicitly right-click each object and add it Surgical deployments; individual fixes; this module's exercises May miss dependent objects if dependencies are not added manually

Import Log Status Codes

Status Meaning Action Required
Inserted Object did not exist in the target — it was created fresh None — expected for new objects
Updated Object already existed in target and was overwritten with the source version Verify the overwrite was intentional; check for local target changes that may have been lost
Skipped Object exists in target and is identical to source — no change was needed None — confirms the environments are in sync for this object
Error Import of this object failed — dependency missing, constraint violation, or corrupt data Must investigate and resolve — do not proceed to production with unresolved errors

04 Exercises

TRS-1 What Gets Transported?
Goal: Understand the transportable vs non-transportable distinction with concrete examples; classify a list of OIM objects as one or the other.

Key Steps

  • Discussion exercise: present the trainee with a mixed list of OIM objects to classify
  • For each object, ask the trainee to say "transportable" or "not transportable" and explain why
  • Objects to classify: Process template, Person record, Mail template, AD Account object, Script function, Job queue entry, Report definition, UID value, Web portal nav node, PersonInRole record
  • After the discussion, open OIM Designer → Transport and examine the object categories available for packaging
  • Confirm the trainee can explain the principle without referring to notes before proceeding
Mentor Hint
Use the git analogy: transport = committing configuration changes. A git commit contains code (config) not the database records your app creates at runtime. Similarly, OIM transport contains process definitions (config) but not the Person and account records the processes create at runtime. UIDs are environment-specific database identifiers — transporting them would cause UID collisions and break referential integrity in the target.
Common Mistake
Trainees often try to include Person records or PersonInRole memberships in a transport package, thinking they can migrate employee data this way. Explain clearly: data lives in each environment and is managed through sync processes or manual entry. The transport mechanism is exclusively for configuration objects.
TRS-2 Creating a Transport Package and Adding Objects
Goal: Create a Manual transport package in Designer; add a customised template (or process) to it with precision.

Key Steps

  • Open OIM Designer and ensure you have a customised object ready (e.g. a modified mail template or process from an earlier module)
  • Navigate to Transport in the Designer tree
  • Right-click the Transport node → New Package → select Manual mode
  • Name the package clearly, e.g. "TRS-Exercise-<date>"
  • Navigate to your customised template in its normal location in the Designer tree
  • Right-click the template → Add to transport package → select your named package
  • Return to the Transport node and expand your package to verify the object is listed
  • Discuss with the trainee: what dependencies does this object have? Should those be added too?
Mentor Hint — Manual vs Automatic
Automatic packages detect changes automatically during the Designer session — useful for releasing an entire feature. Manual packages let you choose exactly what goes in, which is better for surgical deployments: a specific bug fix, a single template change, or a carefully scoped delivery. Always use Manual for this exercise so the trainee experiences the deliberate selection process rather than letting the tool decide.
Common Mistake — Adding the Parent Folder
Trainees sometimes right-click the folder containing the object and add the folder to the package rather than the specific object. This exports every object in that folder, which is rarely what you want for a surgical deployment. Always navigate into the folder and right-click the individual object.
Dependency Awareness
When adding an object to a transport, consider its dependencies. A process template that calls a script function requires that script function to also exist in the target environment. If the dependency is missing in the target, the import will succeed but the object will fail at runtime. Check the object's Dependencies tab before finalising the package.
TRS-3 Export and Import
Goal: Export the package to a ZIP file; import it in the target environment (or re-import in the same environment); read and interpret the full import log.

Key Steps

  • In Designer, right-click your transport package → Export → choose a save location → confirm the ZIP is created
  • Verify the ZIP file exists and has a non-zero file size
  • Connect Designer to the target environment (or remain on the same environment for a test re-import)
  • Navigate to Transport → Import → browse to the exported ZIP file
  • Review the preview/summary screen before confirming the import
  • Confirm the import and wait for it to complete
  • Open the import log — scroll through every line and identify the status of each object
  • Ask the trainee to narrate what each log entry means using the status codes from the Core Concepts section
  • Verify the imported object is visible and functional in the target environment
Mentor Hint — Reading the Import Log
Walk trainees through the import log carefully and methodically. "Updated" = object already existed and was overwritten. "Inserted" = new object created. "Skipped" = no change needed (already up to date). "Error" = something went wrong — must not be ignored, even on non-critical objects. One unresolved error can cascade into runtime failures. Make the trainee read every line of the log in this exercise so the habit is established.
Common Mistake — Ignoring Import Errors
Trainees and even experienced practitioners sometimes close the import log at first glance if "most" objects show Inserted or Updated. A single Error entry near the bottom can go unnoticed. Errors often indicate a missing dependency in the target environment. Establish the rule: if there is any Error in the import log, the import is not considered successful until the error is investigated and resolved.
Re-Import on Same Environment
If only one environment is available, re-importing the same package produces an import log where everything shows "Skipped" (objects are already current). This is still a valid teaching exercise — it confirms the log is readable and the trainee understands what "Skipped" means. In production, seeing all-Skipped on a re-import is useful confirmation that the target already has the correct version.
TRS-4 Conflict Resolution
Goal: Simulate a transport conflict by modifying the same object in both Dev and Test; import from Dev to Test; observe the conflict indicator and choose Overwrite or Skip.

Key Steps

  • In Dev: modify a mail template (e.g. change the subject line) and package it in a new Manual transport
  • In Test: modify the same mail template (e.g. change the body text) — this creates a divergence
  • Export the Dev package to a new ZIP file
  • Import the Dev ZIP into the Test environment
  • Observe the conflict indicator in the import dialog — this flags that the target version differs from the source version
  • Discuss with the trainee: which version should win? What are the implications of each choice?
  • Choose Overwrite — the Dev version replaces the Test version completely
  • Verify the result in Test — confirm the Dev version is now active
  • Repeat the exercise but this time choose Skip — confirm the Test version is preserved unchanged
Mentor Hint — Conflict Decision Framework
In production, ALWAYS review conflicts manually — never click through conflict prompts without reading them. The rule of thumb: if the target has changes that are NOT in the source (e.g. a hotfix applied directly to production), "Skip" is safer. If the source is the definitive version of truth (Dev is the master), use "Overwrite". Establish the principle early: Dev is always the source of truth. Direct edits to Test or Prod environments are an anti-pattern because they will be overwritten by the next transport.
Production Conflicts Require Extra Care
When importing to Production, treat every conflict as a potential data loss event. Before overwriting a production object, take note of the current production version (or export it as a backup package). This allows rollback if the imported version has issues.
Expected Outcome: The trainee can safely move changes between environments and handle merge scenarios. They understand that Overwrite replaces the target with the source version, and Skip preserves the target version unchanged. They have the habit of reading the import log fully before considering an import complete.

05 Troubleshooting Reference

# Symptom Likely Cause Resolution
1 Objects missing from package after export Objects not saved or compiled in Designer before packaging Save all changes in Designer, recompile affected objects, then re-add to the transport package and re-export
2 Import fails with "Object not found" error A dependency of the imported object is missing in the target environment Check the Dependencies tab on the object in Dev; add missing dependencies to the transport package and re-import
3 Changes invisible after successful import Import succeeded but DBQueue not run, or a service restart is required Run DBQueue Processor; if still invisible, restart the OIM Job Service and API Server in the target environment
4 No conflict prompt during import despite known divergence Import ran in silent or non-interactive mode; conflict detection bypassed Check import settings in Designer; ensure interactive mode is selected so conflicts are surfaced for manual review
5 ZIP file appears corrupt or import refuses to open it File transfer corruption (e.g. via email or FTP in ASCII mode) Verify file size matches the source; re-copy using binary transfer; check ZIP integrity with a standard unzip tool
6 Template overwrote a newer version in target Chose "Overwrite" on a conflict without reviewing both versions Restore from the backup export (if taken before import); going forward, always review conflict content before choosing Overwrite
7 Import applied to wrong environment Designer was connected to the wrong server at import time Always verify the environment name shown in the Designer title bar before importing; establish a pre-import checklist for production deployments
