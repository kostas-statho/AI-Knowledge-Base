Module 015 — Reports (RPS)

OIM's reporting module generates business-ready reports on identity data. Reports query OIM views via SQL, render to PDF, Excel, or HTML, and can be scheduled for automatic delivery. This module covers building, parameterizing, scheduling, and publishing reports to the web portal.

01 Learning Objectives

  • 1Understand OIM's report architecture: data definition → SQL query on OIM view → template → rendered output
  • 2Create a basic list report using an OIM view (v_Person) and map result columns to the report layout
  • 3Add parameters to filter report output, including a ForeignKey dropdown parameter
  • 4Configure output format options: PDF, Excel, and HTML; understand the rendering engine behind each
  • 5Schedule recurring report generation via the Job Service and verify the schedule in the Job Queue
  • 6Publish a report to the web portal under a named navigation node for self-service access by standard users

02 Prerequisites

  • Basic SQL SELECT knowledge
  • OIM Manager access
  • OIM Designer access
  • Web portal access for testing
  • Crystal Reports runtime installed (for PDF)
Mentor Note
Trainees need only basic SQL (SELECT, FROM, WHERE, JOIN). They do not need to be SQL experts. If SQL is weak, spend 15 minutes on the pattern: SELECT columns FROM view WHERE filter before starting RPS-1. The view naming convention (v_ prefix) is a key concept to drill in from the first exercise.

Output Formats

PDF

Fixed-layout business reports; requires Crystal Reports runtime on the report server; best for compliance documents

Excel

Data-first export; allows downstream manipulation; no Crystal dependency; best for data analysts who need to filter further

HTML

Browser-renderable; works without Crystal; best for portal self-service where users just need to read data online

Always Use Views, Never Raw Tables

OIM views (e.g. v_Person, v_PersonInOrg, v_Department) automatically apply row-level security and exclude deleted, archived, and inactive records. Direct queries against raw tables (e.g. Person) bypass security trimming and return stale, deleted, or security-restricted data. This is not just a best practice — it is a security requirement. Every SQL query in every report in this module must use the v_ prefixed view, never the raw table.

Parameter Types

Type Input Method in Portal Use Case
String Free-text input box Search by name fragment, description keyword
Boolean Checkbox or toggle Filter by IsExternal, IsInActive flags
Date Date picker Filter by creation date range, effective date
ForeignKey Dropdown populated from a referenced table Filter by Department, Business Role, Location — the portal generates the dropdown automatically
Connection to Module PRO
Scheduled reports run via the OIM Job Service. The schedule definition creates a Job Queue entry that fires at the configured time. This is the same Job Service covered in Module 007 (PRO). When trainees ask "how does the schedule trigger?", the answer is the same mechanism that triggers all automated processes in OIM.

04 Exercises

RPS-1 First Report — Active Person List
Goal: Create a report listing all active Persons with first name, last name, department, and email address. Render a preview before configuring any output format.

Key Steps

  • Open OIM Designer → navigate to Reports in the object tree
  • Right-click Reports → New Report → choose "List Report" as the type
  • Name the report "Active Person List"
  • In the Data Definition tab, enter the SQL query (see below)
  • Map the returned columns to report display columns: FirstName, LastName, DefaultEmailAddress, DepartmentName, Description
  • Click Preview — verify data rows appear; confirm no deleted or inactive persons are returned
  • Ask the trainee: "What would happen if we used the raw Person table instead of v_Person?"
  • Save the report definition

SQL Query for This Report

SELECT
    p.FirstName,
    p.LastName,
    p.DefaultEmailAddress,
    d.DepartmentName,
    p.Description
FROM v_Person p
LEFT JOIN v_Department d
    ON p.UID_Department = d.UID_Department
WHERE p.IsInActive = 0
ORDER BY p.LastName, p.FirstName
Mentor Hint — Preview First
Always start with a preview before configuring the output format. The preview renders data immediately in the Designer grid without the overhead of PDF rendering or Crystal Reports processing. If the data looks wrong in preview (wrong columns, too many rows, wrong filter), fix it there rather than discovering the problem after a slow PDF render. Establish this as the standard workflow: preview first, format second.
Common Mistake — Raw Table Query
Using Person (raw table) instead of v_Person (view) returns deleted records (marked with XMarkedForDeletion), archived persons, and bypasses row-level security trimming. The report will appear to work but will show data that users should not see. Always use the v_ prefixed view — make this a non-negotiable rule the trainee can recite.
LEFT JOIN vs INNER JOIN
The query uses LEFT JOIN on v_Department so that Persons without a department assignment still appear in the report. An INNER JOIN would silently exclude undepartmented persons. For compliance reports, silently missing records is dangerous — prefer LEFT JOIN unless you have a specific reason to exclude non-matches.
RPS-2 Parameterized Report
Goal: Add a department filter parameter to the Active Person List report so users can scope output to a single department; demonstrate the ForeignKey parameter's auto-populated dropdown in the portal.

Key Steps

  • Open the Active Person List report from RPS-1 in Designer
  • Navigate to the Parameters tab of the report definition
  • Add a new parameter: Name = UID_Department, Type = ForeignKey, Referenced Table = Department, Referenced Column = UID_Department
  • Set the parameter as optional (not required) — users should be able to run the report without selecting a department
  • Update the SQL WHERE clause to include the department filter (see code block below)
  • Save and run Preview — with no parameter value, all active persons should still appear
  • Run Preview again with a specific department selected — verify only that department's persons appear
  • Deploy the report to the portal (if available) and show the trainee the auto-generated department dropdown

Updated WHERE Clause

-- Original WHERE clause from RPS-1:
WHERE p.IsInActive = 0

-- Updated WHERE clause with optional department parameter:
WHERE p.IsInActive = 0
  AND (@UID_Department IS NULL OR p.UID_Department = @UID_Department)

-- The IS NULL check makes the parameter optional.
-- When no department is selected, @UID_Department IS NULL evaluates TRUE
-- and the department filter is bypassed, returning all active persons.
Mentor Hint — ForeignKey "Magic"
ForeignKey parameters automatically generate a dropdown list in the portal by querying the referenced table. OIM populates the dropdown from v_Department (or whichever table is referenced) without any additional configuration. Show the trainee this behaviour in the portal — it often surprises them that a dropdown appears with real department names without writing any dropdown-populating code. This is one of OIM's most useful report features.
Common Mistake — Unregistered Parameter
Adding @UID_Department to the SQL WHERE clause but forgetting to register it as a parameter in the report's Parameters tab causes a SQL execution error when the report runs. OIM does not know to supply a value for an unregistered placeholder. The parameter definition in the Parameters tab is mandatory — the SQL alone is insufficient.
Optional vs Required Parameters
If the parameter is set as Required, users cannot run the report without selecting a department — useful for reports where running without a filter would be prohibitively slow. If Optional, users can omit the parameter to get all results. For this exercise, Optional is correct because the report is intentionally designed to work both filtered and unfiltered.
RPS-3 Schedule and Portal Publication
Goal: Schedule the Active Person List report to run daily at 06:00; publish it to the "Compliance Reports" portal navigation node; verify it is accessible and usable by a standard portal user.

Key Steps

  • Open the Active Person List report in Designer → navigate to the Schedule tab
  • Enable scheduling → set Recurrence = Daily → set Time = 06:00 → Save
  • After saving, open the OIM Job Queue (Manager or Designer) and locate the scheduled report entry — confirm it shows the next run time as tomorrow at 06:00
  • Return to the report definition → navigate to the Publish or Portal tab
  • Set the Navigation Node = "Compliance Reports" (create the node first if it does not exist)
  • Save and run DBQueue
  • Log into the web portal as a standard user → navigate to the Compliance Reports section
  • Confirm the "Active Person List" report appears in the list
  • Run the report from the portal → verify the department dropdown appears → run both filtered and unfiltered
  • Test output formats: download as PDF, Excel, and HTML if all three are enabled
Mentor Hint — Scheduled Reports and the Job Service
Scheduled reports run via the OIM Job Service. The schedule definition you created in the Report's Schedule tab causes OIM to create a recurring Job Queue entry. Show this entry to the trainee in the Job Queue — it makes the scheduling mechanism concrete rather than abstract. This connects directly back to Module 007 (PRO): scheduled reports are just another process triggered by the Job Service, using the same mechanism as any other scheduled OIM job.
Testing Without Waiting for Schedule
To test the report immediately without waiting until 06:00, run it manually from Designer or from the portal. The schedule only controls automated unattended execution. Manual execution via the portal run button bypasses the schedule and triggers rendering immediately, which is the correct approach for testing during development.
Portal Navigation Node Access
The "Compliance Reports" navigation node must be accessible to the standard user's Application Role. If the user cannot see the Compliance Reports section in the portal, check that the navigation node has the appropriate permission grant linked to a Business Role the user is a member of. This connects back to Module 012 (PMS) — portal visibility is always controlled by the role model.
Expected Outcome: The report is accessible in the portal under Compliance Reports, scheduled to run daily at 06:00 via the Job Service, renders with a working department filter dropdown, and exports correctly to all three output formats. The trainee can trace from portal click through Job Service through SQL view to rendered output without assistance.

05 Troubleshooting Reference

# Symptom Likely Cause Resolution
1 Report preview shows no data SQL query returns no rows — filter too restrictive or view returns nothing in this environment Copy the exact SQL and run it in SQL Server Management Studio against the OIM database; check the WHERE clause, especially IsInActive value
2 Department dropdown is empty in the portal ForeignKey parameter resolution config incorrect; wrong table or column referenced Open parameter definition in Designer; verify Referenced Table = Department and Referenced Column = UID_Department; confirm v_Department has data
3 PDF render fails or produces blank output Crystal Reports runtime not installed or misconfigured on the report server Verify Crystal Reports runtime is installed on the OIM report server; check OIM logs for Crystal-specific errors; test HTML output instead to isolate
4 Schedule not running at the configured time OIM Job Service is stopped, or the schedule definition was not saved with a valid recurrence Check Job Service status in Windows Services; re-open the schedule definition and confirm recurrence is set; check Job Queue for the entry
5 Report not visible in the portal navigation Report not linked to a portal navigation node, or the user lacks access to that node Open report definition → Publish tab → confirm navigation node is set; check that the node is accessible to the user's Application Role
6 Deleted or archived persons appearing in report results SQL query uses raw Person table instead of v_Person view Update all SQL references from Person to v_Person; similarly replace any raw table references with their v_ view equivalents
7 Department filter makes the report very slow Missing database index on UID_Department in the underlying table, or the view itself is unoptimised Test the raw SQL in SSMS with an execution plan; add a SQL query hint if appropriate; consider pre-filtering in the view or adding an index via a DBA
