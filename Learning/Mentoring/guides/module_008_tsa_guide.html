Module 008 — Target Systems

Target systems are the external directories and applications that OIM provisions and manages. This module covers registering target systems, defining account schemas, configuring provisioning rules, and managing the full account lifecycle including outstanding objects.

Overview

A target system in OIM is the metadata definition that describes an external directory or application. OIM stores its own representation of the accounts, groups, and containers in that external system, then uses provisioning rules to keep the two in sync. This module focuses on the OIM side of that relationship — how the target system record is structured, how accounts are represented, how names are generated, and how OIM handles discrepancies (outstanding objects) when the external system changes without OIM knowing.

The built-in Active Directory connector serves as the gold-standard reference throughout this module. Before building anything new, trainees should spend time deeply exploring an existing, fully-configured AD target system to understand what a complete implementation looks like at every level.

What OIM Stores

  • Target system definition (metadata)
  • Container hierarchy (OUs, folders, etc.)
  • Account objects (one per person per system)
  • Schema definition (columns and types)
  • Provisioning rules (naming, defaults)

Key Designer / Manager Locations

  • Manager → Target Systems (browse tree)
  • Designer → Target Systems (define structure)
  • Designer → Schema (add custom columns)
  • Designer → Provisioning Rules
  • Manager → Persons → Accounts (view accounts)

Learning Objectives

  • Understand the target system hierarchy (System → Container → Account)
  • Register a new target system definition
  • Define account schema (columns, data types, mandatory fields)
  • Configure provisioning rules (account naming, default values)
  • Assign target systems to departments and IT shop
  • Handle outstanding objects (XMarkedForDeletion)
  • Configure account deletion behaviour
  • Run end-to-end provisioning test

Prerequisites

  • Module BAS (005) completed
  • Basic understanding of directory structures (helpful: Active Directory or LDAP)
  • Access to OIM Manager → Target Systems
  • Designer access with Target System permissions

Key Concepts

Target System Object Hierarchy

1
Target System
Top-level definition object — identifies the external system type, connector, and display name
2
Container
Represents OUs, folders, or organisational groupings within the external system
3
Account
An individual user account in the target system — linked to an OIM Person record via UID_Person
A "Target System" in OIM is the metadata record describing the external system — NOT the external system itself. You could delete the OIM target system record with zero impact on the actual Active Directory. Make this distinction explicit early; it confuses almost every new trainee.

Schema Change Deploy Cycle

Any schema change — adding columns, changing data types, updating display labels — requires a full deploy cycle. Trainees who skip any step will encounter confusing "column not found" errors.

  • Make schema changes in Designer (add column, set type, set display label)
  • Save the changes in Designer
  • Compile the schema: Designer → Database → Compile
  • Deploy the compiled schema: Designer → Database → Deploy
  • Restart the OIM API Web Service (or full IIS reset in test environments)
  • Verify the new column appears in OIM Manager under the account object

Outstanding Objects — XMarkedForDeletion

When a sync runs and an account previously known to OIM can no longer be found in the target system, OIM sets XMarkedForDeletion = 1. This is an outstanding object. OIM does not automatically delete these — it waits for a human or a configured rule to decide.

Resolution Options

  • Republish — re-create the account in the target system
  • Delete from OIM — remove the OIM record entirely
  • Ignore — leave the flag set (not recommended long-term)

Why They Matter

  • Accumulate over time and pollute reports
  • May indicate provisioning failures needing attention
  • Can cause duplicate errors if republished incorrectly

Exercises

TSA-1

Exploring Existing Target Systems

Beginner

Goal: Navigate the target system object tree in OIM Manager. Understand the System → Container → Account hierarchy by exploring a fully configured existing system.

Key Steps
  1. Open OIM Manager and navigate to Target Systems
  2. Expand the tree to find an Active Directory target system
  3. Click the top-level target system node — examine properties (display name, connector type, UID)
  4. Expand into a Container node — note the DN, parent container, and associated accounts
  5. Click an Account node — examine mapped columns (account name, email, status, XMarkedForDeletion)
  6. Navigate to Manager → Persons; find the Person linked to that account; observe the UID_Person link
Use the built-in AD connector as the reference. Encourage trainees to spend at least 20 minutes clicking through the tree before asking questions — the structure becomes intuitive through exploration. Point out that every account has a UID_Person foreign key linking it back to the central Person record. This foreign key is the foundation of the entire provisioning model.
Trainees frequently confuse the "Target System" object in OIM Manager (the metadata record) with the actual external Active Directory server. Make this explicit: deleting the OIM target system record does not affect the real AD. The OIM record is a mirror — a representation, not the master.
Trainee can navigate from a Person record to their associated accounts across multiple target systems, and explain what each level of the hierarchy represents.
TSA-2

Registering a New Target System

Intermediate

Goal: Create a custom target system definition in Designer. This is a conceptual exercise — no real external system is connected. The goal is to understand the OIM metadata structure.

Key Steps
  1. Open OIM Designer and navigate to Target Systems
  2. Click New; provide a display name: CCC Training System
  3. Set connector type to Generic (no real connector)
  4. Configure the container structure: add one root container named Users
  5. Save and compile; navigate to OIM Manager → Target Systems; verify the new system appears
  6. Attempt to view accounts — confirm none exist yet (sync has not run)
Emphasise clearly: the "connector" is the Sync Engine process that periodically exchanges data. The target system record in OIM is just metadata describing the structure. Creating the record does not connect to anything. Many trainees expect accounts to appear immediately — correct this misunderstanding now, before the sync module (CNR-010).
Trainees often expect that saving the target system in Designer immediately makes it operational. Accounts will not appear until a Sync Project (Module 010 CNR) is configured and executed against a real data source. The target system record is the container; the sync project is what fills it.
TSA-3

Account Schema Definition

Intermediate

Goal: Add custom columns to the account schema. Set data types, mandatory flags, and display labels. Complete the full deploy cycle and verify the columns appear in OIM Manager.

Key Steps
  1. In Designer, navigate to Schema and find the account class for the CCC Training System
  2. Add CCC_LoginName: type = String(50), mandatory = Yes, display label = "Login Name"
  3. Add CCC_AccountExpiry: type = DateTime, mandatory = No, display label = "Account Expiry"
  4. Add CCC_IsAdmin: type = Boolean, mandatory = No, display label = "Admin Account"
  5. Save in Designer; compile (Database → Compile); deploy (Database → Deploy)
  6. Restart the web service; verify the three columns appear in OIM Manager on the account schema
Walk trainees through each step of the deploy cycle explicitly. This is the single most common source of "why don't I see my new column?" questions. The cycle is: Save → Compile → Deploy → Restart service. Skipping any step leaves the system in an intermediate state. In production, a service restart may require a change request — explain that this is why schema changes are batched into deployment windows.
Adding columns without completing the full deploy cycle is the most common mistake in this exercise. The column exists in Designer's data model but the OIM runtime has not picked it up yet. A trainee who skips the service restart will insist "it's not working" when the issue is simply a stale service cache.
Trainee can add a schema column, complete the deploy cycle independently, and verify the result in OIM Manager without assistance.
TSA-4

Provisioning Rules

Intermediate

Goal: Configure a naming rule that automatically generates an account name when OIM provisions a new account. The rule uses the Person's first and last name.

Key Steps
  1. In Designer, navigate to Provisioning Rules for the CCC Training System
  2. Create a new rule targeting the CCC_LoginName column on the account object
  3. Set rule type to Template; enter the VB.NET expression below
  4. Save and compile; create a test Person with first name "Jane" and last name "Smith"
  5. Trigger provisioning (assign the target system to a role; assign the role to the Person)
  6. Verify the generated account has CCC_LoginName = "jane.smith"
Provisioning Rule Expression
' AccountName template — generates firstname.lastname in lowercase Value("Person", "FirstName").ToLower() & "." & Value("Person", "LastName").ToLower()
Connect this explicitly to Module 006 (TPL). Provisioning rules ARE templates — the exact same VB.NET template engine runs them. Trainees who have mastered TPL will immediately recognise the syntax. The Value("Person", "ColumnName") form crosses the foreign key from the account object to the linked Person record — identical to cross-table TPL templates.
The account name column name differs between target systems — in AD it is sAMAccountName; in a generic system it may be CCC_LoginName. Always check the actual column name in the schema editor before writing the rule. Targeting the wrong column results in the rule silently doing nothing to the intended field.
TSA-5

Outstanding Objects

Intermediate

Goal: Simulate the outstanding objects scenario. Create an account, simulate its external deletion, observe the XMarkedForDeletion flag, and work through the resolution options.

Key Steps
  1. Manually create a test account record in OIM Manager under the CCC Training System
  2. Verify the account has XMarkedForDeletion = 0
  3. Simulate an external deletion: manually set XMarkedForDeletion = 1 on the record (demo mode)
  4. Open the account in OIM Manager — observe the outstanding object indicator
  5. Walk through all three resolution options: Republish, Delete from OIM, Ignore
  6. Choose Delete from OIM; confirm the account record is removed
Outstanding objects are OIM's way of saying: "I had a record for this, but I can't find it in the target system any more." Republish recreates the account in the target — only appropriate if the deletion was accidental. Delete from OIM removes the local mirror — correct if the account was legitimately deleted. Ignore is almost never the right long-term choice; use it only as a temporary hold while investigating.
Ignoring outstanding objects is the most common mistake in production environments. They accumulate silently over months, then appear in compliance reports and cause confusion. Configure an automated cleanup rule for old outstanding objects rather than letting them pile up indefinitely.
Trainee can explain what XMarkedForDeletion means, identify outstanding objects in OIM Manager, and select the appropriate resolution strategy for a given scenario.
TSA-6

End-to-End Provisioning Test

Advanced

Goal: Complete the full provisioning lifecycle: assign a role → DBQueue processes → account created. Then remove the role → account disabled or deleted. Trace every step in the chain.

Key Steps
  1. Create a test Person in OIM with a known first and last name
  2. Assign a role configured to provision to the CCC Training System
  3. Wait for DBQueue to process (or trigger manually via Job Queue)
  4. Navigate to the test Person → Accounts — verify a new account with the correct generated name
  5. Check CCC_LoginName matches the template rule from TSA-4
  6. Remove the role from the Person; wait for DBQueue; verify account disabled or deleted per the deprovisioning rule
  7. Document the full lifecycle in the trainee's notes
This exercise ties together everything from TSA-1 through TSA-5. Run it as a live walkthrough the first time — then ask the trainee to repeat it independently with a different test Person. The ability to trace the provisioning chain (role assignment → process → DBQueue → account creation) is a fundamental skill used daily in OIM consulting work.
Trainee can independently trace the full provisioning lifecycle from role assignment through account creation to deprovisioning, and explain each step in the chain without assistance.

Troubleshooting Reference

SymptomDiagnosis & Resolution
Account not created after role assignmentCheck Job Queue Info for errors on the provisioning job. Verify that a provisioning rule exists for this target system and that DBQueue is actively processing. Check Job Service status in Windows Services.
Wrong account name generatedInspect the AccountName provisioning rule expression in Designer. Trace the VB.NET logic manually with the actual Person column values. Verify the column name in the expression matches the schema column name exactly (case-sensitive).
Target system not visible in OIM ManagerDesigner save was not followed by a compile and deploy. Check the deploy log in Designer → Database → Deploy Log for errors. A failed deploy leaves the system in the pre-change state.
Schema columns missing in OIM ManagerThe OIM web service or API service needs a restart after a schema deploy. Restart the relevant Windows service or IIS application pool. In test environments, an IIS Reset is sufficient.
Outstanding objects not clearing after syncThe sync project is not configured to handle deletions. Check the sync project's Provisioning Rules and Synchronization Rules for deletion handling configuration. See Module 010 CNR for detailed sync troubleshooting.
Provisioning to wrong containerThe container assignment on the target system definition is missing or incorrect. Check the provisioning rule for container selection logic. In AD, verify the target OU is correctly specified in the provisioning rule's container template.
Account stays enabled after role removalThe deprovisioning rule is not configured. Navigate to Designer → Provisioning Rules for the target system and verify a rule exists for the deletion/disable action when the assignment is removed.
Test Person not visible in provisioningPerson type filter or active flag excludes the test Person. Check Person.IsInactive and Person.XIsInEffect. Some provisioning rules filter by PersonType or employment status — verify the test Person meets all filter criteria.
