Module 024_004 — Test Assist

Advanced track: PowerShell test automation using Pester 4.x and Gherkin BDD with OIM integration

Prerequisites

  • Pester version 4.x ONLY — NOT version 5.x. Breaking API change: Invoke-Gherkin was removed in Pester v5 and does not exist there. If v5 is installed, it must be downgraded or v4 installed side-by-side.
  • Deployment Manager configured and working — Test Assist depends on the DM module for DB connectivity.
  • PowerShell 7 — required for TestAssist compatibility.
  • VS Code extensions: alexkrechik.cucumberautocomplete (Gherkin syntax highlighting and step autocomplete) and ms-vscode.powershell (PowerShell language support and test runner).
TA_01

Install Test Assist and Prerequisites

Requirements
Unzip and install TestAssist 1.2.2 with all required dependencies: Pester 4.x, Selenium (version 3.0.1), and NameIT. The installation must result in a working DM DB connection verified through Console Mode.
Solution Steps
  1. Unzip TestAssist-1.2.2 to C:\Users\OneIM\Desktop\TestAssist.
  2. Install Pester version 4.x — specifically the latest 4.x release (4.10.1). Do NOT install version 5.
  3. Install Selenium — open TestAssist\utility\Install-Selenium.ps1 and modify the path: change all occurrences of Selenium/2.0.0/ to Selenium/3.0.1/ before running the script.
  4. Install the NameIT module.
Hint
Important: Avoid spaces in the folder path for the TestAssist installation — some internal scripts fail silently with spaces in paths. Import-Module is required each session before using TestAssist commands. Test DB connectivity using Console Mode before proceeding to TA_02.
Verification
Pass criteria: Pester version confirmed as 4.x (run Get-Module Pester -ListAvailable); Selenium and NameIT modules are importable; DM connection works in Console Mode.
Mentor Tips / Troubleshooting
Mentor Notes
  • Pester 4 vs 5 is a BREAKING change — Invoke-Gherkin exists only in v4. If the trainee has v5 installed, they must either downgrade (Uninstall-Module Pester -AllVersions then install v4) or install v4 side-by-side using Install-Module Pester -RequiredVersion 4.10.1 -Force -SkipPublisherCheck.
  • The Selenium path change in Install-Selenium.ps1 is easy to miss — confirm the file was edited before running it.
  • DM connection must work before any further exercises — all test APIs run as a continuous DM session.
TA_02

Configure Setup.ps1 and Encrypt Secrets

Requirements
Configure the TestAssist module connection to the OIM database. The connection string must be encrypted using "password" as the key for this training exercise. Setup.ps1 must load without errors and DB connectivity must be confirmed.
Solution Steps
  1. Encrypt the connection string used in DM, using "password" as the encryption key.
  2. Replace the plaintext connection string with the encrypted value in C:\Users\OneIM\Desktop\Deployment\Config\Example\projectConfig.
  3. Create a Decryption Key entry for the key "password" so TestAssist can decrypt the connection string at runtime.
  4. Configure TestAssist\Setup.ps1 with the correct paths and settings for this environment.
  5. Reload the module with -Force flag and verify DB connectivity.
Hint
Note: Setup.ps1 is loaded when importing the TA module. Know how to force-reload it after making changes: Import-Module TestAssist -Force. Once connected, all TestAssist API calls run as a continuous session similar to DM Console Mode. Training note: "password" as the encryption key is for training only — production environments must use secure key management.
Verification
Pass criteria: Module imports without errors; DB connection is confirmed (no connection refused or authentication errors).
Mentor Tips / Troubleshooting
Mentor Notes
  • "password" encryption key is for training only — reinforce that production environments must use proper secret management.
  • Always use Import-Module TestAssist -Force after any change to Setup.ps1 — the module caches its state and changes do not take effect without the force reload.
  • If connection fails after configuration, check the encrypted value was pasted into the correct field in projectConfig without any extra whitespace.

TA_04

Create Test Stub Files

Requirements
Use the TestAssist utility cmdlets to auto-generate stub test files for custom OIM objects: Processes, Scripts, and Templates. The stubs are scaffolding files tied to the specific OIM instance's custom objects — they will not pass tests until completed in later exercises.
Solution Steps
  1. Run C:\Intragen\TestAssist\utility\Write-ProcessTestStubs.ps1 and create process stub tests.
  2. Run Write-ScriptTestStubs.ps1 and create script stub tests.
  3. Run Write-TemplateTestStubs.ps1 and create template stub tests.
Hint
Note: Stubs are scaffolding only — they will not pass until the test logic is written in later exercises (TA_05, TA_06). The utility cmdlets introspect the OIM Designer DB to find custom objects, so a working DM connection from TA_02 is a hard prerequisite.
Verification
Pass criteria: Stub files are created and their names match the custom objects present in the OIM instance (Processes, Scripts, Templates).
Mentor Tips / Troubleshooting
Mentor Notes
  • Utility cmdlets read directly from the OIM Designer DB — the DM connection from TA_02 must be working first. If stubs generate with no content, the DB query returned zero results (check filters/scope).
  • Stub files won't pass until the test logic is written — this is expected behavior, not a bug.
  • Confirm stub file names match the actual custom objects in the trainee's OIM instance — they will differ between environments.

TA_06

Write a Script Unit Test

Requirements
Convert a generated script stub into a working Pester unit test. The completed test file TA_06_ScriptStubsFinal.tests.ps1 must run with Invoke-Pester and show pass/fail details. The test must properly account for SQL transaction behavior.
Solution Steps
  1. Run Write-ScriptTestStubs and create the stub test file.
  2. Modify the stub to create the completed test file TA_06_ScriptStubsFinal.tests.ps1.
  3. Run the test with Invoke-Pester.
Hint
Think carefully about SQL transactions and rollback. OIM scripts that modify data run inside a transaction context — if a test causes a DB write, consider whether the test should roll back that change to keep the database clean for subsequent runs. TestAssist may provide transaction wrapping utilities — check the /lib directory for patterns.
Verification
Pass criteria: Invoke-Pester runs; individual test pass/fail status and details are shown in the output.
Mentor Tips / Troubleshooting
Mentor Notes
  • SQL transaction behavior is the key nuance here — script tests that write to the DB may leave dirty data if not wrapped in a rollback. Guide trainees to understand the difference between test isolation (rollback) and integration testing (commit).
  • If a test passes on the first run but fails on subsequent runs, it is likely leaving persistent data that violates a uniqueness constraint on re-run.
TA_07

Gherkin Execution Flow

Requirements
Understand and be able to explain the complete Gherkin execution pipeline in TestAssist — from a .feature file through step matching to PowerShell execution and result collection. No formal test table; the goal is conceptual mastery of the flow.
Solution Steps — Execution Flow

Gherkin Execution Pipeline

  1. Pester reads the .feature file
  2. TestAssist parses the Gherkin syntax (Given/When/Then steps)
  3. For each step, TestAssist finds a matching regular expression in the corresponding .steps.ps1 files
  4. Parameters are extracted from the step text using regex capture groups
  5. The PowerShell code block in the matching step definition is executed with the extracted parameter values
  6. Test results (pass/fail) are collected for each step
  7. Teardown logic runs if defined (e.g., cleanup steps)
Concrete Example
In the gherkin directory: example .feature files.
In the lib directory (Glue Code): matching .steps.ps1 files.

Example trace: webtest.feature line 5 captures Namespace="webtest", Key="webuser". Line 6 identifies a step, which matches with lib/common-person_attr.steps.ps1 line 65. The matched code block executes: $person = Resolve-StepTestData.
Supported Languages
Gherkin supports multiple languages — .feature files and .steps.ps1 files can be translated so tests are written in the business stakeholder's language. Be familiar with regular expressions for parameter matching — the regex in a step definition is the bridge between human-readable Gherkin text and executable PowerShell code.
Test Cases

No formal test table for TA_07. The mentor should ask the trainee to verbally trace through the execution flow using a provided example feature file, identifying: which .steps.ps1 matches, which regex captures which parameter, and what PowerShell executes.

Mentor Tips / Troubleshooting
Mentor Notes
  • The regex in .steps.ps1 is the bridge between business-readable Gherkin text and PowerShell code — capture groups extract parameters that become PowerShell variables in the code block.
  • Ask the trainee to trace a step manually: "Given this Gherkin line, find the regex that matches it, identify the capture groups, and show what value each variable receives."
  • Confirm the trainee understands that Pester 4's Invoke-Gherkin is the entry point for feature files — not Invoke-Pester alone.
TA_08

Create a Gherkin Feature File

Requirements
Create a complete Gherkin feature test from scratch: both a .feature file and its companion .steps.ps1 file. The feature must execute successfully with Invoke-Gherkin and produce visible test output.
Solution Steps
  1. Create TA_08_personTemplate.feature with at least one scenario and meaningful Given/When/Then steps for a Person template test.
  2. Create the companion TA_08_personTemplate.steps.ps1 in the same directory as the feature file, with regex step definitions matching all steps in the feature file.
  3. Run the feature file with Invoke-Gherkin.
Hint
The .steps.ps1 file must be in the same directory as the .feature file — placing it elsewhere is the most common mistake. Setting logging to DEBUG in Setup.ps1 shows step matching details in the output, which is invaluable for troubleshooting step registration failures.

TA_09

Custom Gherkin Assertion with Parameters

Requirements
Extend the TA_08 approach by adding parameterized Gherkin steps. Create a feature file and steps file where step definitions use regex capture groups to extract parameter values from the Gherkin text and pass them as variables to the PowerShell assertion code.
Solution Steps
  1. Create TA_09_personTemplate_parms.feature with scenarios that use parameterized Gherkin steps (e.g., values passed directly in the step text like "Given a Person with department <name>").
  2. Create the companion TA_09_personTemplate_parms.steps.ps1 in the same directory with regex capture groups that extract the parameter values and use them in the PowerShell assertion.
  3. Run both scenarios and verify parameter values are correctly captured and used.
Hint
Start with a simple assertion (no parameters) first, confirm it passes, then add parameterization. Understand how parameters work by studying the examples in the /lib directory before writing your own. Regex capture groups use (...) syntax — each group becomes a parameter in the PowerShell step definition's param() block.
Test Cases
# Test Scenario Test Steps Expected Results Mentor Notes
1 Feature without parameters Run a simple Gherkin step with no capture groups — hardcoded values only Step matches and the assertion passes with no parameter extraction errors Confirm this baseline works before adding parameterization — it isolates regex matching from parameter handling issues.
2 Feature with parameters Run a Gherkin step where a value (e.g., a name or number) is embedded in the step text and extracted by the regex capture group The parameter value is captured correctly from the step text and used in the assertion; the test passes with the correct value Ask the trainee to log the captured parameter value to the console to confirm it matches what was written in the feature file. This makes the capture group mechanics visible.
Mentor Tips / Troubleshooting
Mentor Notes
  • Same file-location rule as TA_08: the .steps.ps1 MUST be in the same directory as the .feature file.
  • If a parameterized step is "not found", the regex likely does not match the exact phrasing in the feature file. Use DEBUG logging to see what text TestAssist is trying to match.
  • The most instructive debugging step: ask the trainee to copy the step text from the .feature file and test the regex against it in isolation using PowerShell's -match operator before relying on TestAssist to match it.
  • Regex capture groups in step definitions use standard PowerShell [regex] syntax — $matches[1], $matches[2], etc., correspond to the first and second capture groups.
