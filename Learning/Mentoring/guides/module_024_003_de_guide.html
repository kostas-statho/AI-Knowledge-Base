DE_01 Permissions & Access Setup
Requirements
Configure the complete permission structure required to make DataExplorer+ visible and usable for the correct identities. This involves a Permissions Group, an AERole, Configuration Parameters, a Predefined SQL, and a Program Function — all must be present and correctly linked.
Solution Steps
  1. Create a Permissions Group and an AERole for DataExplorer+ access.
  2. Create all required Configuration Parameters in OIM Designer.
  3. Create a Predefined SQL (BaseData → Advanced → Predefined SQL) and assign it to the Permission Group.
  4. Create a Program Function and assign it to the Permission Group.
  5. Login with a properly configured user in ITShop and verify DataExplorer+ is accessible.
Test Cases
# Test Scenario Test Steps Expected Results Mentor Notes
1 Login with invalid Person (no access) Login to ITShop as a Person who has NOT been assigned the DataExplorer+ permission group DataExplorer+ is NOT available in the portal Use a clearly separate test identity with no DataExplorer+ group membership to confirm the negative case.
2 Login with valid Person Login to ITShop as a Person who has been assigned the DataExplorer+ permission group via AERole DataExplorer+ is available and functional in the portal Both the Program Function AND the Predefined SQL must be assigned — missing either makes the feature invisible.
Mentor Tips / Troubleshooting
Mentor Notes
  • Why both Program Function AND Predefined SQL? They serve different purposes and the portal checks both independently:
    • Program Function — controls navigation visibility. The portal checks this gate to decide whether the DataExplorer+ menu item appears at all. Think of it as the "can this user see this feature?" check.
    • Predefined SQL — provides the actual data query. Once the menu item is visible, the portal uses the Predefined SQL (resolved via Configuration Parameters) to load the view's data.
    If the Program Function is missing, the menu item never appears and no error is shown. If the Predefined SQL is missing or incorrectly linked, the menu item appears but shows no data and no helpful error. Both failures look identical to the end user: a blank or missing feature. This is the most common source of "it's not working" support calls on DataExplorer+ deployments.
  • Permission Group assignment propagates via the AERole. Confirm the test identity has the AERole assigned and that OIM has processed the role membership through the DBQueue before testing.
  • Configuration Parameters must match the expected key names exactly — the portal reads these at runtime to resolve which Predefined SQL to invoke. A single character mismatch causes silent data failure with no error in the UI.
DE_02 Predefined SQL for Line Manager Attestations
Requirements
Create and configure two Predefined SQL queries that power the Line Manager Attestations view in DataExplorer+. Both queries must be assigned to the Permission Group and linked via Configuration Parameters so the portal can discover and invoke them by name.
Solution Steps
  1. Create the two Predefined SQL queries and assign them to the Permission Group:
    • CCC_DE_LineManagerAttestations_Sub_MainData
    • CCC_DE_LineManagerAttestations_Sub_BusinessRoles
  2. Create the required Configuration Parameters so the portal resolves these queries by their exact names at runtime.
Test Cases
# Test Scenario Test Steps Expected Results Mentor Notes
1 Identity without valid attestation Login as a Person who has no pending or active attestation cases No matching data — attestation view is empty Confirm the SQL is running and returning zero rows (not throwing an error), which is the correct empty-state.
2 Identity with valid attestation Login as a Person who has pending attestation cases assigned to them as Line Manager Attestation cases are displayed in DataExplorer+ Predefined SQL names are config-driven and link to specific UI views by name — a mismatch in Config Parameters means data won't load even if the SQL is correct.
Mentor Tips / Troubleshooting
Mentor Notes
  • The Predefined SQL names in OIM must match exactly what the Configuration Parameters reference. A single character difference = no data loaded, no error shown.
  • Test both queries individually in OIM Designer (BaseData → Advanced → Predefined SQL) to confirm they return data for the expected context before integrating them.
  • If attestations exist in the system but aren't showing, check the WHERE clause scope in the SQL — it must filter by the current user's UID as Line Manager.
DE_03 Attestation Accept / Reject API Endpoints
Requirements
Create two C# API endpoints — one to accept and one to reject an attestation case. Both endpoints must validate that a reason is provided; if no reason is given, they must return an appropriate error message. Build and deploy the *.CompositionApi.Server.Plugin.dll.
Solution Steps
  1. Create the required Configuration Parameters for the accept/reject endpoints.
  2. Create both endpoints (accept and reject), implement reason validation, build the solution, and load the *.CompositionApi.Server.Plugin.dll.
Test Cases
# Test Scenario Test Steps Expected Results Mentor Notes
1 Check "No Reason" functionality Attempt to accept or reject an attestation without providing a reason in the reason field Error message: "No reason to be granted" The validation must run server-side in the C# endpoint, not just client-side in Angular — test by posting directly to the endpoint if needed.
2 Check "Reason" functionality Provide a valid reason text and submit an accept decision Attestation is accepted and the case status updates correctly in OIM Verify the AttestationCase record in Designer to confirm the decision was persisted with the correct reason value.
Mentor Tips / Troubleshooting
Mentor Notes
  • Each new feature follows the same pattern: new Config Parameters + new endpoint class + rebuild DLL + redeploy. This is the same DLL deployment pattern as module 017.
  • After each DLL update, always recycle the IIS application pool for the API Server.
  • The reason string is typically posted in the request body — ensure the C# request model matches the Angular payload structure.
DE_04 Business Role Membership Management
Requirements
Allow the Line Manager to manage Business Role memberships from the DataExplorer+ side-sheet. Only directly assigned and ITShop-requested roles may be removable — dynamic and system-assigned roles must not show remove buttons. After an attestation is approved, remove buttons must disappear for that attestation's roles.
Solution Steps
  1. Create the required Configuration Parameters for Business Role membership management.
  2. Reconfigure the two existing Predefined SQL queries to include membership source/type data needed to determine removability.
  3. Create and load the two required endpoints; build and deploy *.CompositionApi.Server.Plugin.dll.
Test Cases
# Test Scenario Test Steps Expected Results Mentor Notes
1 Only direct and requested roles removable Assign a Business Role via Dynamic rule; assign another via ITShop request; assign another directly. Open the side-sheet and attempt to remove all three. Only the directly assigned and ITShop-requested Business Roles show remove buttons. The dynamic role does not. The membership source data comes from the reconfigured Predefined SQL — verify the SQL correctly identifies assignment origin. Dynamic memberships typically have a specific flag or source value in OIM.
2 Buttons gone after attestation approved Approve an attestation case; then open the DataExplorer+ side-sheet for that identity and re-assign a role to test the current state After attestation approval, no remove buttons are visible for roles covered by that attestation This is a state-dependent behavior — the endpoint must check the attestation approval status before returning the removability flag. Confirm the AttestationCase status is "Approved" in OIM Designer.
Mentor Tips / Troubleshooting
Mentor Notes
  • The Predefined SQL reconfiguration in step 2 is the key data change — without the correct source/type fields, the endpoint cannot make the correct removability decision.
  • Test both assignment types independently before testing the side-sheet to confirm each type is correctly categorized in the SQL output.
DE_05 Bulk Remove Business Roles Button
Requirements
Add a "Bulk Remove" button to the DataExplorer+ side-sheet that removes all removable Business Roles at once. The button must only appear when at least one removable role exists; it must disappear after all roles are removed.
Solution Steps
  1. Create the required Configuration Parameters for the bulk remove feature.
  2. Create and load the two required endpoints; build and deploy *.CompositionApi.Server.Plugin.dll.
  3. Create the two required Predefined SQL queries to identify removable roles for the bulk operation.
Test Cases
# Test Scenario Test Steps Expected Results Mentor Notes
1 Button visible when removable roles exist Assign at least one directly-assigned removable Business Role to a test identity, then open the DataExplorer+ side-sheet The "Bulk Remove" button is visible and, when clicked, removes all removable roles correctly Verify the roles are actually deleted in OIM Designer after the bulk remove — confirm no partial-remove state.
2 Button NOT visible when no removable roles After performing bulk remove (or when the identity has no directly-assigned roles), open the side-sheet again The "Bulk Remove" button is NOT visible The button visibility is driven by whether the Predefined SQL returns any removable rows — if it returns zero rows, the Angular component hides the button.
Mentor Tips / Troubleshooting
Mentor Notes
  • Button visibility is data-driven from the Predefined SQL — ensure the SQL returns an empty result set (not an error) when no removable roles exist.
  • The bulk remove endpoint should use a loop calling the same delete logic as DE_04's individual remove endpoint to avoid code duplication.
DE_06 CCC_DE_Recommendations Table + Import/Remove Scripts
Requirements
Create a custom schema table CCC_DE_Recommendations to store attestation recommendations. Use Data Importer scripts triggered by Process steps on the AttestationCase table to populate and clean up recommendations. Integrate color indicators (green for direct roles only) and update the existing Line Manager Attestations SQL.
Solution Steps
  1. Create the CCC_DE_Recommendations table using the One Identity schema extension tool. The CCC_ prefix is required for custom tables.
  2. Create Import and Remove Scripts via Data Importer; remove any logging statements and hardcoded parameters from the generated scripts to keep them clean and environment-independent.
  3. Create a Process on the AttestationCase table:
    • Create a new Event on AttestationCase; use the ORDERGRANTED event for the import trigger and the DENY event for the remove trigger.
    • Add an "Import" Process Step linked to the import script.
    • Add a "Remove" Process Step linked to the remove script.
  4. Modify CCC_DE_LineManagerAttestations_Sub_BusinessRoles to join with the new CCC_DE_Recommendations table and include the color indicator column.
Test Cases
# Test Scenario Test Steps Expected Results Mentor Notes
1 Recommendations populated properly Run an attestation cycle and open the DataExplorer+ view for a Line Manager identity with pending cases Recommendations are populated and visible in the DataExplorer+ side-sheet Check the CCC_DE_Recommendations table directly in OIM Designer after the Process step fires to confirm data was inserted by the import script.
2 Green indicator only on direct roles Review the DataExplorer+ side-sheet for an identity that has both direct and inherited/dynamic Business Roles The green recommendation color indicator appears only for directly assigned roles The color is derived from the SQL join with CCC_DE_Recommendations — verify the SQL WHERE clause correctly filters by assignment type.
3 Recommendations removed on close Close / reject the attestation case (DENY event) and then check the CCC_DE_Recommendations table Recommendation records for that attestation are removed from the table The DENY event is critical for cleanup — if the remove script does not fire, the table accumulates stale data. Verify the Process step event binding is DENY, not just ORDERGRANTED.
Mentor Tips / Troubleshooting
Mentor Notes
  • The schema extension tool creates the custom table — the CCC_ prefix is required by OIM naming conventions for custom objects.
  • Data Importer scripts are triggered by Process steps on AttestationCase using the ORDERGRANTED event (import) and DENY event (cleanup). The DENY event is critical — without it, stale recommendation data accumulates.
  • After modifying the CCC_DE_LineManagerAttestations_Sub_BusinessRoles SQL, test it in Designer before relying on it in the portal to confirm the JOIN returns expected rows.
  • Remove hardcoded parameters from generated Data Importer scripts — they break when moved between environments (dev, UAT, prod).

Key Reminders Across All Exercises
  • DE_01: DataExplorer+ requires both a Program Function AND a Predefined SQL assigned to the permission group — missing either makes the feature invisible with no error.
  • DE_02: Predefined SQL names are config-driven — they link to specific UI views by exact name in Configuration Parameters. A single character mismatch causes silent failure.
  • DE_03/04/05: Each new feature requires new Config Parameters + rebuild DLL + redeploy. Same DLL deployment pattern as module 017 — always recycle the IIS application pool after each deploy.
  • DE_06: The schema extension tool creates the custom table (CCC_ prefix required). Data Importer scripts are triggered by Process steps on AttestationCase using ORDERGRANTED and DENY events. The DENY event is critical for cleanup — never omit it.
  • General: All DB calls in C# endpoints must use .ConfigureAwait(false); use $"..." interpolation (not string.Format) for strings with \n.
