Scenarios (SCN)

Real-world identity lifecycle scenarios using IT Shop custom request properties, process chains, mail templates, and AD provisioning. These exercises build a complete "create employee via IT Shop" scenario end-to-end.

Key Concepts

  • Request Properties (custom fields on IT Shop request forms)
  • ShoppingCartItem / PersonWantsOrg tables
  • Multi-request resource
  • CentralAccount generation via script
  • Process Automation (scheduled jobs)
  • Mail Templates & Pre-Scripts

Exercise Overview

  • SCN-1: Create Employee via IT Shop (Custom Request Property)
  • SCN-2: Email Notification on External Employee Account Creation
  • SCN-3: Email 15 Days Before Identity Expiry

SCN-1: Create Employee via IT Shop (Custom Request Property)

1

Create Employee via IT Shop

Build a complete "create identity" workflow through the IT Shop, collecting entry/exit dates and company information on the request form itself. This is the foundational exercise for the entire module.

Solution Steps
  1. Step 1 — Extend Tables: Extend both ShoppingCartItem (items added to cart) and PersonWantsOrg (requested items) to add custom columns for Entry/Exit Date and Company fields.
  2. Step 2 — Create Request Property "SCN1": Create a new Request Property named SCN1 using the extended fields. Configure "Obsolete parameters" so fields become read-only after the request is approved.
  3. Step 3 — Multi-request Resource: Create a Multi-request resource for the IT Shop. Add it to the IT Shop with a new Service Item named Create_Employee_SCN1.
  4. Step 4 — Apply Request Property: Assign the SCN1 request property to the Service Item Create_Employee_SCN1.
  5. Step 5 — Script for CentralAccount: Create a script to generate a random Central Account value for new identities.
  6. Step 6 — Create Process: Build the process that ties all previous steps together. Configure the trigger event. Add and configure the process step Insert Person to create the identity.
Important — Obsolete Parameters The "Obsolete parameters" setting on the Request Property means the custom fields become read-only once the request is approved. This is the standard production pattern for data that must only be set at request time and must not be changed afterwards.
Test Cases
#Test ScenarioStepsExpected ResultResult
1 Happy Flow Request via IT Shop with all parameters properly filled Identity created correctly in the system PASS
2 False email format Use email value wrong_format in the request form Process freezes on Insert step (implementation for error on request might be needed) PASS
3 Missing property Request without filling all required fields Request does NOT go to Cart — form validation prevents submission PASS
Mentor Note Test case 2 is worth discussing with trainees: the error occurs at the process step level (Insert Person), not at the form level. This is because email format validation is not enforced by the Request Property itself. A production-ready implementation would add a Pre-Script on the process step to validate email format before attempting the insert.

SCN-2: Email Notification on External Employee Account Creation

2

Email Notification on External Employee Account Creation

Notify the responsible manager by email when an account is created for an external employee. Introduces mail templates and Pre-Script conditions on process steps.

Requirements

When an external employee account is created, send an automated email to the employee's manager containing the newly generated CentralAccount value. The notification should only be sent if the manager has a default email address configured.

Solution Steps
  1. Step 1 — Create Mail Template: Create a new mail template that includes the external employee's name and generated CentralAccount in the email body.
  2. Step 2 — Set Pre-Script on Process Step: Add a Pre-Script condition to the notification process step. The script must check that the employee's manager exists AND has a default email address. If not, the step is skipped.
  3. Step 3 — Add Process Step: Add the email notification step to the existing process (from SCN-1 or a new process). Wire the mail template to the step and configure the recipient as the manager's default email.
Test Cases
#Test ScenarioStepsExpected ResultResult
1 External employee with manager having a default email Request for external employee whose manager has a default email address configured Manager receives email containing the CentralAccount of the newly created employee PASS
2 External employee — manager has no default email Request for external employee whose manager does NOT have a default email address Email is NOT sent; process continues normally without error PASS
Mentor — Pre-Script Pattern The Pre-Script on a process step is a VB.NET or C# code block that returns True to execute the step or False to skip it. This is the standard OIM mechanism for conditional process steps. Show trainees the difference between a step with no Pre-Script (always runs) vs. one with a Pre-Script gate.

SCN-3: Email 15 Days Before Identity Expiry

3

Automated Email 15 Days Before Identity Expiry

An automated daily check that finds identities expiring within 15 days and notifies their manager. Introduces Process Automation (scheduled jobs) — fundamentally different from event-driven processes.

Requirements

Run a daily scheduled automation that identifies identities expiring within 15 days, sends one email to the responsible manager, and sets a flag to prevent duplicate notifications.

Solution Steps
  1. Step 1 — Extend Person Table: Add a new custom boolean field CCC_IsManagerNotifiedForExpire to the Person table via schema extension. This flag prevents the same notification from being sent more than once per identity.
  2. Step 2 — Create Process Automation "CCC_SendMailBeforeExpire": Create a new Process Automation (not a regular process) configured to run on a daily schedule. The automation queries for identities where expiry is within 15 days AND CCC_IsManagerNotifiedForExpire = 0.
  3. Step 3 — Create Process "CCC_SentMail15daysBeforeExpire": Create the process that is triggered when the automation returns matching identities. This process sends the notification email and sets CCC_IsManagerNotifiedForExpire = 1 on the identity.
  4. Step 4 — Create Mail Template "CCC_15daysBeforeExpire": Create the email template used by the process, including the identity's name and expiry date.
Critical — The Flag Mechanism The CCC_IsManagerNotifiedForExpire flag is essential. Without it, the daily automation would send a new email every single day for 15 days. The flag ensures only one notification is ever sent per expiring identity. Make sure trainees understand this pattern — it applies to any "send once" automation.
Mentor — Process Automation vs. Regular Process A regular Process fires on a table event (INSERT, UPDATE, DELETE). A Process Automation fires on a SCHEDULE (daily, hourly, etc.) and runs a query to collect objects. Show trainees the Schedules section in Designer. The automation produces a list of objects, and then for each object, the associated process runs.
Test Cases
#Test ScenarioStepsExpected ResultResult
1 Happy Flow Create a user configured to be deactivated within 15 days (not yet notified) Manager receives email at their default email address PASS
2 Run automation again after notification sent Run the automation a second time without changing anything Automation does NOT gather any identity — the flag (CCC_IsManagerNotifiedForExpire = 1) prevents re-collection PASS
3 Run automation after identities have expired Allow the identities to pass their expiry date, then run the automation Automation does NOT gather any identity — expired identities are outside the 15-day window PASS

SCN-4: Update External Employee via IT Shop

4

Update External Employee via IT Shop

A custom request property that allows a manager to update data on an external employee they are responsible for. Introduces UID-type fields in request properties with filtered lookup conditions.

Requirements

The request form must let a manager select: (1) the external employee to update (filtered to only show external employees they manage), and (2) a new internal manager/value (filtered to active internal users). Updates are then applied via a process.

Solution Steps
  1. Step 1 — Extend Tables: Extend both PersonWantsOrg and ShoppingCartItem with 2 new UID fields to hold the person selections.
  2. Step 2 — Create Request Property "CCC_UpdateExternalEmployee": Create the request property with the following WHERE conditions on each field:
    • CCC_UID_Person01: { isExternal = 1 AND UID_PersonHead = '%USERUID%' } — only external employees managed by the current user
    • CCC_UID_Person02: { isExternal = 0 AND isInActive = 0 } — only active internal users
  3. Step 3 — Create MultiRequest Resource & Service Item: Create the multi-request resource and corresponding service item for the IT Shop.
  4. Step 4 — Add to IT Shop: Add the resource to the IT Shop and assign the request property to the service item.
  5. Step 5 — Set Field Templates: Configure templates for the fields to be populated on the target identity when the request is processed.
Key Pattern — %USERUID% in Conditions The %USERUID% token in the WHERE condition dynamically substitutes the UID of the currently logged-in user. This is how you build "show me only the records I am responsible for" filters in request property fields.
Test Cases
#Test ScenarioStepsExpected ResultResult
1 Happy Flow Request via IT Shop with all parameters properly filled External employee's data is updated correctly PASS
2 False email format Enter email value wrong_format in the request form Process freezes on Update step (same behaviour as SCN-1 Test #2) PASS
3 Missing property Request without filling all required fields Request does NOT go to Cart — form validation prevents submission PASS

SCN-5: AD Account Provisioning via IT Shop (System Role)

5

AD Account Provisioning via IT Shop (System Role)

Full Active Directory account provisioning via IT Shop using a System Role. Introduces AD Forest, Domain, and Account Definition configuration within OIM as a prerequisite to AD provisioning.

Requirements

When an identity requests a specific System Role via the IT Shop, an AD account is automatically created in the configured domain with an initial random password.

Solution Steps
  1. Step 1 — Create AD Forest "CCC_Forest01": Configure the AD Forest object in OIM Designer or Manager. This is the top-level container for the AD structure.
  2. Step 2 — Create AD Domain "CCC_Domain01": Create the AD Domain under the forest. Configure the domain connection parameters.
  3. Step 3 — Create AD Account Definition "CCC_AccDef01": Create the Account Definition that specifies how AD accounts are created (naming scheme, default OU, etc.).
  4. Step 4 — Create System Role "CCC_SR01" & Service Item: Create a System Role and link it to the Account Definition. Create a Service Item for it and add it to the IT Shop.
  5. Step 5 — Set "InitialRandomPassword" Config Parameter: Configure the InitialRandomPassword parameter to ensure new AD accounts receive a randomly generated password on creation.
Config Parameter — InitialRandomPassword This parameter controls whether new AD accounts get a random password assigned at creation time. Without it, accounts may be created with no password or a default password, which is a security risk. Always verify this is set in the AD connector configuration.
Test Cases
#Test ScenarioStepsExpected ResultResult
1 Happy Flow Request System Role CCC_SR01 via IT Shop for any identity AD Account is created in CCC_Domain01 with an initial random password PASS
Mentor — System Role to AD Account Link The connection chain is: Service Item → System Role → Account Definition → AD Domain. Each link must be correctly configured. A common error is forgetting to link the Account Definition to the System Role's Target System tab. Always verify this in the test.

Mentor Tips

SCN-1 — Schema Extension First The ShoppingCartItem/PersonWantsOrg table extension is the key enabler for custom request forms. Show trainees the schema extension tool BEFORE starting this exercise. They need to understand that these are database-level changes before the Request Property can reference them.
SCN-1 — Obsolete Parameters Explained The "Obsolete parameters" setting in the Request Property means the fields expire and become read-only after the request is approved. This is a deliberate production pattern for data that should only be set at request time and never modified after approval.
SCN-3 — Process Automation vs. Regular Process Process Automation (Scheduler) is fundamentally different from a regular process. A regular process fires on a table event (INSERT, UPDATE, DELETE). A Process Automation fires on a SCHEDULE — daily, hourly, etc. Show trainees the Schedules section in Designer and contrast the two types side by side.
SCN-3 — The Flag Is Critical The CCC_IsManagerNotifiedForExpire flag prevents the same manager from receiving multiple emails for the same expiring identity. This is a recurring production pattern — any "notify once" automation requires a flag or timestamp to prevent re-notification. Test cases 2 and 3 both validate this flag works.
SCN-5 — InitialRandomPassword Config Parameter The InitialRandomPassword config parameter controls whether new AD accounts get a random password on creation. Always verify this is configured before running SCN-5. Without it, the AD account may be created in a disabled or passwordless state.
