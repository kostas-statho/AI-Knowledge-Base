AAD No-Mailbox Alerting Process

End-to-end implementation guide — time-based alerting when an AAD account has been active for more than one hour with no Exchange Online mailbox provisioned, with automatic escalation at the four-hour mark.

What You Will Build

  • 2 custom boolean columns on AADUser
  • 2 configuration parameters (recipient email addresses)
  • 2 mail templates (alert + escalation)
  • 2 regular processes (one per template)
  • 2 process automations (hourly scheduled queries)

Key OIM Concepts

  • Process Automation (scheduled SQL-based trigger)
  • Regular Process triggered by automation
  • Mail template with $Token$ substitution
  • Pre-Script on process step (double-check guard)
  • Deduplication flag pattern (one-send guarantee)
  • Schema Extensions (custom boolean columns)

Architecture Overview

▶

Process Flow

Every Hour — Process Automation SQL Query
Finds AADUser records: no linked O365Mailbox, created >1hr ago, flag not set
↓
CCC_AAD_SendNoMailboxAlert
Regular Process: Pre-Script double-check → Send email → Set CCC_IsFirstAlertSent = 1
↓
Every Hour — Second Automation SQL Query
Same objects, but created >4hrs ago AND CCC_IsEscalationSent = 0
↓
CCC_AAD_SendNoMailboxEscalation
Regular Process: Send escalation email to IT Manager → Set CCC_IsEscalationSent = 1
Why Process Automation — Not a Regular Process Event? A regular Process fires once on an INSERT or UPDATE event and cannot re-evaluate itself hours later. For time-gap checks (1 hour, 4 hours) the only correct OIM pattern is a Process Automation — a scheduled job that runs a SQL query and then triggers a Regular Process for each matching object. This is the same pattern as the SCN-3 "15 Days Before Expiry" exercise.
≡

Four Components at a Glance

#NameTypePurpose
1 CCC_AAD_AutomationAlert Process Automation Runs hourly. Queries AADUser: no mailbox, created >1hr ago, first-alert flag not set.
2 CCC_AAD_AutomationEscalation Process Automation Same cadence, but threshold is >4hrs and escalation flag not set.
3 CCC_AAD_SendNoMailboxAlert Regular Process Triggered by Automation 1. Sends Template 1 to Service Desk. Sets CCC_IsFirstAlertSent = 1.
4 CCC_AAD_SendNoMailboxEscalation Regular Process Triggered by Automation 2. Sends Template 2 to IT Manager. Sets CCC_IsEscalationSent = 1.

Step 1 — Schema Extensions

1

Add Custom Boolean Columns to AADUser

These two columns are the deduplication flags. Without them every automation run would re-send the email to the same account. This follows exactly the same pattern as CCC_IsManagerNotifiedForExpire in the SCN-3 exercise.

Schema Extensions is a Separate Tool Schema Extensions is its own OIM tool — opened from Launchpad, not from inside Designer. Open it the same way you open Designer or Manager.
Navigation
Launchpad→ Schema Extensions

Once the Schema Extensions tool is open:

AADUser→ Column Extensions→ New
Columns to Create
Column NameData TypeDefaultPurpose
CCC_IsFirstAlertSent Bool (Bit) 0 Set to 1 after the 1-hour alert is sent. Prevents repeat sends on subsequent automation runs.
CCC_IsEscalationSent Bool (Bit) 0 Set to 1 after the 4-hour escalation is sent. Same mechanism.
Solution Steps
  1. Open Schema Extensions from the Launchpad.
  2. Find the AADUser table in the left navigation tree.
  3. Go to Column Extensions → New. Set Name = CCC_IsFirstAlertSent, Type = Bool, Default = 0. Save.
  4. Repeat for CCC_IsEscalationSent.
  5. Open Designer and click Commit to database in the toolbar to compile the schema change.
  6. In Designer navigate to Database → Update database to apply the physical columns to the SQL Server table.
  7. Confirm: in SSMS run SELECT CCC_IsFirstAlertSent, CCC_IsEscalationSent FROM AADUser WHERE 1=0 — should return column headers with no error.
Two Separate Tools — Two Separate Actions Schema changes are created in the Schema Extensions tool, but compiled and deployed from Designer (Commit to database + Database → Update database). Both actions are required. Until "Update database" runs, the new columns will not exist physically in SQL Server and will not appear in process parameter dropdowns.

Step 1 Pre-Script — double-check guard (VB.NET)

This script re-checks whether a mailbox has appeared since the automation collected this object. If one now exists it returns False to skip the email step cleanly.

' Re-check: has a mailbox been provisioned since the automation queued this process?
Dim uid As String = $UID_AADUser$
Dim count As Integer = Session.Source().Count("O365Mailbox", "UID_AADUser = '" & uid & "'")
If count > 0 Then
    Return False   ' Mailbox now exists — skip email step
End If
Return True
Pre-Script Context The Pre-Script runs at step execution time, not at process queueing time. This means it can catch mailboxes that were provisioned in the window between the automation's SQL query and the step actually executing. It is the only safe place for this check.
Process Step 2 — Set_AlertSentFlag
SettingValue
Step nameSet_AlertSentFlag
ComponentSQL / HandleObject
ActionUpdate — set CCC_IsFirstAlertSent = 1 on the current AADUser object
WhereClauseUID_AADUser = '$UID_AADUser$'
Step Order Matters Step 2 (set flag) must run even if Step 1 is skipped by the Pre-Script (mailbox appeared). This ensures the flag is always set so the object is excluded from future automation runs — even if the email was not sent because the mailbox turned up just in time.
  1. Create the process header with the values above. Save.
  2. Add Step 1 (Notifications/Mail component). Configure mail template and recipient. Paste the Pre-Script.
  3. Add Step 2 (SQL/HandleObject). Configure the WhereClause and set CCC_IsFirstAlertSent = 1.
  4. Ensure Step 2 has no dependency on Step 1 succeeding (run regardless of Step 1 outcome).
  5. Save and Commit to database.

Navigation

Configuration
FieldValue

Object typeAADUser
ProcessCCC_AAD_SendNoMailboxAlert (created in Step 4)

EnabledYes
SQL Query — paste into the Query field

This query runs against the AADUser table and returns the UID_AADUser values of matching objects. OIM then triggers the linked process for each returned UID.

UID_AADUser NOT IN (
    SELECT UID_AADUser FROM O365Mailbox WHERE UID_AADUser IS NOT NULL
)
AND XDateInserted < DATEADD(HOUR, -1, GETUTCDATE())
AND XDateInserted > DATEADD(HOUR, -48, GETUTCDATE())
AND ISNULL(CCC_IsFirstAlertSent, 0) = 0
Query Logic Explained
ConditionPurpose
NOT IN (SELECT … O365Mailbox)Account has no linked Exchange Online mailbox — the core trigger condition
XDateInserted < DATEADD(HOUR, -1, …)Account created more than 1 hour ago
XDateInserted > DATEADD(HOUR, -48, …)Upper cap — prevents picking up accounts older than 48hrs that were never resolved. Adjust to your SLA.
ISNULL(CCC_IsFirstAlertSent, 0) = 0Deduplication flag — once set to 1 after send, this object is excluded forever
Verify the Column Name in O365Mailbox Before deploying, confirm the exact foreign key column name in the O365Mailbox table using Designer → Schema Browser → O365Mailbox. It is usually UID_AADUser but may vary by connector version.
6b

CCC_AAD_AutomationEscalation (4-Hour)

Create a second Process Automation using the same method. Only the name, linked process, and SQL query differ.

Configuration (differences only)
FieldValue
NameCCC_AAD_AutomationEscalation
ProcessCCC_AAD_SendNoMailboxEscalation (Step 5)
SQL Query
UID_AADUser NOT IN (
    SELECT UID_AADUser FROM O365Mailbox WHERE UID_AADUser IS NOT NULL
)
AND XDateInserted < DATEADD(HOUR, -4, GETUTCDATE())
AND XDateInserted > DATEADD(HOUR, -48, GETUTCDATE())

The Two Automations Are Independent An account can receive both the 1-hour alert AND the 4-hour escalation. The automations check different flags (CCC_IsFirstAlertSent vs CCC_IsEscalationSent) independently. An account that resolves its mailbox after the 1-hour email but before 4 hours will be picked up by the 1-hour automation only — the escalation automation's NOT IN O365Mailbox check will exclude it at the 4-hour mark.
  1. Create the second automation with name CCC_AAD_AutomationEscalation.
  2. Link it to CCC_AAD_SendNoMailboxEscalation.
  3. Paste the 4-hour SQL query above. Note only the HOUR, -4 threshold and CCC_IsEscalationSent flag differ from the 1-hour version.
  4. Assign the same hourly schedule.
  5. Enable and Commit to database.

Troubleshooting

TSG

Common Issues & Resolutions

SymptomLikely CauseResolution
Automation never picks up any objects SQL query has wrong column name or threshold too strict Run the SQL directly in SSMS against the OIM database. Verify O365Mailbox.UID_AADUser in Schema Browser. Temporarily set DATEADD(MINUTE, 0, …) to test without the time gap.
Process in Job Queue but no email received SMTP not configured, or wrong recipient address in config param Check Base Data → Mail Server and send a test email. Verify config parameter value at Custom\AAD\Notifications\ServiceDeskEmail.
Token renders as literal $DisplayName$ Column name wrong, or Base object not set to AADUser Open template in Designer → verify Base object = AADUser. Cross-check column name against AADUser in Schema Browser.
Same account receives the email every automation run CCC_IsFirstAlertSent flag not being set — Set step is failing Check Job Queue → find Set_AlertSentFlag step — is it Finished or Error? Check WhereClause and column name. Query the DB to confirm flag value.
Pre-Script skips email unexpectedly Mailbox appeared between automation query and step execution — this is correct behaviour Verify by querying O365Mailbox for the account. If it is a script logic error, temporarily remove the Pre-Script to test, then re-add it.
Process step status = Frozen A Wait step condition is never satisfied (should not occur in this design) Navigate to Job Queue → Frozen → inspect the frozen step. Unfreeze after investigating root cause.
Schema extension columns not visible in process parameters Schema compiled but physical database not updated After adding columns in Designer, run Database → Update database. This creates the physical columns in SQL Server — Commit to database alone is insufficient.
Generating Condition syntax error Multi-line VB.NET used in the Generating Condition field The Generating Condition only accepts a single inline expression: {Value = SomeBoolExpr}. Move complex logic to a Pre-Script on the process step.
