Overview What Changed and Why

The v1.0 process chain architecture is sound. The two-stage escalation pattern with independent flag deduplication is the correct approach for scheduled monitoring in OIM. The improvements in this supplement address four specific weaknesses identified during review.

Scope This document does not change the overall architecture — the same 4 OIM objects from v1.0 still apply. You are patching SQL queries, adding one config parameter, adjusting two process step settings, and adding one new automation + process pair.

Summary of Changes

#ImprovementTypePriority
1 NOT EXISTS over NOT IN — fixes potential NULL-safety issue; improves query performance on large O3EMailbox tables SQL Correctness High
2 Config parameter for 48-hour age cap — makes SLA window ops-configurable without a Designer edit Flexibility Medium
3 Retry + flag dependency fix — prevents the flag being set when an email send fails silently Resilience High
4 Reset automation for resolved accounts — clears flags when mailbox appears so accounts can be re-monitored if mailbox is later removed State Integrity Medium
Apply improvements 1 and 3 first These address correctness issues. Improvements 2 and 4 add capability. If you are applying changes to a live environment, prioritise 1 and 3 in the first deployment window.

Improvement 1 SQL: NOT EXISTS over NOT IN
The Problem with NOT IN and NULLs

The v1.0 generating conditions use:

UID_AADUser NOT IN ( SELECT UID_AADUser FROM O3EMailbox WHERE UID_AADUser IS NOT NULL )

The WHERE UID_AADUser IS NOT NULL guard was added specifically to prevent a SQL NULL-propagation trap: if any row in the subquery returns a NULL, NOT IN silently returns zero rows for the entire outer query — no accounts are picked up, no errors are raised, and the automation appears to succeed. The guard patches this, but it is fragile. If the schema is extended and a new O3EMailbox row is inserted with a NULL FK before the join column is populated, the guard may not fire depending on transaction timing.

Why this matters A single NULL in the O3EMailbox subquery silently disables the entire automation. Accounts without mailboxes are never alerted. This failure mode produces no errors in the Job Queue and no system journal entries — it is invisible.

NOT EXISTS avoids this entirely. It evaluates row-by-row against the outer query's current UID_AADUser — a correlated subquery — and handles NULLs correctly by design. It also allows the database engine to short-circuit on first match, making it faster on large O3EMailbox tables.

Corrected Alert Query (Generating Condition — CCC_AAD_AutomationAlert)
Before (v1.0)
UID_AADUser NOT IN ( SELECT UID_AADUser FROM O3EMailbox WHERE UID_AADUser IS NOT NULL ) AND XDateInserted < DATEADD(HOUR, -1, GETUTCDATE()) AND XDateInserted > DATEADD(HOUR, -48, GETUTCDATE()) AND ISNULL(CCC_IsFirstAlertSent, 0) = 0
After (v1.1)
NOT EXISTS ( SELECT 1 FROM O3EMailbox m WHERE m.UID_AADUser = AADUser.UID_AADUser ) AND XDateInserted < DATEADD(HOUR, -1, GETUTCDATE()) AND XDateInserted > DATEADD(HOUR, -48, GETUTCDATE()) AND ISNULL(CCC_IsFirstAlertSent, 0) = 0
Corrected Escalation Query (Generating Condition — CCC_AAD_AutomationEscalation)
Before (v1.0)
UID_AADUser NOT IN ( SELECT UID_AADUser FROM O3EMailbox WHERE UID_AADUser IS NOT NULL ) AND XDateInserted < DATEADD(HOUR, -4, GETUTCDATE()) AND XDateInserted > DATEADD(HOUR, -48, GETUTCDATE()) AND ISNULL(CCC_IsEscalationSent, 0) = 0
After (v1.1)
NOT EXISTS ( SELECT 1 FROM O3EMailbox m WHERE m.UID_AADUser = AADUser.UID_AADUser ) AND XDateInserted < DATEADD(HOUR, -4, GETUTCDATE()) AND XDateInserted > DATEADD(HOUR, -48, GETUTCDATE()) AND ISNULL(CCC_IsEscalationSent, 0) = 0
NULL-safe Better performance on large tables No schema changes required

Improvement 2 Config Parameter for Age Cap
The Problem — Hardcoded SLA Window

The v1.0 queries cap account age at 48 hours:

AND XDateInserted > DATEADD(HOUR, -48, GETUTCDATE())

This value is hardcoded in both automation generating conditions. Changing the SLA window (e.g. from 48 to 72 hours as the organisation grows its provisioning pipeline) requires opening Designer, editing the SQL in both automations, and recompiling. This is an admin-level task gated on a developer tool.

Moving this to a configuration parameter makes it an operations-level setting — no Designer access required.

New Configuration Parameter

Create the following parameter under Base Data → Configuration parameters:

FieldValue
Full pathCustom\AAD\Notifications\MaxAgeHours
Data typeString
Default value48
DescriptionMaximum age (hours) of AADUser accounts to include in no-mailbox alerting. Accounts older than this value are ignored. Default: 48. Increase if your provisioning pipeline SLA exceeds 48 hours.
Why String, not Integer? OIM configuration parameters are typed as String in Designer. The value is read as a string and cast to an integer in the Pre-Script before use. This is standard OIM practice.
Pre-Script Pattern to Read the Parameter

Add the following Pre-Script to Step 1 (Send_AlertEmail / Send_EscalationEmail) to perform the late-check using the configurable cap. This replaces the optional Pre-Script from v1.0 and adds the config-driven age check.

Pre-Script (VBScript) — applies to both alert and escalation processes
' Read configurable max-age cap from OIM configuration Dim maxHoursStr As String maxHoursStr = Session.Config().GetConfigParm("Custom\AAD\Notifications\MaxAgeHours") Dim maxHours As Integer = 48 If IsNumeric(maxHoursStr) Then maxHours = CInt(maxHoursStr) ' Late-check: skip if mailbox has appeared since automation queued this process Dim mailboxCount As Integer = Session.Source().Count( _ "O3EMailbox", "UID_AADUser = '$UID_AADUser$'") If mailboxCount > 0 Then Return False ' Late-check: skip if account is now older than the configured cap Dim insertedUtc As Date = CDate("$XDateInserted$") Dim ageHours As Double = DateDiff("h", insertedUtc, Now()) If ageHours > maxHours Then Return False Return True
Step 2 (Set_*Flag) still runs unconditionally If the Pre-Script returns False on Step 1 because the account is now too old, Step 2 should still set the flag to prevent the account from being re-queued on every automation run. This is the same design principle as v1.0.
Ops-configurable — no Designer access needed No SQL changes required

Improvement 3 Retry and Flag Dependency Fix
The Silent Failure Problem

In v1.0, both processes have two steps:

  1. Send_AlertEmail — sends the email via the Notifications component
  2. Set_AlertSentFlag — sets CCC_IsFirstAlertSent = 1 unconditionally

Step 2 runs regardless of whether Step 1 succeeded. If the SMTP server is temporarily unavailable or the mail component throws an error, Step 1 fails — but Step 2 still runs and sets the flag. The account is now permanently marked as alerted, but no email was ever sent. The next automation run excludes the account from the query. The failure is invisible.

Consequence An SMTP outage during an automation run silently marks all accounts as alerted without any notifications being delivered. There is no audit trail, no retry, and no way to detect this without manually querying the database.
Fix A — Enable Wait-Mode Retry on Email Steps

In Designer, open each Send step (Send_AlertEmail, Send_EscalationEmail) and configure the following on the Error Handling tab:

SettingValue
On errorWait
Number of retries3
Retry interval (minutes)10

This causes OIM to hold the process step in the Job Queue and retry it up to 3 times before marking the step as failed. Transient SMTP outages lasting less than 30 minutes are handled automatically.

Fix B — Make Flag Step Conditional on Email Success

Open the Set_AlertSentFlag step (and Set_EscalationSentFlag) in Designer and change the previous step condition:

Before (v1.0) — unconditional
Previous step result: <Always>
After (v1.1) — conditional on success
Previous step result: On success

With this change, if Send_AlertEmail exhausts all 3 retries and ultimately fails, Set_AlertSentFlag does not run. The account remains unalerted and un-flagged. On the next hourly automation run, the account is picked up again and the process is re-queued. The failure is also now visible in the Job Queue error log.

Designer navigation Process → Process Step → Set_AlertSentFlag → Properties → General tab → "Previous step" dropdown. Change from Always to On success. Repeat for Set_EscalationSentFlag.
Prevents silent missed alerts Auto-recovers from transient SMTP outages No schema changes required

Improvement 4 Reset Automation for Resolved Accounts
Why Permanent Flags Are Problematic

In v1.0, CCC_IsFirstAlertSent and CCC_IsEscalationSent are set to 1 and never cleared. This is appropriate for the common case — an account is provisioned normally and its mailbox arrives within the SLA window.

The problem arises in edge cases:

  • A mailbox is provisioned (flags set), then later removed (e.g. licence revoked, migration error). The account now has no mailbox again, but the flags prevent re-alerting.
  • A test account is created, alerted on, then the mailbox is manually removed and the account is re-used. Same result — permanently suppressed.
  • An administrator clears an O3EMailbox row directly in the database during troubleshooting. The alert system does not recover.

The fix is a third automation that runs on accounts where a mailbox now exists but the alert flags are still set. It clears both flags, restoring the account to a monitorable state.

New: CCC_AAD_AutomationResetFlags New

Create a third Process Automation in Designer alongside the existing two:

FieldValue
NameCCC_AAD_AutomationResetFlags
Object typeAADUser
ScheduleSame hourly schedule as alert and escalation automations
Process to queueCCC_AAD_ResetNoMailboxFlags
Generating Condition SQL
EXISTS ( SELECT 1 FROM O3EMailbox m WHERE m.UID_AADUser = AADUser.UID_AADUser ) AND ( ISNULL(CCC_IsFirstAlertSent, 0) = 1 OR ISNULL(CCC_IsEscalationSent, 0) = 1 )

This picks up accounts that now have a mailbox but still carry one or both alert flags — exactly the accounts that were previously alerted and have since been resolved.

New: CCC_AAD_ResetNoMailboxFlags Process New

Create a new Regular Process with one step:

1
Reset_AlertFlags

No email is sent on reset This is an internal housekeeping process. It clears flags silently. The next time this account loses its mailbox, the alert and escalation automations will pick it up again as a fresh unalerted account.
