INTRAGEN. Audit

Control 3 — AD Group Automation Evidence

Control 3  Audit Walkthrough   4 queries  |  One Identity Manager 9.3

Overview
Control Statement "Adding / removing users to AD groups is an automated process triggered by HR and updates are executed by the tool One Identity Manager. New / leaving / moving employees are automatically updated on AD groups by the tool."

What these queries prove

  • AD group memberships originate from automated assignment (not manual)
  • The automation is driven by HR data (Business Role → AD Group wiring)
  • No unauthorised changes were made to process configuration
  • Leaver accounts are disabled and group memberships cleaned up automatically

Prerequisites

  • Read access to OIM SQL Server (STATHOPOULOSK\OneIM or client DB)
  • SSMS or equivalent query tool
  • No schema changes required — all tables are standard OIM 9.x
  • Run queries in the order presented for a logical audit narrative

Key tables touched

  • ADSAccountInADSGroup — membership + XOrigin
  • OrgHasADSGroup — HR role → AD group wiring
  • PersonInOrg — HR-driven role assignments
  • JobChain — process definition change log
  • Person, ADSAccount — identity + AD account

Audit narrative order

  • Q1 — Show memberships are mostly automated
  • Q2 — Confirm no config changes were made
  • Q3 — Show the HR → AD assignment path
  • Q4 — Confirm leavers are cleaned up
XOrigin Bitfield Reference

Every membership record in OIM carries an XOrigin integer that records how the assignment was created. This is the primary audit evidence that memberships are automated rather than manually maintained.

XOrigin ValueLabelMeaningAudit Implication
1 Direct Manual assignment An administrator explicitly added this person to this group inside OIM. Requires justification — should be rare or zero for automated controls.
2 Inherited Role-based (HR) Assigned automatically because the person holds a Business Role (Org) that is linked to this AD group. Expected — confirms HR-driven automation.
4 Dynamic Rule-based Assigned by a dynamic group rule (SQL WHERE clause evaluated at sync time). Expected — confirms rule-based automation.
8 Requested IT Shop request Employee or manager placed an IT Shop request that was approved. Acceptable — governed by approval workflow.
3, 6, 10 … Combined Bitwise combination — e.g. 3 = Direct + Inherited (both apply simultaneously). Interpret each bit independently.
Key point for auditors XOrigin values 2, 4, and 8 all represent system-managed assignments. Only XOrigin = 1 (Direct) represents a purely manual entry. Q1 below quantifies the distribution across all current memberships.
Query 1 — AD Group Assignment Origin Summary
Q1

ADGroup_OriginSummary — Automated vs. Manual Breakdown

Purpose

Proves what percentage of current AD group memberships are automated (XOrigin ≠ 1) versus manually assigned (XOrigin = 1). The distribution summary is the headline figure for the auditor. The second query lists any purely manual memberships with timestamps for spot-check.

Query A — Origin Distribution (headline figure)
-- AD group assignment origin breakdown — automated vs. manual audit proof
-- Tables: ADSAccountInADSGroup, ADSGroup, Person, ADSAccount
-- OIM 9.3

SELECT
    CASE aag.XOrigin
        WHEN 1 THEN 'Direct (manual)'
        WHEN 2 THEN 'Inherited (role-based)'
        WHEN 4 THEN 'Dynamic (rule-based)'
        WHEN 8 THEN 'Requested (IT Shop)'
        ELSE        CAST(aag.XOrigin AS VARCHAR(10)) + ' (combined)'
    END                        AS AssignmentOrigin,
    COUNT(*)                   AS MembershipCount,
    CAST(COUNT(*) * 100.0 /
        SUM(COUNT(*)) OVER ()
        AS DECIMAL(5,2))       AS PctOfTotal
FROM ADSAccountInADSGroup aag
GROUP BY aag.XOrigin
ORDER BY MembershipCount DESC;
Query B — Detail of Manual-Only Entries (XOrigin = 1)
-- Show any purely manual (direct) AD group assignments with identity detail
SELECT TOP 50
    p.FirstName + ' ' + p.LastName  AS FullName,
    p.CentralAccount,
    ag.cn                           AS ADGroup,
    ag.DistinguishedName,
    aag.XDateInserted               AS AssignedOn
FROM ADSAccountInADSGroup aag
JOIN ADSAccount a  ON a.UID_ADSAccount = aag.UID_ADSAccount
JOIN Person     p  ON p.UID_Person     = a.UID_Person
JOIN ADSGroup   ag ON ag.UID_ADSGroup  = aag.UID_ADSGroup
WHERE aag.XOrigin = 1   -- direct / manual only
ORDER BY aag.XDateInserted DESC;
How to Interpret Results
ScenarioQuery A showsAudit conclusion
Control operating effectively XOrigin 2 / 4 / 8 account for >95% of rows; XOrigin 1 is zero or near-zero Automation confirmed. Manual entries absent or negligible.
Some manual entries exist XOrigin 1 appears with a small count Review Query B. Each entry needs a business justification or exception record.
High manual count XOrigin 1 > 20% of total Control gap — automated process may not be functioning or is being bypassed.
Presenter tip Run Query A first — present the percentage table on screen. If XOrigin 1 is zero, state it clearly: "All current AD group memberships were created by the automation — no manually assigned entries exist." If XOrigin 1 entries do exist, use Query B to walk through each one with the auditor.
Query 2 — Process Configuration Change Log
Q2

Control3_ProcessChangeLog — Were Any Process Definitions Changed?

Purpose

Directly answers the auditor's question: "Please confirm if there were further changes performed to the automated functionality." The JobChain table holds all process definitions. Its XDateUpdated column is updated every time a process is saved in OIM Designer. Zero rows in the date range = zero configuration changes.

Query A — All Process Chains Changed (last 90 days)
-- All process chain definitions changed in last 90 days
-- Zero rows = no changes to automation configuration
-- Tables: JobChain
-- OIM 9.3

SELECT
    Name                AS ProcessChain,
    ProcessDisplay      AS FriendlyName,
    XDateUpdated        AS LastModified,
    XUserUpdated        AS ModifiedBy
FROM JobChain
WHERE XDateUpdated >= DATEADD(DAY, -90, GETDATE())
ORDER BY XDateUpdated DESC;
Query B — Narrowed to AD-Related Chains
-- AD-group-related process chains changed in last 90 days
-- Covers common OIM naming patterns for AD provisioning processes

SELECT
    Name,
    ProcessDisplay      AS FriendlyName,
    XDateUpdated        AS LastModified,
    XUserUpdated        AS ModifiedBy
FROM JobChain
WHERE XDateUpdated >= DATEADD(DAY, -90, GETDATE())
  AND (
       Name LIKE '%ADS%'
    OR Name LIKE '%ADSGroup%'
    OR Name LIKE '%Membership%'
    OR Name LIKE '%Synchroniz%'
    OR Name LIKE '%Provision%'
  )
ORDER BY XDateUpdated DESC;
Query C — Full Process Inventory (all registered chains)
-- All registered process chains — baseline inventory
-- Use to show auditors the complete list of automation processes

SELECT
    Name                AS ProcessChain,
    ProcessDisplay      AS FriendlyName,
    XDateUpdated        AS LastModified
FROM JobChain
ORDER BY XDateUpdated DESC;
How to Interpret Results
ScenarioWhat you seeAudit conclusion
No changes since last audit Query A returns 0 rows No modifications to process automation. Control unchanged.
Changes exist — AD-unrelated Query A has rows; Query B returns 0 Changes were to unrelated processes. AD automation untouched.
AD-related changes exist Query B returns rows Review each row. If changes were authorised improvements, provide change request reference. If unexpected, investigate.
Note on date range The queries default to last 90 days. Adjust DATEADD(DAY, -90, ...) to match the audit period (e.g. -365 for a full-year review, or replace with a specific date: WHERE XDateUpdated >= '2025-01-01').
Query 3 — HR to AD Provisioning Path
Q3

Control3_HRtoAD_Path — How HR Data Drives AD Group Membership

Purpose

Visualises the complete chain: Person → Business Role (Org) → AD Group. The join OrgHasADSGroup is the configuration record that wires an HR organisational unit or business role to an AD group. This is the structural proof that the control is "triggered by HR".

Query A — Full Person → Business Role → AD Group Chain
-- HR-to-AD provisioning path: Person → Business Role → AD Group
-- OrgHasADSGroup is the key config record linking HR roles to AD groups
-- Tables: Person, PersonInOrg, Org, OrgHasADSGroup, ADSGroup
-- OIM 9.3

SELECT
    p.FirstName + ' ' + p.LastName  AS Employee,
    p.CentralAccount,
    o.Ident_Org                     AS BusinessRole,
    ag.cn                           AS ADGroup,
    ag.DistinguishedName,
    pio.XDateInserted               AS HRAssignmentDate,
    CASE pio.XOrigin
        WHEN 1 THEN 'Direct'
        WHEN 2 THEN 'Inherited'
        WHEN 4 THEN 'Dynamic'
        WHEN 8 THEN 'Requested'
        ELSE CAST(pio.XOrigin AS VARCHAR)
    END                             AS BRAssignmentOrigin
FROM PersonInOrg       pio
JOIN Person            p   ON p.UID_Person    = pio.UID_Person
JOIN Org               o   ON o.UID_Org       = pio.UID_Org
JOIN OrgHasADSGroup    oag ON oag.UID_Org     = o.UID_Org
JOIN ADSGroup          ag  ON ag.UID_ADSGroup = oag.UID_ADSGroup
ORDER BY p.LastName, o.Ident_Org, ag.cn;
Query B — Business Role → AD Group Wiring Summary
-- Summary: how many AD groups each Business Role provisions
-- Use to show auditors that the wiring is configuration-controlled, not ad-hoc

SELECT
    o.Ident_Org      AS BusinessRole,
    COUNT(*)         AS LinkedADGroups
FROM OrgHasADSGroup oag
JOIN Org o ON o.UID_Org = oag.UID_Org
GROUP BY o.Ident_Org
ORDER BY LinkedADGroups DESC;
How to Interpret Results
ScenarioWhat you seeAudit conclusion
Automation fully wired Query A returns rows; Query B shows Business Roles with ≥1 linked AD group HR role assignments directly drive AD group provisioning. The path is configuration-managed.
No wiring exists Query B returns 0 rows OrgHasADSGroup is empty — the HR→AD automation path is not configured. Investigate before the meeting.
Partial wiring Some business roles have no linked AD groups Some roles may be managed outside OIM (e.g. direct AD, another tool). Document scope boundary.
Presenter tip Walk the auditor through one concrete example row from Query A: "This person was assigned to Business Role [X] by HR. OIM detected that change, looked up which AD groups [X] maps to via OrgHasADSGroup, and automatically provisioned the corresponding ADSAccountInADSGroup record. No manual step was involved."
Query 4 — Leaver Cleanup Check
Q4

Control3_LeaverCheck — Leavers Disabled and Groups Removed

Purpose

Proves the "leaving employees" leg of the control. When a person is deactivated in OIM (typically fed from HR), the tool should automatically disable their AD account and remove them from all AD groups. This query shows inactive persons alongside their AD account status and remaining group count.

Query — Inactive Persons with AD Account Status
-- Leaver check: confirm disabled persons have disabled AD accounts and zero group memberships
-- IsInActive = 1 on Person means the identity has been deactivated (leaver)
-- IsDisabled = 1 on ADSAccount means the AD account is locked/disabled
-- Tables: Person, ADSAccount, ADSAccountInADSGroup
-- OIM 9.3

SELECT
    p.FirstName + ' ' + p.LastName  AS Employee,
    p.CentralAccount,
    p.XDateUpdated                  AS PersonDeactivatedOn,
    a.SAMAccountName,
    CASE a.IsDisabled
        WHEN 1 THEN 'Yes — Disabled'
        WHEN 0 THEN 'No — Still Active'
        ELSE 'No AD Account'
    END                             AS ADAccountDisabled,
    COUNT(aag.UID_ADSGroup)         AS RemainingGroupMemberships
FROM Person p
LEFT JOIN ADSAccount           a   ON a.UID_Person      = p.UID_Person
LEFT JOIN ADSAccountInADSGroup aag ON aag.UID_ADSAccount = a.UID_ADSAccount
WHERE p.IsInActive = 1
GROUP BY
    p.FirstName, p.LastName, p.CentralAccount, p.XDateUpdated,
    a.SAMAccountName, a.IsDisabled
ORDER BY p.XDateUpdated DESC;
How to Interpret Results
ScenarioADAccountDisabledRemainingGroupMembershipsAudit conclusion
Clean leaver Yes — Disabled 0 Control working. Account disabled and all group memberships removed by OIM.
Account disabled, groups pending sync Yes — Disabled > 0 Account is locked. Groups may be queued for removal — check if sync has run recently (see SyncJournal.sql).
Account still active No — Still Active Any Gap — deactivation has not propagated to AD. Investigate process chain execution for this identity.
No AD account No AD Account 0 Person never had an AD account, or account was deleted. Acceptable if consistent with HR record.
Timing note OIM processes changes via an asynchronous job queue. A person deactivated today may show 1–2 remaining group memberships if the AD sync has not yet run. Check DPRJournal (SyncJournal.sql) to confirm the last successful sync time before drawing conclusions from this query.
How to Run These Queries
▶

Step-by-step instructions for the meeting

Before the meeting
  1. Open SSMS and connect to the client's OIM SQL Server (confirm server name in advance).
  2. Select the OIM database (typically OneIM or IdentityManager).
  3. Run each query in a new query tab — keep all 4 tabs open and ready.
  4. Check that all queries return rows without errors before the meeting starts.
  5. Note any XOrigin = 1 entries or inactive persons with active accounts — prepare explanations.
During the walkthrough
  1. Q1 — Present the origin distribution table. Confirm automated % to auditor.
  2. Q2 — Run live. If 0 rows: state "no process configuration changes in the audit period". If rows exist: walk through each change and reference the corresponding change request.
  3. Q3 — Pick one specific employee, show their Business Role → AD Group path. This is your "here is how HR triggers the automation" moment.
  4. Q4 — Show the leaver list. Confirm disabled = Yes and groups = 0 for the majority. Address any exceptions.
Supplementary queries (already in the SQL library)
  • OIM/SQL/Sync/SyncJournal.sql — last AD sync run times (supports Q4 timing disclaimer)
  • OIM/SQL/Queue/ProcessChain.sql — job execution history by chain name
  • OIM/SQL/Membership/BusinessRole_Membership.sql — full business role membership detail
